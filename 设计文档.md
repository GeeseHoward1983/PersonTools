### 1. 应用定位与功能清单

PersonalTools 是一个 **C# / .NET 10 (`net10.0-windows7.0`) 的 WPF 桌面工具箱**（`OutputType` 为 `WinExe`），面向开发者/逆向/运维场景，把若干独立的小工具集中到一个带选项卡（`TabControl`）的单窗口中。其能力由 `MainWindow.xaml` 中的各 `TabItem` 直接枚举，可分为两大类：

#### 1.1 二进制文件分析

| 选项卡 | 承载控件 | 说明 |
| --- | --- | --- |
| PE 文件分析器 | `FileTabHostControl`（宿主）→ `PEAnalyzerControl` | 解析 Windows PE（`*.exe / *.dll / *.sys`）：DOS/NT/可选头、节表、导入/导出表、版本资源、证书、CLR(.NET) 元数据、图标资源，并构建惰性依赖树。 |
| ELF 文件分析器 | `FileTabHostControl`（宿主）→ `ELFAnalyzerControl` | 解析 ELF（`*.elf / *.so / 可执行文件`）：ELF 头、程序头、节头、节↔段映射、符号表/动态符号表、动态节、版本符号/依赖、重定位（.rela.dyn / .rela.plt）、GOT/GOT.PLT、Note、ARM 属性、`.ARM.exidx` 异常索引。 |
| Mach-O 文件分析器 | `MachOAnalyzerControl` | 选项卡存在，`MachOAnalyzerControl.xaml.cs` 仅 `InitializeComponent()`，为占位/在建模块。 |

#### 1.2 编码 / 加解密 / 哈希 / 查询工具

| 选项卡 | 承载控件 | 核心服务类 |
| --- | --- | --- |
| 常用错误码对应错误信息查看器 | `ErrorCodeQueryControl`（内含多个 `ErrorCodeQueryItem`） | `ConstString` 下的各错误码表（Windows errno / 系统错误、Linux errno、Mac errno、HTTP 状态码、SQL Server、MySQL、Oracle SQLCODE、ODBC） |
| HASH 值计算 | `HashComputeControl` | `MD5/SHA1/SHA256/SHA384/SHA512`（BCL）、自实现 `Sha224`、自实现 `Sha3`（SHA3-256/384/512） |
| CRC 计算 | `CrcComputeControl` | `CrcCalculator` + `CrcAlgorithm`（内置 CRC-8/16/32 多种变体） |
| Base64 编解码 | `Base64EncoderDecoderControl` | 控件内 BCL `Convert` |
| URL 编解码 | `UrlEncoderDecoderControl` | 控件内 BCL |
| AES 加解密 | `AesEncryptionControl` | `AesCryptoService`（`Aes` + `CryptoStream`） |
| RSA 加解密 | `RsaEncryptionControl` | `RsaCryptoService`（`RSA`，OAEP-SHA256 / PKCS#1+SHA256） |

错误码查询的本地化由 `GlobalState.CurrentLanguageType` 在启动时按 `CultureInfo.CurrentCulture.Name` 求值一次并缓存（`zh-CN`→简体、`zh-TW/HK/MO`→繁体、其余→英文），各 `ConstString` 表据此返回对应语言的描述。

### 2. 整体分层

应用遵循「**WPF 外壳 → 功能控件 → 分析/服务层 → 模型/枚举/常量**」的分层结构，越靠下越与 UI 无关、越可独立测试。

```
┌──────────────────────────────────────────────────────────────────────────┐
│  WPF 外壳层 (Shell)                                                          │
│  App (App.xaml/.cs)  ──启动──▶  MainWindow (MainWindow.xaml/.cs)            │
│   · StartupUri 指向 MainWindow            · TabControl 列出所有功能选项卡     │
│   · 全局异常兜底 (Dispatcher / AppDomain) · 配置两个 FileTabHostControl       │
└───────────────┬────────────────────────────────────────────────────────────┘
                │ 托管 / 实例化
                ▼
┌──────────────────────────────────────────────────────────────────────────┐
│  功能控件层 (UserControls)                                                   │
│  ┌── 文件分析宿主 ──────────────┐   ┌── 独立工具控件 ───────────────────┐   │
│  │ FileTabHostControl           │   │ Hash / CRC / Base64 / URL          │   │
│  │  (多文件 Tab + 拖放 + 打开)   │   │ AES / RSA / ErrorCodeQuery / MachO │   │
│  │   ├ AnalyzerFactory()         │   └────────────────────────────────────┘  │
│  │   └ IFileAnalyzerView 实现:   │                                            │
│  │      PEAnalyzerControl        │   共享: FileDropHelper (拖放取文件+读字节) │
│  │      ELFAnalyzerControl       │        DependencyNode (PE 依赖树节点)      │
│  └──────────────────────────────┘                                           │
└───────────────┬───────────────────────────────┬────────────────────────────┘
                │ 调用                            │ 调用
                ▼                                 ▼
┌─────────────────────────────────┐   ┌────────────────────────────────────┐
│  分析层 (Analysis)               │   │  服务层 (Services, 与 UI 无关)        │
│  PEAnalyzer.* :                  │   │  AesCryptoService  RsaCryptoService   │
│   PEParser(编排) + Parsers/      │   │  CrcCalculator     Sha224  Sha3       │
│   Resources/ + PEHeaderPresenter │   │  ConvertUtils (Hex/位反转/输入字节)   │
│  ELFAnalyzer.* :                 │   │  DependencyResolver (DLL 路径解析)    │

└───────────────┬─────────────────┘

                ▼
┌──────────────────────────────────────────────────────────────────────────┐
│  模型 / 枚举 / 常量层 (Data)                                                  │
│  PEAnalyzer.Models (PEInfo, IMAGE*…)  ELFAnalyzer.Models (ELFHeader…)        │
│  Enums (57 个: 机器类型/重定位类型/段类型/ABI 属性…)  ConstString (错误码表)  │
└──────────────────────────────────────────────────────────────────────────┘
```

各层职责：

- **WPF 外壳层**：`App` 负责进程级生命周期与全局异常；`MainWindow` 是唯一窗口，只负责把功能控件挂到选项卡上，并在构造函数里向两个文件宿主注入「分析器工厂 + 过滤器 + 空状态文案」。外壳层不含任何解析/加密逻辑。
- **功能控件层（`PersonalTools.UserControls`）**：每个选项卡对应一个 `UserControl`，负责 UI 交互（按钮、拖放、表格绑定）并把请求转交分析层/服务层；自身不实现算法。
- **分析层**：PE/ELF 的解析编排与各内容解析器，把字节流转换为模型对象，再经 Presenter/UIHelper 转为可显示的文本或表格行数据。
- **服务层**：纯算法/无 UI 依赖的静态类（加解密、CRC、哈希、Hex 工具、依赖解析）。
- **数据层**：模型结构体/类、枚举（数量庞大，覆盖 ELF 多架构重定位与 ARM ABI 属性）、错误码常量表。

命名空间约定要点：
- 加解密/哈希/通用服务类集中在 `Utils/` 子目录树下：`AesCryptoService`、`RsaCryptoService` 落在 `PersonalTools.Utils.Crypto`（`Utils/Crypto/`）；`Sha3`、`Sha224`、`CrcCalculator` 落在 `PersonalTools.Utils.Hash`（`Utils/Hash/`）；`ConvertUtils`、`GlobalState`、`FileDropHelper` 落在 `PersonalTools.Utils`（`Utils/`）。应用根命名空间仍是 `PersonalTools`。
- PE/ELF 解析按子命名空间细分：`PersonalTools.PEAnalyzer` / `.PEAnalyzer.Parsers` / `.PEAnalyzer.Resources` / `.PEAnalyzer.Models`；`PersonalTools.ELFAnalyzer` / `.ELFAnalyzer.Core` / `.ELFAnalyzer.UIHelper` / `.ELFAnalyzer.Models`。
- 一个值得注意的命名细节：编排器类型 `ELFAnalyzer.ELFAnalyzer`（类型名与命名空间末段同名），代码中以全限定 `ELFAnalyzer.ELFAnalyzer` 出现。
- **按职责拆分文件**是 ELF/PE 解析层的核心组织手段：ELF 解析逻辑分布在一个编排器（`ELFParser.cs`，持有 sealed 的 `ELFParser` 类）外加一组按方面拆分的静态辅助类，每个类各自独立成文件并以其所持类型命名（如 `ELFHeaderInfo.cs` 持 `ELFHeaderInfo`、`ELFSectionHeaderReader.cs` 持静态 `ELFSectionHeaderReader`、`ELFParserConvertUtils.cs` 持 `ELFParserUtils`、`ELFSymbolTableReader.cs`、`ELFRelocation.cs`、`ELFExidxInfo.cs` 等约 20 个文件）；`PEParser` 拆到 `PEParser.cs / PEHeaderParser.cs / PEImportParser.cs / PEExportParser.cs / PEParser.CLR*.cs / PEParserConvertUtils.cs`，其中 `PEParserCLR` 仍以 `partial class` 跨 `PEParser.CLR.*.cs` 三个文件；`PEResourceParser` 拆到 `PEResourceParser.Version.* / .Icon.* / .Certificate.cs` 等点号分组文件，资源目录/版本节点读取器另立为 `ResourceDirectoryReader.cs` / `VersionNodeReader.cs`。这样既让每个文件聚焦一个职责，又控制单文件复杂度（见 §7 维护性度量）。

### 4. 控件宿主与多文件 / 拖放 / 依赖树 UI 模式

PE 与 ELF 两个分析器共享同一套「**宿主 + 视图接口 + 工厂**」机制，UI 模式高度统一。

#### 4.1 文件 Tab 宿主：`FileTabHostControl`

`FileTabHostControl`（`UserControls/FileTabHostControl.xaml.cs`）是**与具体分析器无关**的多文件容器，对外暴露三个可配置点：

```csharp
internal Func<IFileAnalyzerView>? AnalyzerFactory { get; set; } // 创建一个分析视图
public string FileFilter { get; set; }                          // 打开对话框过滤器
public string EmptyHintText { get; set; }                       // 空状态提示文案
```

`MainWindow` 构造时为两个宿主注入不同配置，从而复用同一套宿主代码承载 PE 与 ELF：

```csharp
PEHost.AnalyzerFactory  = () => new PEAnalyzerControl();
ELFHost.AnalyzerFactory = () => new ELFAnalyzerControl();
```

核心逻辑 `OpenOrUpdate(path)`：
1. `Path.GetFullPath` 规范化路径（非法路径 `ArgumentException` 直接忽略）。
2. 遍历现有 `TabItem`，若有相同完整路径（`OrdinalIgnoreCase`，路径存于 `TabItem.Tag`）→ 调用其 `IFileAnalyzerView.LoadFile(full)` **覆盖刷新**并选中，不新建 Tab。
3. 否则 `AnalyzerFactory()` 造新视图，挂回调 `FilesDropped`，`LoadFile(full)`，新建 Tab（`Tag`/`ToolTip` 存完整路径，`Header` 为「文件名 + ✕ 关闭按钮」），加入并选中。

「**按完整路径建/覆盖**」是该宿主的关键策略——同一文件不会开出多个 Tab，重复打开等于热刷新。`UpdateEmptyHint()` 据 Tab 数切换居中空状态提示的可见性。

#### 4.2 统一视图接口：`IFileAnalyzerView`

```csharp
internal interface IFileAnalyzerView
{
    void LoadFile(string filePath);
    Action<IReadOnlyList<string>>? FilesDropped { get; set; }
}
```

`PEAnalyzerControl` 与 `ELFAnalyzerControl` 均实现该接口（且都是 `UserControl`，可直接作为 Tab 内容）。`LoadFile` 是宿主统一入口；`FilesDropped` 用于**把内层视图收到的拖放上抛给宿主**，由宿主按完整路径决定新建/覆盖，避免内外两层都处理拖放造成重复。

#### 4.3 拖放（Drag & Drop）模式

- 宿主层：`FileTabHostControl.xaml` 设 `AllowDrop="True"`，`Host_PreviewDragOver` 仅在 `FileDrop` 时给 `Copy` 效果，`Host_Drop` 取 `string[]` 走 `OpenOrUpdateFiles`。
- 视图层：`PEAnalyzerControl.PEAnalyzer_Drop` / `ELFAnalyzerControl.ELFAnalyzerTab_Drop` 在拿到文件后 **`e.Handled = true` 阻止冒泡到宿主**，再 `FilesDropped?.Invoke(files)` 上抛，统一交宿主处理。
- 独立工具控件（如 `HashComputeControl`）走另一条共享路径：`FileDropHelper.GetFirstDroppedFile(e)` 取首个文件、`FileDropHelper.ReadAllBytes(path)`（`FileStream` + `ReadExactly`）读全部字节，消除各控件重复的拖放/读取样板。

#### 4.4 依赖树（PE 专属）

PE 分析视图右侧的 `DependencyTree`（`TreeView`）以 `DependencyNode`（`UserControls/DependencyNode.cs`）为节点，体现**惰性解析 + 环检测 + 深度限制**：

- 根节点 `DependencyNode.CreateRoot(currentPEInfo)` 表示已打开文件本身，`EnsureLoaded()` 预解析直接依赖并默认展开。
- `CanExpand`（可解析、未成环、深度 `< MaxDepth(16)`）为真时预置「加载中…」占位子节点，使节点显示「+」。
- `DependencyNode_Expanded` / 双击 `DependencyTree_MouseDoubleClick` 触发 `EnsureLoaded()`：调用 `PEParser.ParsePEFile` 解析该 DLL，按本 PE 位数（`PEParserUtils.Is64Bit/Is32Bit`）经 `DependencyResolver.Resolve` 解析每个依赖的完整路径（32 位优先 SysWOW64、64 位优先 System32），并用 `ancestors`（忽略大小写的路径集合）做**环检测**，对成环/未找到/超深度分别给出「(循环依赖)」「(未找到)」等 `Display` 文案。
- 双击节点还会把右侧导入/导出表 (`ShowFunctions`) 切换为该节点自身的 `PEInfo` 数据。

### 5. 全局异常处理

`App.OnStartup`（`App.xaml.cs`）注册两道兜底，避免解析/控件抛出的异常导致应用静默崩溃：

```csharp
DispatcherUnhandledException += App_DispatcherUnhandledException;          // UI 线程
AppDomain.CurrentDomain.UnhandledException += App_UnhandledException;      // 非 UI 线程/致命
```

- `App_DispatcherUnhandledException`：弹 `MessageBox` 显示「发生未处理的错误：{Message}」并设 `e.Handled = true`，**应用继续运行**。
- `App_UnhandledException`：对 `AppDomain` 级异常弹「发生严重错误」（此路径无法阻止进程退出，仅作提示）。

这是「最后一道防线」。在更近的层面，**各功能控件已就地捕获预期异常**并友好提示，从而不依赖全局兜底：
- `PEAnalyzerControl.LoadPEFile` 分别捕获 `FileNotFoundException / UnauthorizedAccessException / IOException / ArgumentException`。
- `ELFAnalyzerControl.AnalyzeELFFile` 用 `when` 过滤一组与解析相关的异常（`IndexOutOfRangeException / EndOfStreamException / OverflowException / InvalidDataException / FormatException` 等），弹「分析 ELF 文件时出错」。
- 解析器内部对畸形输入采取**防御式返回空**而非抛异常：如 `ELFParser.GetSectionData` 越界返回 `[]`、`CopySectionData` 校验 `sh_offset+sh_size` 落在文件内、`PEParser.ParsePEFile` 对 DOS/NT/可选头/节头做边界校验后抛 `InvalidDataException`；内容目录解析「每一步自行处理异常，互不影响」。

### 6. 关键设计决策

1. **编排器 + 按职责拆分的解析类**：`PEParser`（`PEParser.cs`）与 `ELFAnalyzer`/`ELFParser`（`ELFParser.cs`）作为薄编排器，先解析头部、再按固定顺序驱动各内容解析器；具体解析逻辑分散在按职责命名、各以其类型命名的文件中（ELF 侧为一组独立的静态解析器，PE 侧另含 `PEParserCLR` 的 `partial` 拆分）。`ELFAnalyzer` 仅是 `ELFParser` 的薄封装，对外暴露若干 `GetFormatted*Info` 转发到对应格式化器。
2. **表驱动派发（data-driven dispatch）**：把「要执行哪些步骤」表达为一张可遍历的数组，增删步骤只需改表，降低核心方法的圈复杂度与耦合：
   - `PEParser.ContentParsers` 是 `Action<FileStream, BinaryReader, PEInfo>[]`（导入→导出→版本→证书→CLR→图标），`ParseContentDirectories` 仅 `foreach` 调用。
   - `ELFAnalyzerControl.DisplaySteps` 是 `Action<ELFAnalyzer>[]`（16 个展示步骤），`AnalyzeELFFile` 仅 `foreach` 调用。
   - `CrcCalculator.Algorithms` / `HashComputeControl.hashRows` 也是「算法即数据」的表，前者承载 CRC 参数，后者承载 `(名称, Func<byte[],byte[]>)`。
3. **Presenter 模式（PE）**：`PEHeaderPresenter` 把 `PEInfo` → `List<(string Title, Dictionary<string,string>)>` 的展示数据，将对模型/`PEParserUtils`/描述文案的引用从控件中剥离；控件 `DisplayHeaderInfo/DisplayAdditionalInfo` 仅遍历分组渲染 `GroupBox`。ELF 侧由 `UIHelper` 系列格式化器（如 `ELFHeaderHelper.GetFormattedELFHeaderInfo`）承担同等职责，返回多行文本或 `List<*Info>` 行数据。
4. **共享读取原语**：
   - PE：`PEParserUtils.RvaToOffset`（RVA→文件偏移，含跨节/.bss 越界校验）、`ReadAtOffset<T>`（临时定位+`finally` 复位）、`ReadNullTerminatedString`、`ReadUnicodeStringWithMaxLength/Bytes`、`AlignTo4`（版本资源 4 字节对齐）。

## 2 PE 解析层


本节描述 PersonTools 中负责解析 Windows PE（Portable Executable）二进制（`.exe` / `.dll` / 驱动 / .NET 程序集）的解析层。源码位于 `PEAnalyzer/Parsers/`（解析逻辑）与 `PEAnalyzer/Models/`（数据模型与常量），上层展示由 `PEAnalyzer/PEHeaderPresenter.cs` 与 `UserControls/DependencyNode.cs` 消费。

PE 解析层的总体设计原则是：**单一编排器 + 按职责拆分的静态解析类 + 共享的底层读取原语**。`PEParser` 只负责"流程编排"，把"头部解析"与"各内容目录解析"分派给专门的解析类；所有解析类共用 `PEParserUtils` 中与位数无关的读取原语（`RvaToOffset`、`ReadAtOffset`、`ReadNullTerminatedString` 等）。这样可以让"加一种内容目录"只需向一张表里追加一个方法引用，而无需改动核心流程。

---

### 1. 模块概览与职责划分

| 文件 | 类型 | 职责 |
| --- | --- | --- |
| `PEParser.cs` | `PEParser`（static） | 顶层编排器：打开文件流，依次解析头部与内容目录 |
| `PEHeaderParser.cs` | `PEHeaderParser`（static） | 解析 DOS 头 / NT 头 / 可选头（PE32 与 PE32+）/ 节头 |
| `PEImportParser.cs` | `PEImportParser`（static） | 解析标准导入表 + 延迟加载导入表（ILT/IAT thunk 遍历） |
| `PEExportParser.cs` | `PEExportParser`（static） | 解析导出表（导出目录 + 转发导出） |
| `PEParser.CLR.cs` | `PEParserCLR`（static partial） | 解析 `IMAGE_COR20_HEADER`，构造 `CLRInfo` |
| `PEParser.CLR.Metadata.cs` | `PEParserCLR`（static partial） | 解析 .NET 元数据头、流目录（`#~`/`#Strings`）与 TypeDef 表 |
| `PEParser.CLR.Helpers.cs` | `PEParserCLR`（static partial） | ECMA-335 索引宽度计算 + `#Strings` 堆字符串读取 |
| `PEParserUtils.cs` | `PEParserUtils`（static） | 共享读取原语：RVA→偏移、定位读取、字符串读取、位数判定、4 字节对齐 |
| `DependencyResolver.cs` | `DependencyResolver`（static） | 把依赖 DLL 名称解析为磁盘完整路径（按位数优先的系统目录搜索） |

`PEParserCLR` 采用 C# `partial class` 拆成三个文件（CLR 头 / 元数据表 / 辅助函数），既保持逻辑同属一个类型（共享私有常量与方法），又把"COR20 头读取"、"元数据流/表遍历"、"ECMA-335 索引数学"三类职责物理分离，提升可读性与可维护性。

数据模型集中在 `PEAnalyzer/Models/`，结构体（`struct`，多数标注 `[StructLayout(LayoutKind.Sequential)]`）一一对应 PE 规范中的磁盘结构（`IMAGE_DOS_HEADER`、`IMAGE_NT_HEADERS`、`IMAGE_FILE_HEADER`、`IMAGE_OPTIONAL_HEADER`、`IMAGE_SECTION_HEADER`、`IMAGE_DATA_DIRECTORY`、`IMAGE_IMPORT_DESCRIPTOR`、`IMAGE_DELAYLOAD_DESCRIPTOR`、`IMAGE_EXPORT_DIRECTORY`、`IMAGE_COR20_HEADER`），而面向展示的可变信息（`PEInfo`、`ImportFunctionInfo`、`ExportFunctionInfo`、`DependencyInfo`、`CLRInfo`、`PEAdditionalInfo`）则用 `sealed class`。

> 注：`IMAGE_NT_HEADERS` 虽然按规范包含 `OptionalHeader` 字段，但实际解析中可选头与节头单独保存在 `PEInfo.OptionalHeader` / `PEInfo.SectionHeaders`，`PEHeaderParser.ParseNtHeaders` 只读取 `Signature` 与 `FileHeader`。

### 2. 核心编排：`PEParser.ParsePEFile`

#### 2.1 总流程

入口签名：

```csharp
public static PEInfo ParsePEFile(string filePath)
```

流程（`PEParser.cs`）：

1. 新建 `PEInfo { FilePath = filePath }`。
2. 以只读、`FileShare.Read` 方式打开 `FileStream`，并包裹一个 `BinaryReader`（`using` 保证释放）。
3. 调用 `ParseHeaders(fs, reader, peInfo)`：解析并**严格校验** DOS 头 → NT 头 → 可选头 → 节头；任何关键校验失败都抛 `InvalidDataException`（即整个解析失败，向调用方传播）。
4. 调用 `ParseContentDirectories(fs, reader, peInfo)`：按表依次执行各内容目录解析步骤。
5. 返回填充完成的 `PEInfo`。

`fs` 与 `reader` 共享同一底层流（`reader.BaseStream == fs`），整个解析过程通过移动 `fs.Position` 来定位，读取后大多由各方法用 `try/finally` 或 `PEParserUtils.ReadAtOffset` 恢复原位置，互不干扰。

#### 2.2 表驱动的内容目录派发

核心耦合控制点是这张"内容解析步骤表"：

```csharp
private static readonly Action<FileStream, BinaryReader, PEInfo>[] ContentParsers =
[
    PEImportParser.ParseImportTable,
    PEExportParser.ParseExportTable,
    PEResourceParserVersion.ParseVersionInfo,
    PEResourceParserCertificate.ParseCertificateInfo,
    PEParserCLR.ParseCLRHeaderInfo,
    PEResourceParserIcon.ParseIconInfo,
];
```

`ParseContentDirectories` 只是 `foreach` 调用表中每个委托。**设计决策**：用数据（委托数组）而非硬编码的顺序调用来表达对各解析模块的依赖，使得"增删一种内容目录"成为一行数组改动，核心方法零修改。每个步骤自行 `try/catch`（吞掉 `IOException`/`UnauthorizedAccessException` 等并 `Console.WriteLine` 记录），因此**单个目录解析失败不会影响其余目录**——例如导出表损坏不会阻止导入表与 CLR 信息被解析出来。

> 本节聚焦导入/导出/CLR/依赖解析与共享原语；版本（`PEResourceParserVersion`）、证书（`PEResourceParserCertificate`）、图标（`PEResourceParserIcon`）属于资源解析层，详见资源解析章节。

### 3. 头部解析：`PEHeaderParser`

#### 3.1 DOS 头与边界校验（`ParseHeaders`）

`ParseDosHeader(BinaryReader)` 从偏移 0 顺序读出完整 `IMAGE_DOS_HEADER`（含 `e_res1[4]`、`e_res2[10]` 保留数组），关键字段是 `e_magic`（应为 `0x5A4D`，即 "MZ"）与 `e_lfanew`（NT 头在文件中的偏移）。

`ParseHeaders` 逐级做防御性校验，任一失败即抛 `InvalidDataException`：

- `e_magic != PEConstants.DosSignature` → "DOS头签名错误"。
- `e_lfanew >= fs.Length` → "e_lfanew 指向文件外部"。
- 定位 `fs.Position = e_lfanew` 后，`fs.Position + 24 > fs.Length` → "NT头不完整"（24 = 4 字节签名 + 20 字节 `IMAGE_FILE_HEADER`）。

#### 3.2 NT 头

`ParseNtHeaders(BinaryReader)` 读取 4 字节 `Signature` 与 `IMAGE_FILE_HEADER`（`Machine`、`NumberOfSections`、`TimeDateStamp`、`PointerToSymbolTable`、`NumberOfSymbols`、`SizeOfOptionalHeader`、`Characteristics`）。校验：

- `Signature != PEConstants.NtSignature`（`0x00004550`，"PE\0\0"）→ "NT头签名错误"。
- `SizeOfOptionalHeader == 0` 或 `fs.Position + SizeOfOptionalHeader > fs.Length` → "可选头大小不正确"。

#### 3.3 可选头：PE32 / PE32+ 双格式（`ParseOptionalHeader`）

签名：

```csharp
internal static IMAGE_OPTIONAL_HEADER ParseOptionalHeader(BinaryReader reader, ushort sizeOfOptionalHeader)
```

PE32（32 位，魔数 `0x10b`）与 PE32+（64 位，魔数 `0x20b`）的差异集中在三处与位数相关的字段宽度上，本解析器把这些差异收敛到三个私有方法中，主流程保持线性可读：

1. 先读取与位数无关的"通用前缀"：`Magic`、链接器版本、各 Size、`AddressOfEntryPoint`、`BaseOfCode`。
2. `ValidateOptionalHeaderMagic(magic, sizeOfOptionalHeader)`：校验魔数并返回 `is64Bit`；同时校验 `SizeOfOptionalHeader` 不短于该格式的最小值（`Pe32OptionalHeaderBaseSize = 96` / `Pe32PlusOptionalHeaderBaseSize = 112`），魔数非法则抛异常。
3. `ReadImageBaseFields(reader, ref oh, is64Bit)`：
   - PE32：`BaseOfData`（4 字节）+ `ImageBase`（4 字节）。
   - PE32+：**无 `BaseOfData`**（置 0），`ImageBase` 为 8 字节。
4. 中段与位数无关的字段（对齐、版本、`SizeOfImage`、`CheckSum`、`Subsystem`、`DllCharacteristics` 等）顺序读取。
5. `ReadStackHeapFields(reader, ref oh, is64Bit)`：栈/堆的保留与提交大小——PE32 各 4 字节，PE32+ 各 8 字节。
6. `LoaderFlags`、`NumberOfRvaAndSizes`，随后 `ReadDataDirectories` 读数据目录数组。
7. **关键收尾**：`reader.BaseStream.Position = optionalHeaderStart + sizeOfOptionalHeader;` 直接跳到可选头之后（即节表起始），而不是靠"96/112 基准 + 目录数量"的算术推断来定位。**设计决策**：这样对异常或非标准的 `SizeOfOptionalHeader` 更健壮（调用方已校验该范围在文件内），避免因目录数量计算误差导致节表错位。

`IMAGE_OPTIONAL_HEADER` 是单一统一结构（`PEAnalyzer/Models/IMAGE_OPTIONAL_HEADER.cs`），用 `ulong` 容纳 `ImageBase` 与栈/堆字段，从而 PE32/PE32+ 共用同一模型，位数差异只体现在"读多少字节"。

#### 3.4 数据目录数组（`ReadDataDirectories`）

```csharp
private static IMAGE_DATA_DIRECTORY[] ReadDataDirectories(BinaryReader reader, uint numberOfRvaAndSizes)
```

每项 8 字节（`VirtualAddress` + `Size`），最多 `MaxDataDirectories = 16` 项。读取数量取 `min(numberOfRvaAndSizes, 16)`，并进一步收敛到"流中实际剩余可读项数"`(Length - Position) / 8`，**防止被截断的文件触发 `EndOfStreamException`**。后续各内容目录通过常量索引访问该数组：`DirectoryExport=0`、`DirectoryImport=1`、`DirectoryResource=2`、`DirectorySecurity=4`、`DirectoryDelayImport=13`、`DirectoryClrHeader=14`（均定义在 `PEConstants`）。

#### 3.5 节头（`ParseSectionHeaders`）

读取 `NumberOfSections` 个 40 字节（`SectionHeaderSize = 40`）的 `IMAGE_SECTION_HEADER`（`Name[8]`、`VirtualSize`、`VirtualAddress`、`SizeOfRawData`、`PointerToRawData`、各指针/计数、`Characteristics`），存入 `List<IMAGE_SECTION_HEADER>`。`ParseHeaders` 在读取前先校验 `NumberOfSections * 40` 落在文件内（"节头不完整"）。节表是后续所有 RVA→偏移转换的基础数据。

### 4. RVA → 文件偏移：`PEParserUtils.RvaToOffset`

这是整个 PE 解析层最核心的共享原语。PE 文件中绝大多数内部引用都是 **RVA（相对虚拟地址，相对镜像基址的内存地址）**，而磁盘读取需要**文件偏移**。两者因"节在内存中按 `SectionAlignment` 对齐、在文件中按 `FileAlignment` 对齐"而不同。

签名：

```csharp
internal static long RvaToOffset(uint rva, List<IMAGE_SECTION_HEADER> sections, uint requiredLength = 1)
```

算法（逐节匹配）：

1. 基本校验：`rva == 0` 或节表为空 → 返回 `-1`。
2. 遍历每个节，跳过 `SizeOfRawData == 0` 或 `PointerToRawData == 0` 的节（无文件数据）。
3. 计算节的内存大小 `sectionVirtualSize`：当 `VirtualSize != 0` 用之；**否则回退用 `SizeOfRawData`**。这是为兼容部分旧链接器生成的 `VirtualSize == 0` 的合法节——否则该节内的 RVA 会被误判为无法解析，导致导入/导出/CLR 数据被静默丢失。
4. 命中判断：`rva >= VirtualAddress && rva < VirtualAddress + sectionVirtualSize`。
5. 命中后算节内相对偏移 `relativeOffset = rva - VirtualAddress`。
6. **`requiredLength` 边界校验**：`relativeOffset + requiredLength > SizeOfRawData` → 返回 `-1`。该参数表示"从此 RVA 起需要连续读取的字节数"，作用有二：
   - 排除落在未初始化数据节（如 `.bss`，`VirtualSize > SizeOfRawData` 的尾部）中的 RVA；
   - 避免"跨节读取"——即从本节末尾越界读到下一节的数据。
7. 返回 `PointerToRawData + relativeOffset`（用 `long` 运算防溢出）。

`requiredLength` 默认 1（只要求该字节自身有效）。导出表调用时显式传入 `PEConstants.ExportDirectorySize`（40），保证整个导出目录结构都在同一节的文件数据内才接受。**设计决策**：`RvaToOffset` 只读取 32 位节表字段，因此**与 PE 位数无关**，导入/导出/CLR/资源解析全部复用同一实现，避免每个解析器各自实现一遍 RVA 转换（这类逻辑一旦各自实现极易产生不一致的边界 bug）。

---

### 5. 其他共享读取原语（`PEParserUtils`）

- `ReadAtOffset<T>(FileStream fs, long offset, T fallback, Func<T> read)`：定位到 `offset` 执行 `read()`，用 `try/finally` 保证（含异常时）恢复原 `fs.Position`；`offset` 无效（`<0` 或 `>= Length`）则直接返回 `fallback` 且不移动流。异常不在此捕获，向上传播。这是"临时跳到某偏移读一段、读完归位"模式的统一封装，被导入/导出反复使用以避免重复的保存/恢复样板代码。
- `ReadNullTerminatedString(BinaryReader)`：读取以 `\0` 结尾的 ASCII 字符串，上限 `MaxLength = 1024`（防畸形文件无终止符导致超长读取）。对字节做净化：可打印 ASCII（32–126）与 `\t \n \r` 原样保留，其余替换为 `'?'`。
- `ReadUnicodeStringWithMaxLength` / `ReadUnicodeStringWithMaxBytes`：读取 UTF-16 字符串（按 `ushort` 逐字符），主要供版本资源解析使用。
- `Is64Bit` / `Is32Bit`：以 `OptionalHeader.Magic` 是否等于 `Pe32PlusMagic` / `Pe32Magic` 判定位数。
- `AlignTo4(long)`：`(position + 3) & ~3`，向上对齐 4 字节，供元数据流名称与版本资源等使用。

---

### 6. 导入表解析：`PEImportParser`

#### 6.1 入口与两类导入表

`ParseImportTable(fs, reader, peInfo)` 串行解析两类导入并合并到 `peInfo.ImportFunctions`：

1. `ParseStandardImportTable` —— 标准导入表（`DirectoryImport = 1`）。
2. `ParseDelayLoadImportTable` —— 延迟加载导入表（`DirectoryDelayImport = 13`），其返回列表 `AddRange` 进主列表，函数项带 `IsDelayLoaded = true` 标记。

整个入口外层 `try/catch`（仅捕获 IO 类异常）保证导入解析失败不影响其他目录。

#### 6.2 标准导入表流程

1. 校验数据目录索引存在且 `DirectoryImport` 的 `VirtualAddress != 0`。
2. `importOffset = RvaToOffset(importRVA, sections)`，无效或越界则返回。
3. 保存 `fs.Position`，跳到 `importOffset`，**循环读取 `IMAGE_IMPORT_DESCRIPTOR`（每个 20 字节，`ImportDescriptorSize`）直到遇到全零终止符**（`IsTerminatorDescriptor`：五个字段全 0）。读取前每轮校验 `fs.Position + 20 <= fs.Length`。
4. 对每个描述符：用 `ReadStringAtRva(... importDesc.Name)` 把 DLL 名称 RVA 解析为以 null 结尾的字符串；名称 RVA 无法解析（返回 `-1`）则 `continue` 跳过该描述符。
5. 非空 DLL 名经 `AddDependencyIfMissing` 去重（`OrdinalIgnoreCase`）加入 `peInfo.Dependencies`。
6. 调 `ParseImportFunctions` 解析该 DLL 的导入函数。
7. `finally` 恢复 `fs.Position`。

`ReadStringAtRva` 是一个内聚的小工具：`RvaToOffset` → 失败返回 `(-1, "")`，否则用 `ReadAtOffset` + `ReadNullTerminatedString` 读取并自动归位，返回 `(offset, value)`。

#### 6.3 ILT/IAT 选择与 thunk 遍历

`ParseImportFunctions` 决定使用哪张 thunk 表：

```csharp
uint thunkRVA = importDesc.OriginalFirstThunk != 0 ? importDesc.OriginalFirstThunk : importDesc.FirstThunk;
```

即**优先 `OriginalFirstThunk`（ILT，Import Lookup Table）**，为 0 时回退到 `FirstThunk`（IAT，Import Address Table）。ILT 在磁盘上保留原始的名称/序号信息，是优先选择；IAT 在加载后会被写成真实地址，但在磁盘镜像中通常与 ILT 内容相同，故作为回退可用。

随后 `is64Bit = PEParserUtils.Is64Bit(...)`，通过 `ReadAtOffset` 调用统一遍历器 `WalkThunkTable`。

#### 6.4 统一遍历器 `WalkThunkTable`

```csharp
private static List<ImportFunctionInfo> WalkThunkTable(
    FileStream fs, BinaryReader reader, PEInfo peInfo,
    long thunkTableOffset, bool is64Bit, string dllName, bool isDelayLoaded)
```

算法：

- 步长 `thunkSize = is64Bit ? 8 : 4`；序号标志位 `ordinalFlag = is64Bit ? OrdinalFlag64 (0x8000000000000000) : OrdinalFlag32 (0x80000000)`。
- 定位 `thunkTableOffset`，循环读取每个 thunk（64 位 `ReadUInt64`、32 位 `ReadUInt32`），直到读到值为 0 的 thunk（表终止）或剩余不足一项。
- 对每个非零 thunk 判定导入方式：
  - **序号导入**（`thunkValue & ordinalFlag != 0`，即最高位为 1）：`IsOrdinalImport = true`，`Ordinal = thunkValue & 0xFFFF`，`FunctionName = "#<序号>"`。
  - **名称导入**：thunk 值作为指向 Hint/Name 表项的 RVA，调用 `ImportByName` 解析函数名。

**设计决策**：标准导入与延迟加载导入**共用同一个 `WalkThunkTable`**。注释明确指出：曾因两处各自实现 thunk 步长/序号位逻辑而导致"延迟加载名表步长 off-by-one"的 bug，统一为一个实现后从根本上杜绝两处逻辑漂移。`isDelayLoaded` 仅作为标记透传到每个 `ImportFunctionInfo`。

#### 6.5 名称导入解析 `ImportByName`

- `nameOffset = RvaToOffset((uint)thunkRva, sections)`，无效则函数名记为 `INVALID_RVA_<hex>`。
- 校验 `fs.Length - nameOffset > 2`（Hint/Name 表项至少含 2 字节 Hint），否则记 `NAME_TOO_SHORT`。
- 保存位置 → 跳到 `nameOffset` → 读 2 字节 `hint`（存入 `Ordinal`）→ `ReadNullTerminatedString` 读函数名 → 恢复位置。空名记 `EMPTY_NAME`，读取异常记 `READ_ERROR: <msg>`。

这种"出错也产出一个带诊断名称的条目而非丢弃"的策略，便于在 UI 中暴露异常导入而不是静默吞掉。

#### 6.6 延迟加载导入表

`ParseDelayLoadImportTable`：

1. 校验 `DirectoryDelayImport` 存在、`VirtualAddress != 0`、`RvaToOffset` 有效。
2. 按 `DelayLoadDescriptorSize = 32` 步进遍历 `IMAGE_DELAYLOAD_DESCRIPTOR`（字段含 `Attributes`、`DllNameRVA`、`ModuleHandleRVA`、`ImportAddressTableRVA`、`ImportNameTableRVA`、`BoundImportAddressTableRVA`、`UnloadInformationTableRVA`、`TimeDateStamp`）。循环条件确保每个描述符整体落在文件内。
3. 用 `ReadStringAtRva(... DllNameRVA)` 取 DLL 名：RVA 无效 → `break`（结束扫描，没有显式全零终止符，靠名称无效与边界终止）；名称为空 → `continue`。
4. 名称去重加入依赖，再用 `ImportNameTableRVA` 调 `ParseDelayLoadFunctions` → `WalkThunkTable(..., isDelayLoaded: true)` 解析函数。

延迟加载描述符的 RVA 字段历史上既可能是 RVA 也可能是 VA（取决于 `Attributes` 位），本实现按 RVA 处理（通过 `RvaToOffset`）。

#### 6.7 导入数据流小结

`DataDirectory[1/13]` → `RvaToOffset` → 遍历描述符 → 解析 DLL 名（加入 `Dependencies`）→ 选 ILT/IAT → `WalkThunkTable` 区分序号/名称导入 → 产出 `List<ImportFunctionInfo>`（`DllName`、`FunctionName`、`Ordinal`、`IsOrdinalImport`、`IsDelayLoaded`）→ 存入 `peInfo.ImportFunctions`。

### 7. 导出表解析：`PEExportParser`

#### 7.1 定位与导出目录

`ParseExportTable` 检查 `DirectoryExport = 0`，`exportOffset = RvaToOffset(exportRVA, sections, ExportDirectorySize)`（要求整个 40 字节目录都在节内）。定位后读取 `IMAGE_EXPORT_DIRECTORY`（`Characteristics`、`TimeDateStamp`、版本、`Name`、`Base`、`NumberOfFunctions`、`NumberOfNames`、`AddressOfFunctions`、`AddressOfNames`、`AddressOfNameOrdinals`）。`exportRVA` 与 `exportSize` 后续用于转发判定。

#### 7.2 三张并行数组 + 名称映射

导出表由三张 RVA 指向的并行数组构成，分别用辅助方法读取（每项读前校验剩余长度，避免越界）：

- `functionAddresses = ReadUInt32ListAtRva(AddressOfFunctions, NumberOfFunctions)` —— EAT，导出函数 RVA（按导出索引排列）。
- `functionNameRVAs = ReadUInt32ListAtRva(AddressOfNames, NumberOfNames)` —— 指向导出名字符串的 RVA（按名称排序）。
- `nameOrdinals = ReadUInt16ListAtRva(AddressOfNameOrdinals, NumberOfNames)` —— 与 `functionNameRVAs` 一一对应的"无偏序号"（指向 EAT 的下标）。

`BuildExportFunctionNameMap` 把名字关联回函数下标：取 `min(nameRVAs.Count, ordinals.Count)` 条，对每条用 `RvaToOffset` + `ReadNullTerminatedString` 读名字，建立 `Dictionary<int, string>`，键为 `ordinals[i]`（即 EAT 下标），值为函数名。**这一步是导出表"按序号导出 vs 按名称导出"的关键**：EAT 中所有项都是导出函数，但只有出现在名称表中的项才有名字，其余只能通过序号引用。

#### 7.3 逐项构造导出函数与转发导出

`BuildExportFunction` 对 EAT 第 `i` 项：

- `Ordinal = i + Base`（`Base` 是导出序号基准，通常为 1）。
- `RVA = functionAddresses[i]`。
- 名字：`functionNames.TryGetValue(i, ...)`；无名字则记为 `Ordinal_<序号>`（仅按序号导出）。
- **转发导出判定**：若 `functionRVA != 0 && exportRVA <= functionRVA < exportRVA + exportSize`，即函数 RVA 落在导出目录自身范围内，则该项不是代码地址，而是指向形如 `"DLL.Function"` 的转发字符串。用 `ReadForwarderString` 读出后，名字改写为 `"<名字> (forwarded -> <目标>)"`。

逐项结果加入 `peInfo.ExportFunctions`（`ExportFunctionInfo`：`Name`、`Ordinal`、`RVA`，并有 `OrdinalDisplay` 同时显示十进制与十六进制）。

#### 7.4 导出数据流小结

`DataDirectory[0]` → 读导出目录 → 三张并行数组 → 名称映射（序号↔名字）→ 逐 EAT 项构造（序号 + 名字/`Ordinal_x` + 转发目标）→ `peInfo.ExportFunctions`。

### 8. CLR / .NET 解析：`PEParserCLR`

仅当 `DirectoryClrHeader = 14` 存在且非零时才进入；非 .NET 的原生 PE 此目录为空，`peInfo.CLRInfo` 保持 `null`。

#### 8.1 `IMAGE_COR20_HEADER`（`PEParser.CLR.cs`）

`ParseCLRHeaderInfo` → `RvaToOffset` → `ParseCLRHeader`：校验剩余 `>= Cor20HeaderMinSize (72)` 后，`ReadCor20Header` 顺序读出 72 字节结构：`cb`、运行时主/次版本、`MetaData`（数据目录）、`Flags`、`EntryPointTokenOrRva`、随后 6 个数据目录（`Resources`、`StrongNameSignature`、`CodeManagerTable`、`VTableFixups`、`ExportAddressTableJumps`、`ManagedNativeHeader`）。

`BuildClrInfo(clrHeader, machine)` 把磁盘头映射为展示用 `CLRInfo`，按 ECMA-335 / COR 头解释 `Flags` 标志位：

- `IsILonly = Flags & 0x1`
- `Is32BitRequired = Flags & 0x2`
- `IsStrongNameSigned = Flags & 0x8`
- `Is32BitPreferred = Flags & 0x00020000`（`COMIMAGE_FLAGS_32BITPREFERRED`）
- `HasMetaData/HasResources/HasStrongNameSignature` 由对应数据目录的 `VirtualAddress != 0` 判定。
- `PEMachineType = machine`（保留 PE 头机器类型用于更准确判定架构）。

`CLRInfo.Architecture` 由 `(Is32BitRequired, Is32BitPreferred)` 表驱动派生：`(true, _) → "x86"`、`(false, true) → "x86"`、`(false, false) → "Any CPU"`。`FlagDescriptions` 把布尔标志聚合为人类可读列表。

随后若 `MetaData.VirtualAddress != 0`，调用 `ParseMetaData` 解析元数据以获取公开类型名。

#### 8.2 元数据头与流目录（`PEParser.CLR.Metadata.cs`）

`ParseMetaData(... metaDataRVA)`：

1. `RvaToOffset` 定位元数据根。
2. 读元数据头：`signature`（必须为 `0x42534A42`，即 "BSJB"）、Major/Minor、Reserved、`length`（版本字符串长度，已 4 字节对齐）。签名不符直接返回。
3. **跳过版本字符串**：`fs.Position = versionStart + length`（先把 `length` 收敛到不超过文件剩余）。
4. 读 `Flags`(2) 与 `streams`(2，流数量)。
5. `CollectStreamDirectory` 先**遍历全部流目录项**收集位置——因为 `#Strings` 堆可能排在 `#~` 表流之后，必须先读全才能保证两者都拿到。
6. 若同时找到表流与 `#Strings` 堆，调 `ParseMetadataTables`。

`CollectStreamDirectory`：每个流目录项为 `offset`(4) + `size`(4) + 以 null 结尾、按 4 字节对齐的流名。返回 `(tablesStreamOffset, tablesStreamSize, stringHeapOffset)`：

- 流名为 `#~` 或 `#-` → 表流（`#~` 为压缩/优化表，`#-` 为未优化表，二者结构兼容）。
- 流名为 `#Strings` → 字符串堆。
- 偏移均为 `metaDataOffset + offset`（相对元数据根）。

`ReadStreamName` 逐字节读名（上限 32 字节，ECMA-335 规定的流名最长），再用 `AlignTo4` 对齐到 4 字节边界。

#### 8.3 元数据表流 `#~` 表头（`ParseMetadataTables`）

1. 校验表流落在文件内且 `size >= 24`。
2. 读 24 字节表头：Reserved(4)、Major(1)、Minor(1)、`heapSizes`(1)、Reserved(1)、`maskValid`(8，64 位掩码：每位表示对应表是否存在)、MaskSorted(8)。
3. 对 0..63 每个置位的表（`IsTablePresent(maskValid, i)`）读取一个 4 字节行数到 `rowCounts[i]`。行数数组之后即 `tablesDataOffset`（各表行数据的起始）。
4. 若 TypeDef 表（索引 2）存在，调用 `ParseTypeDefTable`。

`heapSizes` 的位含义（`PEParser.CLR.Helpers.cs` 的 `HeapIndexSize`）：bit0=`#Strings`、bit1=`#GUID`、bit2=`#Blob`，对应位为 0 时该堆索引为 2 字节，否则 4 字节。

#### 8.4 可变行宽与编码索引（ECMA-335 II.22 / II.24.2.6）

这是 CLR 解析最精细的部分。每个元数据表的行宽**不是固定的**，取决于堆大小标志与各表行数（索引可能 2 或 4 字节）。要读到 TypeDef 表的某一行，必须先算出它前面 Module、TypeRef 表的行宽以跳过它们，再算出 TypeDef 自身的行宽以正确步进。

`ParseTypeDefTable` 计算各索引宽度：

- `stringIndexSize = HeapIndexSize(heapSizes, 0)`，`guidIndexSize = HeapIndexSize(heapSizes, 1)`。
- `resolutionScopeSize = CodedIndexSize(2, rowCounts, ResolutionScopeTables)`，其中 `ResolutionScopeTables = [0,26,35,1]`（Module/ModuleRef/AssemblyRef/TypeRef）。
- `typeDefOrRefSize = CodedIndexSize(2, rowCounts, TypeDefOrRefTables)`，`TypeDefOrRefTables = [2,1,27]`（TypeDef/TypeRef/TypeSpec）。
- `fieldIndexSize = SimpleIndexSize(rowCounts, 4)`，`methodIndexSize = SimpleIndexSize(rowCounts, 6)`。

索引宽度规则：

- **简单索引**（`SimpleIndexSize`）：目标表行数 `< 2^16` 时 2 字节，否则 4 字节。
- **编码索引**（`CodedIndexSize(tagBits, ...)`）：索引族中各表最大行数 `< 2^(16 - tagBits)` 时 2 字节，否则 4 字节（tag 位占用一部分编码空间，故阈值下降）。

各相关表行宽（ECMA-335 II.22）：

```
moduleRowSize  = 2 + stringIndexSize + 3 * guidIndexSize        // Generation + Name + Mvid/EncId/EncBaseId
typeRefRowSize = resolutionScopeSize + 2 * stringIndexSize       // ResolutionScope + Name + Namespace
typeDefRowSize = 4 + 2 * stringIndexSize + typeDefOrRefSize
                 + fieldIndexSize + methodIndexSize              // Flags + Name + Namespace + Extends + FieldList + MethodList
```

`ComputeTypeDefStart`：从 `tablesDataOffset` 起，若 Module(0) 存在则加 `rowCounts[0] * moduleRowSize`，若 TypeRef(1) 存在则加 `rowCounts[1] * typeRefRowSize`，得到 TypeDef(2) 表数据起点。

#### 8.5 逐行读取 TypeDef 与公开类型收集

对 `i = 0 .. typeDefCount-1`，`rowStart = typeDefStart + i * typeDefRowSize`，校验该行整体在文件内（否则 `break`），定位后调 `TryCollectTypeDefRow`：

- 读 `Flags`(4)、`nameIndex`（按 `stringIndexSize`）、`namespaceIndex`（按 `stringIndexSize`）；后续 Extends/FieldList/MethodList **无需读取**，下一行直接由 `rowStart` 算术推进。
- `visibility = Flags & 0x7`（VisibilityMask）；仅当 `Public(1)` 或 `NestedPublic(2)` 才收集，其余跳过。
- 用 `ReadStringFromHeap` 从 `#Strings` 堆按索引读 `typeName` 与 `namespaceName`，拼成 `namespace.type`（无命名空间则只用类型名）。
- 非空全名作为一条 `ExportFunctionInfo` 加入 `peInfo.ExportFunctions`（`Name = 全名`，`Ordinal = i`，`RVA = 0`，因为 .NET 类型无 RVA）。

**设计决策**：.NET 程序集没有传统导出表，但用户期望在"导出"视图中看到公开类型。这里**复用 `ExportFunctions` 列表**承载公开 TypeDef 全名，统一了原生导出与托管类型的展示通道。

`ReadStringFromHeap`（`PEParser.CLR.Helpers.cs`）：`heapOffset + index` 定位（`index == 0` 视为空串），UTF-8 逐字节读到 null（上限 1024 字节），读取后恢复位置；异常时返回 `Unknown_Type_<index>`。

### 9. 依赖路径解析：`DependencyResolver`

把导入解析得到的 DLL **名称**解析为磁盘上的**完整路径**，供依赖树（`DependencyNode`）递归展开。

签名：

```csharp
public static string? Resolve(string dllName, string? baseDir, bool? targetIs64Bit = null)
```

`EnumerateSearchDirs` 按以下顺序惰性产出候选目录，`Resolve` 依次 `Path.Combine + File.Exists`，命中即返回 `Path.GetFullPath`，全不命中返回 `null`：

1. `baseDir`（被分析文件所在目录，最高优先，对应应用目录搜索）。
2. **按目标位数决定 System32/SysWOW64 的先后**（核心设计点）：
   - `targetIs64Bit == false`（32 位目标）：先 `SysWOW64`（32 位系统 DLL）后 `System32`。
   - 否则（64 位或未知）：先 `System32` 后 `SysWOW64`。
3. `Windows\System`（16 位历史目录）、`Windows` 目录。
4. `PATH` 环境变量中的各目录（与 Windows 默认 DLL 搜索顺序一致，置于最后）。

**关键平台细节**：本进程为 64 位，故 `Environment.SystemDirectory` 返回真实的 64 位 `System32`，而 32 位系统 DLL 实际位于 `Windows\SysWOW64`（路径名与直觉相反，这是 WoW64 的设计）。注释明确点出这一点。`dllName` 含非法路径字符时 `Path.Combine` 抛 `ArgumentException`，被捕获后跳过该候选。

`targetIs64Bit` 由调用方 `DependencyNode.EnsureLoaded` 依据当前节点 PE 的 `PEParserUtils.Is64Bit/Is32Bit` 推导（未知则 `null`），从而**用父模块的位数选择正确的系统 DLL 变体**，避免 32 位模块错误匹配到 64 位 DLL。

### 10. 数据流与上层消费

整体数据流：

```
文件路径
  → PEParser.ParsePEFile
      → ParseHeaders（DOS/NT/可选头/节头，含严格校验）
      → ParseContentDirectories（表驱动）
          → 导入（Import/DelayImport）→ ImportFunctions + Dependencies
          → 导出（Export）            → ExportFunctions
          → 版本 / 证书 / 图标（资源层）→ AdditionalInfo / Icons
          → CLR（COR20 + 元数据 TypeDef）→ CLRInfo + ExportFunctions(公开类型)
  → PEInfo
```

`PEInfo`（`PEAnalyzer/Models/PEInfo.cs`）是单一聚合载体，汇集所有解析结果：头部结构、`SectionHeaders`、`ImportFunctions`、`ExportFunctions`、`Dependencies`、`Icons`、`CLRInfo`、`AdditionalInfo`。

两条主要消费路径：

- **`PEHeaderPresenter`**（`PEAnalyzer/PEHeaderPresenter.cs`）：把 `PEInfo` 转成 `List<(string Title, Dictionary<string,string> Info)>`——即"分组标题 → 键值表"的纯展示数据（文件信息/DOS/NT/可选头/CLR/节，以及版本/公司/版权/证书/翻译）。**设计决策**：把对模型结构与 `PEParserUtils` 的引用从 WPF 控件中剥离到 presenter，降低 UI 控件与解析模型的耦合，便于测试与复用；位数/架构/子系统等的人类可读文案集中在 `PEHeaderDescriptions`。
- **`DependencyNode`**（`UserControls/DependencyNode.cs`）：构建可惰性展开的依赖树。根节点为已打开文件（`CreateRoot` 直接持有 `PEInfo`）；首次展开（`EnsureLoaded`）时对子依赖逐个 `DependencyResolver.Resolve`（传入父模块位数），再 `PEParser.ParsePEFile` 解析路径以取得下一层依赖。内置 `MaxDepth = 16` 限深、`ancestors`（`OrdinalIgnoreCase` 完整路径集）做**环检测**（循环依赖标记为"循环依赖"而不再递归），未解析到路径标记"未找到"。占位"加载中..."子节点用于在 UI 上显示可展开的"+"。

### 11. 贯穿全层的设计决策总结

1. **编排器 + 按职责拆分的解析类**：`PEParser` 仅编排，`PEHeaderParser`/`PEImportParser`/`PEExportParser`/`PEParserCLR` 各管一类，单文件聚焦单职责，便于独立演进与维护。
2. **表驱动派发**：`ContentParsers` 委托数组使"内容目录"可插拔；每步独立 `try/catch`，做到**部分失败隔离**——一个目录损坏不拖垮整体。
3. **共享读取原语**：`RvaToOffset`（与位数无关）、`ReadAtOffset`（定位即归位）、`ReadNullTerminatedString` 等被所有解析器复用，消除重复实现与随之而来的不一致 bug（如曾经的延迟加载 thunk off-by-one）。
4. **统一 thunk 遍历**：标准/延迟导入共用 `WalkThunkTable`，序号位与步长逻辑只此一份。
5. **位数差异收敛**：PE32/PE32+ 的字段宽度差异集中在 `ReadImageBaseFields`/`ReadStackHeapFields` 与统一的 `IMAGE_OPTIONAL_HEADER`（`ulong` 字段）中，其余代码与位数无关。
6. **防御性边界校验贯穿全程**：每次定位读取前校验 `Position + size <= Length`、`RvaToOffset` 的 `requiredLength` 边界、数据目录数量收敛到实际可读项、字符串读取长度上限——共同抵御截断与畸形文件触发的未捕获异常。
7. **presenter 解耦展示**：`PEHeaderPresenter` 把模型→展示数据的转换从 UI 控件中抽离，控件不直接依赖解析模型与 `PEParserUtils`。
```


## 3 PE 资源解析与展示


本节描述 PersonTools 中 PE 文件“资源段（`.rsrc`）”及附属信息（图标、版本信息、数字证书）的解析子系统，以及把解析结果转换为 UI 展示数据的表示层（presenter）。涉及目录 `PEAnalyzer/Resources/` 与 `PEAnalyzer/` 下的若干文件。

### 1. 总体架构与设计决策

整个资源子系统遵循“**编排器 + 按职责拆分的解析类 + 共享读取原语**”的分层结构：

- **编排入口**：`PEParser.ParsePEFile`（`PEAnalyzer/Parsers/PEParser.cs`）先解析头部，再以**表驱动派发**方式逐个执行内容目录解析器。`ContentParsers` 是一个 `Action<FileStream, BinaryReader, PEInfo>[]` 数组，其中包含 `PEResourceParserVersion.ParseVersionInfo`、`PEResourceParserCertificate.ParseCertificateInfo`、`PEResourceParserIcon.ParseIconInfo` 三个资源相关步骤。每个步骤自行 `try/catch`，互不影响——任何一项解析失败都不会中断其余项或整个文件解析。把依赖以“数据数组”而非硬编码调用序列表达，便于增删步骤、降低核心方法耦合度。
- **按资源类型拆分**：图标、版本、证书各成独立静态类；图标与版本又进一步按“目录遍历 / 数据项 / 子结构 / 辅助函数”拆成多个 `partial`-风格的同名前缀文件（如 `PEResourceParser.Icon.*`、`PEResourceParser.Version.*`）。这样每个文件圈复杂度与维护成本都较低（与项目的可维护指数优化目标一致）。
- **共享读取原语**：所有资源目录树遍历集中到 `ResourceDirectoryReader`；所有 `VS_VERSIONINFO` 节点（6 字节头 + Unicode 键 + 4 字节对齐兄弟边界）的读取集中到 `VersionNodeReader`。各解析器只关心“命中某类型/键后做什么”，不再各自手写目录遍历与对齐逻辑，避免重复与不一致。
- **表示层分离**：`PEHeaderPresenter` 把 `PEInfo` 模型转成“分组标题 → 键值表”，`PEHeaderDescriptions` 提供字段的人类可读描述，`IconImageFactory` 把 `IconInfo` 原始字节解码为可显示的 `IconViewModel`。WPF 控件 `PEAnalyzerControl` 只消费这三者的输出，不直接接触二进制模型与解码细节。

#### 共用的底层原语（`PEParserUtils.cs`）

资源解析反复依赖以下原语（位于 `PEAnalyzer/Parsers/PEParserUtils.cs`）：

- **`RvaToOffset(uint rva, List<IMAGE_SECTION_HEADER> sections, uint requiredLength = 1)`**：把 RVA 转为文件偏移。逐节判断 `rva ∈ [VirtualAddress, VirtualAddress + 节内存大小)`；节内存大小取 `VirtualSize`，为 0 时回退用 `SizeOfRawData`（兼容旧链接器）。命中后偏移 = `PointerToRawData + (rva - VirtualAddress)`，并要求 `relativeOffset + requiredLength ≤ SizeOfRawData`（排除 `.bss` 等仅内存节、防止跨节越读）。失败返回 `-1`。资源目录项里的 `OffsetToData`（数据条目）保存的是 RVA，必须经此转换。
- **`AlignTo4(long position)` = `(position + 3) & ~3`**：4 字节向上对齐，版本信息节点边界全靠它。
- **`ReadUnicodeStringWithMaxBytes` / `ReadUnicodeStringWithMaxLength`**：从当前位置读 UTF‑16LE 字符串，遇 `NUL` 或达到上界停止；内部 `maxReadAttempts` 限制在 1000 字符内，防止畸形文件无终止符导致死循环。

### 2. 资源目录树遍历（`ResourceDirectoryReader`）

文件：`PEAnalyzer/Resources/ResourceDirectoryReader.cs`。

#### 2.1 职责

封装 `IMAGE_RESOURCE_DIRECTORY` / `IMAGE_RESOURCE_DIRECTORY_ENTRY` / `IMAGE_RESOURCE_DATA_ENTRY` 三种结构的读取，以及资源目录树的安全递归遍历。是图标与版本解析器共用的“目录层”原语，避免各处重复手写遍历逻辑。

PE 资源是一棵典型的三层树：**类型层 → 名称/ID 层 → 语言层**，叶子是 `IMAGE_RESOURCE_DATA_ENTRY`。每层都是一个 `IMAGE_RESOURCE_DIRECTORY`（头 16 字节）后跟若干 `IMAGE_RESOURCE_DIRECTORY_ENTRY`（每项 8 字节）。

#### 2.2 关键常量与字段

- `DirectoryHeaderSize = 16`、`DirectoryEntrySize = 8`：两种结构的字节大小。
- `MaxDirectoryDepth = 32`：最大递归深度（正常 PE 仅 3 层）。
- `[ThreadStatic] HashSet<long>? _walkPath`、`[ThreadStatic] int _walkDepth`：**当前递归路径上的目录偏移集合与深度**，用于环路/深度防护。`[ThreadStatic]` 保证多线程下互不干扰，单点防护即可覆盖全部经由 `WalkEntries` 的调用方。

#### 2.3 关键方法与签名

- `IMAGE_RESOURCE_DIRECTORY ReadDirectory(BinaryReader)`：从当前位置读 16 字节目录头（`Characteristics`/`TimeDateStamp`/`Major|MinorVersion`/`NumberOfNamedEntries`/`NumberOfIdEntries`）。
- `bool TryReadEntry(FileStream fs, BinaryReader reader, long directoryOffset, int index, out IMAGE_RESOURCE_DIRECTORY_ENTRY entry)`：读第 `index` 项。条目偏移 = `directoryOffset + 16 + index*8`；越界（`entryOffset + 8 > fs.Length`）返回 `false`。条目两字段为 `NameOrId` 与 `OffsetToData`。
- `IMAGE_RESOURCE_DATA_ENTRY ReadDataEntry(BinaryReader)`：读叶子数据条目 16 字节（`OffsetToData`(RVA)/`Size`/`CodePage`/`Reserved`）。
- `bool IsReadableData(long dataOffset, uint size, FileStream fs)`：判定 RVA 转换后的数据偏移 + 大小是否落在文件内、可安全读取（`dataOffset ≥ 0 && 0 < size ≤ int.MaxValue && dataOffset + size ≤ fs.Length`）。

##### 核心算法：`WalkEntries` 的递归与环路防护

`void WalkEntries(fs, reader, long directoryOffset, long resourceBaseOffset, Action<long> onSubdirectory, Action<long> onDataEntry)` 是树遍历的核心：

1. 边界校验：`directoryOffset < 0` 或 `directoryOffset + 16 > fs.Length` 直接返回。
2. **环路/深度防护**：惰性初始化 `_walkPath`；若 `_walkDepth ≥ 32`，或 `_walkPath.Add(directoryOffset)` 返回 `false`（说明该偏移已在当前递归路径上，**成环**），立即返回。
3. `_walkDepth++`，定位到 `directoryOffset` 读目录头，计算 `totalEntries = NumberOfNamedEntries + NumberOfIdEntries`。
4. 逐项 `TryReadEntry`；读失败即 `break`。对每项判断 `OffsetToData` 最高位（`0x80000000`）：
   - **置位** → 这是“子目录”项，子目录偏移 = `resourceBaseOffset + (OffsetToData & 0x7FFFFFFF)`（**低 31 位是相对资源段基址的偏移**），回调 `onSubdirectory`。
   - **未置位** → 这是“数据条目”项，偏移 = `resourceBaseOffset + OffsetToData`（**完整 32 位**），回调 `onDataEntry`。
5. `finally` 中 `_walkDepth--`；若回到顶层（`_walkDepth == 0`）则把 `_walkPath` 置 `null` 释放；否则 `_walkPath.Remove(directoryOffset)`——**离开本层后允许兄弟分支合法地再次访问相同偏移（支持 DAG，仅禁止真正成环）**。

这是整个目录层防止畸形/循环资源树导致无限递归与 `StackOverflow` 的关键，且因所有递归都经由 `WalkEntries` 分派回调，单点防护即可覆盖图标与版本两条解析链。

##### 顶层定向扫描：`ScanTypeEntries` / `ScanNamedEntries`

不走完整递归，只在“根目录”按条目筛选，用于快速定位某资源类型：

- `bool ScanTypeEntries(fs, reader, long resourceOffset, int totalEntries, uint typeId, Action<long> onMatch, bool stopAtFirst = false)`：遍历根条目，比较 `(NameOrId & 0xFFFF) == typeId`。命中则计算下一级偏移 `resourceOffset + (OffsetToData & 0x7FFFFFFF)`（**低 31 位**），在 `[0, fs.Length)` 内回调 `onMatch`。返回是否命中过；`stopAtFirst=true` 时命中首个即停（版本资源只取第一个 `RT_VERSION`）。
- `void ScanNamedEntries(fs, reader, long resourceOffset, int namedEntries, Action<long> onMatch)`：只遍历前 `namedEntries` 项（命名条目位于目录前部），筛选 `NameOrId` 最高位置位（`0x80000000`，即“按名称命名”），命中则同样按低 31 位下钻回调。

##### 受保护定位执行：`ReadAtOffset<T>` / `RunAtOffset`

把“临时跳转到某偏移读取、再恢复流位置”的高频模式封装为两个原语，统一边界与异常语义：

- `T ReadAtOffset<T>(fs, long offset, int minBytes, T fallback, Func<T> read)`：若 `offset < 0` 或 `offset + minBytes > fs.Length` 直接返回 `fallback` 且不移动流；否则保存 `originalPosition`，跳转执行 `read`，**成功后恢复原位置并返回结果**；捕获 `IOException`/`UnauthorizedAccessException`/`ArgumentOutOfRangeException` 时返回 `fallback`（异常路径不恢复流位置，与图标查找的历史语义一致）。
- `void RunAtOffset(fs, long offset, int minBytes, string errorContext, Action action)`：`void` 版本，异常时以 `errorContext` 写 `Console.WriteLine` 记录日志（保留各 void 图标解析方法原有 `try/catch + Console.WriteLine` 语义）。

> 注意此处的 `ReadAtOffset` 带 `minBytes` 参数，与 `PEParserUtils.ReadAtOffset`（无 `minBytes`、异常向上抛）是两套不同语义的同名工具，分别服务资源层与导入/导出层。

### 3. 图标资源解析

图标解析涉及多个文件，分工如下：

| 文件 | 职责 |
| --- | --- |
| `PEResourceParser.Icon.cs` | 入口编排：定位资源目录、分派 GROUP_ICON / ICON / 命名 / .NET 路径 |
| `PEResourceParser.Icon.Group.cs` | `RT_GROUP_ICON` 解析、`ICONDIR` 组装、生成独立 `.ico` 字节 |
| `PEResourceParser.Icon.Direct.cs` | 直接 `RT_ICON` 资源递归遍历 |
| `PEResourceParser.Icon.Named.cs` | 命名图标资源（WPF/.NET 常用）解析 |
| `PEResourceParser.Icon.Data.cs` | 图标数据项读取、ICO 识别、DIB→ICO 转换 |
| `PEResourceParser.Icon.Helpers.cs` | 按资源 ID 定位图标数据偏移、读资源名 |

最终产物写入 `peInfo.Icons`（`List<IconInfo>`，模型在 `PEAnalyzer/Models/IconInfo.cs`：`Width`/`Height`/`BitsPerPixel`/`Size`/`Data`(byte[])）。

#### 3.1 入口与派发（`PEResourceParserIcon.ParseIconInfo`）

`ParseIconInfo(FileStream fs, BinaryReader reader, PEInfo peInfo)`：

1. 取数据目录第 `PEConstants.DirectoryResource`(=2) 项的 `VirtualAddress`(资源 RVA)，`RvaToOffset` 转文件偏移。
2. 偏移有效 → 调 `ParseResourceDirectoryForIcons`。
3. 否则若 `peInfo.CLRInfo != null`（.NET 程序集）→ `ParseDotNetIcons`，进而（当 `CLRInfo.HasResources`）走命名资源路径 `ParseResourceDirectoryForNamedIcons`，因为 WPF/.NET 程序通常把图标存为命名资源。

`ParseResourceDirectoryForIcons` 读根目录头后按**优先级派发**：

1. **优先**用 `ScanTypeEntries(..., typeId=14, ...)` 查 `RT_GROUP_ICON`（ID=14），命中交给 `PEResourceParserIconGroup.ParseGroupIconResource`，返回 `foundGroupIcon`。
2. **回退**：未找到组图标时，再用 `ScanTypeEntries(..., typeId=3, ...)` 查 `RT_ICON`（ID=3，`PEResourceParserIconDirect.ParseDirectIconResource`），并用 `ScanNamedEntries` 处理命名资源。

这种“组图标优先、直接图标/命名兜底”的顺序，保证常规 EXE/DLL 走标准 GROUP_ICON→ICON 关联路径，同时覆盖只有零散 `RT_ICON` 或把图标藏在命名资源里的边缘情况。

#### 3.2 组图标（`RT_GROUP_ICON` / `RT_ICON`）解析与 DIB→ICO 组装

文件 `PEResourceParser.Icon.Group.cs`，这是图标解析最核心的算法。

- `ParseGroupIconResource(...)`：用 `RunAtOffset` 保护，调用 `WalkEntries` 递归下钻名称层/语言层，叶子交给 `ParseGroupIconDataEntry`。
- `ParseGroupIconDataEntry(...)`：读 `IMAGE_RESOURCE_DATA_ENTRY`，`OffsetToData`(RVA) → `RvaToOffset` → `dataOffset`；`IsReadableData` 且至少够读 6 字节 ICONDIR 头时，定位并 `ParseIconDirectory`。

##### `ParseIconDirectory` —— 解析 GROUP_ICON 目录

GROUP_ICON 资源体是一个 `GRPICONDIR`：6 字节头（`Reserved`/`Type`/`Count`）后跟 `Count` 个目录项。代码读头后**校验 `Type == 1 && 1 ≤ Count ≤ 256`**（图标类型），再 `TryReadIconDirEntries` 读 `Count` 个条目。

> 实现细节：`TryReadIconDirEntries` 按 `ICONDIRENTRY`（`PEAnalyzer/Models/ICONDIRENTRY.cs`）的 16 字节布局逐项读取（`Width`/`Height`/`ColorCount`/`Reserved`/`Planes`/`BitCount`/`BytesInRes`/`ImageOffset`）。注意 GROUP_ICON 条目末字段在磁盘上其实是 2 字节的资源 ID（`nID`），代码统一按 4 字节 `ImageOffset` 读出后只取其低位作为“资源 ID”使用——见下文。每读一项校验 `fs.Position + 16 ≤ fs.Length`，不足则截断返回 `false`。

##### `ProcessGroupIconEntries` → `TryBuildIcon` —— 关联 RT_ICON 并组装独立 .ico

对每个 `ICONDIRENTRY`：

1. **按资源 ID 定位真实位图数据**：`PEResourceParserIconHelpers.FindIconDataByResourceId(fs, reader, peInfo, entry.ImageOffset, resourceBaseOffset)`。GROUP_ICON 里每个条目的 `nID` 指向一个 `RT_ICON`(ID=3) 资源；该函数先在根目录定位 `RT_ICON` 类型目录，再在其下按 `(NameOrId & 0xFFFF) == (resourceId & 0xFFFF)` 找到对应名称/ID 项，进入语言目录取第一个数据条目，最终返回位图数据的文件偏移（失败 `-1`）。
2. 归一化尺寸：`Width/Height` 为 0 表示 256（ICO 约定 0=256）。
3. 大小校验：在 `long` 上算 `fullIconDataSizeLong = 6 + 16 + BytesInRes`，限制 `< 10MB`，避免 `int` 溢出与超大分配。
4. `ReadIconImageData(...)`：完整可读时读 `BytesInRes` 字节；若 `iconDataOffset + BytesInRes` 越过文件尾，**截断兜底**读取剩余字节（处理被裁剪的畸形文件）。
5. `BuildSingleIconFile(entry, imageData, fullIconDataSize)`：**把单个图标拼成一个独立、可单独保存/解码的 `.ico` 文件**——
   - `ICONDIR`(6B)：`Reserved=0`、`Type=1`、`Count=1`（`fullIconData[2]=0x01`、`[4]=0x01`）。
   - `ICONDIRENTRY`(16B @ offset 6)：复制 `Width/Height/ColorCount/Reserved/Planes/BitCount/BytesInRes`，并把图像数据偏移字段写为 `6+16=22`（`fullIconData[18..]`）。
   - 位图数据 @ offset 22：`Array.Copy(imageData, ...)`。
   - 结果 `IconInfo.Data` 即一份合法 ICO，下游 `IconImageFactory` 可直接用 WPF 解码。

#### 3.3 直接图标（`PEResourceParserIconDirect`）

`ParseDirectIconResource(...)`：在没有 GROUP_ICON 时使用，用 `RunAtOffset` + `WalkEntries` 递归遍历 `RT_ICON` 目录，叶子交给 `PEResourceParserIconData.ReadAndProcessIconDataEntry`——即把每个 `RT_ICON` 数据当作独立图标处理（不依赖 GROUP 关联）。

#### 3.4 命名图标（`PEResourceParserIconNamed`）

针对 .NET/WPF 把图标存为“命名资源”的情况：

- `ParseResourceDirectoryForNamedIcons(...)`：只遍历根目录前 `NumberOfNamedEntries` 项，要求 `NameOrId` 最高位置位（命名），下钻并把 `NameOrId & 0x7FFFFFFF`（名称字符串相对资源基址的偏移）传入 `ParseNamedResourceDirectory`。
- `ParseNamedResourceDirectory(...)`：仅当 `ShouldProcessNamedResource` 通过时才 `WalkEntries` 递归；递归进入子目录时 `nameOffset` 传 0。
- `ShouldProcessNamedResource(...)`：`nameOffset == 0`（已下钻到子目录）始终处理；否则读资源名（`PEResourceParserIconHelpers.ReadResourceName`），仅当名字（不分大小写）含 `icon` / `.ico` / `app` 关键字时才处理。这是一个**启发式过滤**，避免把所有命名资源都当图标递归。
- `ReadResourceName(fs, reader, long nameOffset)`：资源名是“2 字节长度前缀 + UTF‑16 串”，读 `nameLength` 后校验 `fs.Position + nameLength*2 ≤ fs.Length`，再 `Encoding.Unicode.GetString`。

#### 3.5 图标数据识别与 DIB→ICO 转换（`PEResourceParserIconData`）

`ReadAndProcessIconDataEntry(...)`（被直接图标与命名图标共用，消除重复）：读 `IMAGE_RESOURCE_DATA_ENTRY` → `RvaToOffset` → 读 `Size` 字节 → `IsIconData` 判定 → `ProcessIconData`。

- `IsIconData(byte[])`：识别两种形态——**完整 ICO 头**（`00 00 01 00`，`IsCompleteIcoHeader`）或 **DIB**（`BITMAPINFOHEADER`，前 4 字节 `biSize == 40`）。
- `ProcessIconData(...)`：
  - 若已是完整 ICO（首 4 字节 `00 00 01 00`）：直接用原始字节构造 `IconInfo`；若 `Length ≥ 22` 再从其目录项提取 `Width`(offset 6)/`Height`(offset 7)/`BitCount`(offset 12)，0 宽高归一为 256。
  - 否则视为 DIB → `ConvertDibToIco`。
- `ConvertDibToIco(...)`：核心是把单张 DIB 包成 ICO——
  1. 校验 `Length ≥ 40 && biSize(offset 0) == 40`。
  2. `width = int@offset4`；**`height = |int@offset8| / 2`**——图标 DIB 的高度是实际高度的两倍（含 XOR 颜色掩码 + AND 透明掩码两段），负数表示自顶向下，先取绝对值再除 2。
  3. `bitCount = ushort@offset14`；校验 `width/height/bitCount` 均 > 0。
  4. `BuildIcoFile(...)`：写 6 字节 ICO 头（`Type=1`/`Count=1`）+ 16 字节目录项（`Width/Height` 取低字节，`Planes=1`，`BitCount`，`BytesInRes=dib长度`，图像偏移=22）+ DIB 数据，得到完整 `.ico`。

异常用 `Console.WriteLine` 记录但不中断。

### 4. 图标位图工厂（`IconImageFactory`）

文件：`PEAnalyzer/IconImageFactory.cs`。

#### 职责

把解析层产出的 `IconInfo`（含 `.ico` 字节）解码为 UI 可绑定的 `IconViewModel`（`PEAnalyzer/Models/IconViewModel.cs`：`BitmapSource? ImageSource` + `Width/Height/BitsPerPixel/Size`）。把 `BitmapImage`/`MemoryStream` 等 WPF 解码细节移出控件，降低 `PEAnalyzerControl` 的复杂度与耦合。

#### 关键方法

`IconViewModel? TryCreate(IconInfo icon)`：

1. 数据无效（`Data` 为空、`Width ≤ 0` 或 `Height ≤ 0`）返回 `null`。
2. 把 `icon.Data` 包成 `MemoryStream`，构建 `BitmapImage`：`BeginInit` → `StreamSource = stream` → **`CacheOption = OnLoad`（加载完即可释放流）** → `EndInit` → `Freeze()`（冻结使其可跨线程安全、提升性能）。
3. 装入 `IconViewModel` 返回；捕获 `IOException`/`UnauthorizedAccessException`/`ArgumentException` 时记日志返回 `null`（解码失败的图标被静默丢弃，不影响其他图标）。

下游 `PEAnalyzerControl.DisplayIcons` 遍历 `peInfo.Icons`，逐个 `TryCreate`，非 `null` 才加入 `IconsDataGrid.ItemsSource`。

### 5. 版本信息解析（`VS_VERSIONINFO`）

版本信息是资源树中最复杂的嵌套结构，拆为多文件：

| 文件 | 职责 |
| --- | --- |
| `PEResourceParser.Version.cs` | 入口：定位 `RT_VERSION`、递归到数据项、定位 `VS_VERSIONINFO` |
| `VersionNodeReader.cs` | 节点头/键/兄弟/子块的共享读取原语 |
| `PEResourceParser.Version.Helpers.cs` | `VS_VERSIONINFO` 头 + `VS_FIXEDFILEINFO` + 子项分派 |
| `PEResourceParser.Version.String.cs` | `StringFileInfo` 块 |
| `PEResourceParser.Version.Table.cs` | `StringTable` 与字符串对（表驱动写回字段） |
| `PEResourceParser.Version.Var.cs` | `VarFileInfo` / `Var` / `Translation` |
| `PEResourceParser.Version.Language.cs` | 语言 ID / 代码页 → 可读名称 |

结果写入 `peInfo.AdditionalInfo`（`PEAnalyzer/Models/PEAdditionalInfo.cs`），失败时把错误串写入 `FileVersion`。

#### 5.1 `VS_VERSIONINFO` 节点的统一形态（`VersionNodeReader`）

`VS_VERSIONINFO` 及其全部子节点（`StringFileInfo`/`StringTable`/`String`/`VarFileInfo`/`Var`）共享统一二进制形态：

- **6 字节头**：`wLength`（本节点含子节点的总字节数）、`wValueLength`（值的长度，文本节点按字符数/字节数语义不一，代码主要靠它判“有无值”）、`wType`（0=二进制，1=文本），各 2 字节。
- **`szKey`**：紧跟头的 `NUL` 结尾 UTF‑16 串。
- **4 字节对齐**：键后、值后、节点间均按 4 字节对齐；兄弟节点起点 = `AlignTo4(本节点起点 + wLength)`。

`VersionNodeReader` 把这些原语单点封装：

- `bool TryReadNodeHeader(fs, reader, out ushort length, out ushort valueLength, out ushort type)`：`fs.Position + 6 > fs.Length` 返回 `false` 且不动流；否则读 3 个 `UInt16`。
- `string ReadKey(fs, reader, ushort length)`：读 `szKey`，上界取 `min(length, 剩余文件长度)`，转交 `ReadUnicodeStringWithMaxBytes`。
- `void AdvanceToSibling(fs, long startPosition, ushort length, long endPosition)`：跳到 `AlignTo4(startPosition + length)`，**仅当目标落在 `(当前位置, endPosition)` 严格区间内才移动**（防止原地踏步或越界）。
- `void ParseChildBlock(fs, long startPosition, ushort length, long endPosition, Action<long> parseChild)`：进入子块——子块起点 = `AlignTo4(当前位置)`，子块终点 = `min(startPosition + length, endPosition)`；有效时定位到起点并以子块终点回调 `parseChild`。

这是所有版本解析器避免重复手写“6 字节头 + Unicode 键 + 4 字节对齐”的关键。

#### 5.2 入口与定位（`PEResourceParserVersion`）

`ParseVersionInfo(...)`：取资源数据目录(#2) RVA → `RvaToOffset`；有效则 `ParseResourceDirectoryForVersionInfo`，否则把诊断串写入 `AdditionalInfo.FileVersion`（如“文件不包含版本资源”“无效的资源偏移…”）。

- `ParseResourceDirectoryForVersionInfo`：读根目录，`ScanTypeEntries(..., typeId=16, ..., stopAtFirst:true)` 找 `RT_VERSION`（ID=16），**只取第一个**，下钻到 `ParseVersionResource`。
- `ParseVersionResource`：`WalkEntries` 递归名称层/语言层，叶子交 `ParseVersionDataEntry`。
- `ParseVersionDataEntry`：读 `IMAGE_RESOURCE_DATA_ENTRY`，`OffsetToData`(RVA) → `RvaToOffset` → 定位到 `VS_VERSIONINFO` 起点，调 `PEResourceParserVersionHelpers.ParseVersionInfoStructure`。各步异常都写入 `FileVersion` 作为诊断。

#### 5.3 `VS_VERSIONINFO` 头与 `VS_FIXEDFILEINFO`（`Version.Helpers.cs`）

`ParseVersionInfoStructure(fs, reader, peInfo)`：

1. 记 `startPosition`，`TryReadNodeHeader` 读 6 字节头；校验 `wLength ≥ 6 && startPosition + wLength ≤ fs.Length`。
2. `ReadKey` 读 `szKey`，**必须等于 `"VS_VERSION_INFO"`**（忽略大小写），否则返回。
3. `VS_FIXEDFILEINFO`（固定 52 字节）紧随其后并对齐：`alignedPosition = AlignTo4(当前位置)`。
   - 若 `wValueLength == 0`：合法地无 `VS_FIXEDFILEINFO`，直接 `ParseChildrenIfPresent(alignedPosition, startPosition + wLength)`。
   - 否则校验 `wValueLength ≥ 52 && alignedPosition + 52 ≤ fs.Length`，`ReadFixedFileInfo` 读 13 个 `DWORD`。
4. `ApplyFixedFileVersions`：**校验签名 `dwSignature == 0xFEEF04BD`**，否则写“无效签名”诊断；通过则用 `FormatVersion(ms, ls)` 把 MS/LS 两个 DWORD 拆成 `主.次.编译.修订` 写入 `FileVersion`(取自 `dwFileVersionMS/LS`) 与 `ProductVersion`(取自 `dwProductVersionMS/LS`)。
   - `FormatVersion(ms, ls)` = `$"{(ms>>16)&0xFFFF}.{ms&0xFFFF}.{(ls>>16)&0xFFFF}.{ls&0xFFFF}"`。
5. 子项（`StringFileInfo`/`VarFileInfo`）从 `AlignTo4(alignedPosition + 52)` 开始，调 `ParseVersionChildren(..., endPosition = startPosition + wLength)`。

##### `ParseVersionChildren` —— 子块兄弟遍历

`while (fs.Position < endPosition && fs.Position + 6 ≤ fs.Length)` 循环：

1. 记 `childStartPos`，`TryReadNodeHeader` 读 `wLength`（为 0 即 `break`）。
2. `ReadKey` 读键，**再把流位置重置回 `childStartPos`** 让具体解析器从节点头重新读起。
3. 按键名分派：`"StringFileInfo"` → `PEResourceParserVersionString.ParseStringFileInfo`；`"VarFileInfo"` → `PEResourceParserVersionVar.ParseVarFileInfo`。
4. 前进到下一兄弟：`nextChildPos = (childStartPos + wLength + 3) & ~3`（即 `AlignTo4`），若 `≥ endPosition` 或回退（`< fs.Position`）则 `break`。

#### 5.4 `StringFileInfo` / `StringTable` / `String`

- `ParseStringFileInfo(fs, reader, peInfo, endPosition)`（`Version.String.cs`）：读头校验 `wLength ≥ 6`，确认键为 `"StringFileInfo"`，用 `ParseChildBlock` 进入其子块并对每个 `StringTable` 调 `PEResourceParserVersionTable.ParseStringTable`；末尾 `AdvanceToSibling`。
- `ParseStringTable(fs, reader, peInfo, endPosition)`（`Version.Table.cs`）：读头后 `ReadKey` 读“语言+代码页”8 位十六进制键（这里读出后丢弃，仅用于推进位置），`stringsPosition = AlignTo4(当前位置)`，`tableEndPosition = min(startPosition + wLength, endPosition)`。然后 `while` 循环逐个 `ParseStringPair`，**若某次解析未推进位置（`fs.Position <= before`）则 `break`**，防止死循环。
- `ParseStringPair(fs, reader, peInfo, endPosition) → bool`：读单个 `String` 节点头，校验 `wLength != 0 && startPosition + wLength ≤ endPosition`；`blockEnd = min(startPosition + wLength, endPosition)`。
  1. `ReadUnicodeWithin(blockEnd)` 读键名。
  2. 值对齐：`valuePosition = AlignTo4(当前位置)`，越界返回 `false`。
  3. 定位到值，`ReadUnicodeWithin(blockEnd)` 读值。
  4. **表驱动写回**：查 `VersionFieldSetters`（`Dictionary<string, Action<PEAdditionalInfo, string>>`，`OrdinalIgnoreCase`）；命中则 `setter(peInfo.AdditionalInfo, value)`。映射包括 `CompanyName`/`FileDescription`/`FileVersion`/`ProductName`/`ProductVersion`/`LegalCopyright`/`LegalTrademarks`/`OriginalFilename`(→`OriginalFileName`)/`InternalName`/`Copyright`。**此表取代了原先的大型 `switch`**，把圈复杂度降为 1，新增字段只需加一行。
  5. `AdvanceToSibling`，返回 `true`。
  - `ReadUnicodeWithin(fs, reader, limit)`：`maxBytes = max(0, min(limit, fs.Length) - fs.Position)`，读到 `NUL` 或上界。

#### 5.5 `VarFileInfo` / `Var` / `Translation`（`Version.Var.cs`）

- `ParseVarFileInfo(...)`：与 `StringFileInfo` 对称，确认键为 `"VarFileInfo"`，`ParseChildBlock` 进入子块调 `ParseVar`，末尾 `AdvanceToSibling`。
- `ParseVar(...)`：`while (fs.Position < endPosition)` 循环读 `Var` 节点，`wLength == 0` 时跳到 `AlignTo4(startPosition)` 后结束；否则 `ParseSingleVar`，返回 `false` 即停。
- `ParseSingleVar(...)`：`ReadKey` 读变量名（通常 `"Translation"`）；**值位置按读键后的实际流位置 `AlignTo4` 推算，而非按字符串长度推算**（避免键名被截断或内嵌 `NUL` 导致错位）；定位到值调 `ApplyTranslationIfPresent`；再 `AlignTo4(startPosition + wLength)` 前进到下一 `Var`。
- `ApplyTranslationIfPresent(...)`：仅当变量名为 `"Translation"` 且 `wValueLength ≥ 4` 且值在文件/块内时处理。读 `wValueLength` 字节，取首个 `DWORD`：**低 16 位 = `languageId`，高 16 位 = `codePage`**；调 `PEResourceParserVersionLanguage.GetReadableTranslationInfo` 得到可读串写入 `AdditionalInfo.TranslationInfo`。

#### 5.6 语言 / 代码页映射（`Version.Language.cs`）

- 职责：把 `Translation` 中的语言 ID（如 `0x0409`）与代码页（如 `1200`）转成可读名称，且**随当前 UI 语言**（`GlobalState.CurrentLanguageType`，`LanguageType` 枚举）切换简体/繁体/英文三套名称数组。
- 数据结构：每种语言/代码页各有三套字符串数组（`englishLanguageNames`/`simplifiedChineseLanguageNames`/`traditionalChineseLanguageNames`，代码页同理），数组末元素是含 `{0:X4}`/`{0}` 占位的“未知”格式串。两张 `Dictionary<uint,int>`（`LanguageIdToIndex`、`CodePageToIndex`）**把 ID/代码页映射为数组下标**——这取代了原先含数百分支的大型 `switch`，把映射方法的圈复杂度降为 1。
- `GetReadableTranslationInfo(uint languageId, uint codePage)`：返回 `$"{语言名} (0x{languageId:X4}), {代码页名} ({codePage})"`。
- `GetLanguageName/GetCodePageName`：先 `SelectLanguageNames/SelectCodePageNames` 按 `CurrentLanguageType` 选数组，再查映射表取下标；未命中则用末元素格式串 `string.Format` 拼“未知语言/代码页 + 原值”。

### 6. 证书信息解析（`WIN_CERTIFICATE`）

文件：`PEAnalyzer/Resources/PEResourceParser.Certificate.cs`。结果写入 `peInfo.AdditionalInfo.IsSigned` 与 `CertificateInfo`。

#### 关键点

`ParseCertificateInfo(fs, reader, peInfo)`：

1. 取数据目录第 `PEConstants.DirectorySecurity`(=4) 项（`IMAGE_DIRECTORY_ENTRY_SECURITY`）。为空 → `IsSigned = false`，`CertificateInfo = "文件未签名"`。
2. **关键差异**：安全目录的 `VirtualAddress` **实际是文件偏移而非 RVA**（这是 PE 规范的特例），因此**不经 `RvaToOffset`**，直接当 `certificateOffset` 使用；`Size` 为整个证书区长度。
3. 边界校验 `certificateOffset/Size != 0 && certificateOffset < fs.Length && (long)certificateOffset + Size ≤ fs.Length`。
4. `ReadCertificates(...)` 解析证书数组；非空则 `IsSigned = true`，单证书直接展示其描述，多证书展示 `"N 个证书 -> ..."`（`string.Join("; ", ...)`）。

##### `ReadCertificates` —— `WIN_CERTIFICATE` 数组遍历（支持多重签名）

安全目录可含多个 8 字节对齐的 `WIN_CERTIFICATE`：

1. `certEnd = certificateOffset + certificateSize`，`pos = certificateOffset`。
2. `while (pos + 8 ≤ certEnd && pos + 8 ≤ fs.Length)`：定位 `pos` 读 8 字节头——`dwLength`(整条含头的长度)、`wRevision`、`wCertificateType`。
3. 校验 `dwLength ≥ 8 && pos + dwLength ≤ certEnd`，否则 `break`。
4. 类型映射 `wCertificateType`：`0x0001→X509`、`0x0002→PKCS#7`、`0x0003→PKCS#1`、其余 `未知`；记录 `$"类型: {certType}, 长度: {dwLength} 字节, 修订版: 0x{wRevision:X4}"`。
5. **8 字节对齐前进**：`pos += (dwLength + 7) & ~7`（`WIN_CERTIFICATE` 之间按 QuadWord 对齐）。
6. 末尾恢复 `originalPosition`，返回描述列表。

代码只读取证书头做描述（类型/长度/修订），不解析 PKCS#7 内部的签名者/证书链。

### 7. 表示层：`PEHeaderPresenter` 与 `PEHeaderDescriptions`

这两类把解析得到的 `PEInfo` 模型转换为 UI 友好的展示数据，是“**presenter**”模式的体现——UI 控件（`UserControls/PEAnalyzerControl.xaml.cs`）只消费它们的输出。

#### 7.1 `PEHeaderPresenter`

文件：`PEAnalyzer/PEHeaderPresenter.cs`。

##### 职责

把 `PEInfo` 转成 `List<(string Title, Dictionary<string, string> Info)>`——“分组标题 → 键值表”。把对 PE 头/可选头/CLR/节模型与 `PEParserUtils` 的引用从 UI 控件剥离，降低控件耦合。

##### 关键方法

- `BuildHeaderSections(PEInfo) → List<(string, Dictionary<string,string>)>`：构建头部分组——
  - **文件信息**：文件路径、文件类型（`PEHeaderDescriptions.GetDetailedFileType`）、架构（`GetArchitectureInfo`，.NET 时附 `CLRInfo.Architecture`）、位数（`GetBitInfo`）；若 `GetDriverType` 非空再加“驱动程序类型”。
  - **DOS 头**：`e_magic`（含 `'MZ'` 双字符）、`e_lfanew`。
  - **NT 头**：签名、机器类型（`GetMachineTypeDescription`）、节数量、时间戳（`UnixTimeStampToDateTime` 转本地时间）、可选头大小、特征标志。
  - **可选头**：魔数（`0x10b→PE32`/`0x20b→PE32+` 内联 `switch`）、链接器/编译器版本（`GetLinkerVersionDescription`/`GetCompilerVersionDescription`）、各类大小与基址（PE32+ 下“数据基址”显示 `N/A`，靠 `PEParserUtils.Is64Bit`）、OS/镜像/子系统版本、子系统、DLL 特征、栈/堆大小、数据目录数量等。
  - **.NET CLR 信息**（仅 `CLRInfo != null` 时）：运行时版本、架构、标志位、入口点、是否含元数据/资源/强名称签名。
  - **节信息**：`CreateSectionInfo` 逐节输出 `节 i (名)` → `RVA/大小/Raw大小/特征`，末加“总计 N 个节”。
- `BuildAdditionalSections(PEInfo)`：构建附加分组——**版本信息**(`FileVersion`/`ProductVersion`)、**公司和产品信息**、**版权和商标信息**、**证书信息**(`IsSigned`/`CertificateInfo`)，并在 `TranslationInfo` 非空时追加**翻译信息**分组。这些字段正是第 5、6 节解析所填。
- 私有辅助：`GetArchitectureInfo` / `GetBitInfo`（按 `CLRInfo.Architecture` 的 `x86/x64/ARM64/Any CPU` 映射位数）/ `CreateSectionInfo` / `UnixTimeStampToDateTime`（1970 UTC 基准 + 秒 → 本地时间）。

#### 7.2 `PEHeaderDescriptions`

文件：`PEAnalyzer/PEHeaderDescriptions.cs`。

##### 职责

提供 PE 头各字段的人类可读描述。把“展示用描述串”从 `PEParserUtils`（二进制读原语）分离，降低 `PEParserUtils` 类耦合。实现上**混用“字典查表”与“`switch` 表达式”**两种策略——枚举值密集且数量大的用字典，分段/区间判断的用 `switch`。

##### 关键方法

- `GetMachineTypeDescription(ushort machine)`：**字典 `s_machineTypes`** 查表（`0x014c→Intel 386 (x86)`、`0x8664→AMD64 (x64)`、`0xAA64→ARM64` 等数十项），未命中返回 `Unknown (0x….)`。机器码取值离散，字典最合适。
- `GetSubsystemDescription(ushort subsystem)`：`switch` 表达式映射（`2→Windows GUI`、`3→Windows CUI`、EFI 系列、`14→XBOX`、`16→Windows Boot Application` 等）。
- `GetLinkerVersionDescription(byte major, byte minor)` / `GetCompilerVersionDescription(byte major, byte minor, bool isNetAssembly=false)`：按主版本号 `switch` 映射到 VS/MSVC 版本；主版本 14 时再调私有 `MsvcVisualStudio(byte minor)` 细分（`10→2015`、`20→2017`、`26-29→2019`、`≥30→2022`），链接器与编译器**共用**这一细分函数。编译器描述在主版本未知且 `isNetAssembly` 时回退 `"Microsoft .NET Compiler"`。两者结尾都附 `[major.minor]` 原始版本。
- `GetOperatingSystemVersionDescription` / `GetSubsystemVersionDescription`：都委托私有 `DescribeWindowsVersion(major, minor, win10Text)`——按 `major` `switch`（3/4/5/6/10）再按 `minor` 细分到具体 Windows 版本（如 `6.1→Windows 7/Server 2008 R2`、`10.0→win10Text`）。两个公开方法**仅 Windows 10 主版本的文案不同**（OS 版用 `"Windows 10/11/Server 2016/2019/2022"`，子系统版用 `"Windows 10/11"`），故抽出共用核心。
- `GetImageVersionDescription(ushort, ushort)`：直接返回 `"Version major.minor"`。
- `GetDetailedFileType(ushort characteristics, ushort subsystem)`：先按 `Characteristics` 位判文件类别（`0x1000→Driver/System File`、`0x2000→Dynamic Link Library`、`0x0002→Executable`、否则 `Object File`），再按 `subsystem` `switch` 追加括号说明，结尾附 `Characteristics`/`Subsystem` 原值。
- `GetDriverType(ushort characteristics, ushort subsystem, ushort dllCharacteristics)`：`Characteristics & 0x1000 == 0`（非系统文件）返回空串；否则按 `subsystem` 给出驱动类型（`1→Windows NT Kernel Mode Driver` 等），并当 `dllCharacteristics & 0x0020`（`IMAGE_DLLCHARACTERISTICS_WDM_DRIVER`）置位时追加 `" (WDM)"`。

### 8. 端到端数据流

以图标为例（版本/证书类似，只是产物写到 `AdditionalInfo`）：

```
用户选文件
  → PEParser.ParsePEFile(path)
      → ParseHeaders（DOS/NT/可选头/节头 → PEInfo）
      → ParseContentDirectories（遍历 ContentParsers 表驱动数组）
          → PEResourceParserVersion.ParseVersionInfo  → 填 PEInfo.AdditionalInfo 版本字段
          → PEResourceParserCertificate.ParseCertificateInfo → 填 IsSigned/CertificateInfo
          → PEResourceParserIcon.ParseIconInfo
              → 资源RVA →RvaToOffset→ 资源段偏移
              → ResourceDirectoryReader.ScanTypeEntries/WalkEntries 递归目录树
              → GROUP_ICON→RT_ICON 关联，组装独立 .ico 字节 → PEInfo.Icons.Add(IconInfo)
  ← 返回 PEInfo
控件 PEAnalyzerControl：
  DisplayHeaderInfo → PEHeaderPresenter.BuildHeaderSections(PEInfo) → 分组键值表 → UI 面板
  DisplayAdditionalInfo → PEHeaderPresenter.BuildAdditionalSections(PEInfo) → UI 面板
  DisplayIcons → 对每个 IconInfo 调 IconImageFactory.TryCreate → IconViewModel → IconsDataGrid
```

总体来说，资源子系统以 `RvaToOffset` 把“内存视图（RVA）”映射回“磁盘视图（文件偏移）”，以 `ResourceDirectoryReader`/`VersionNodeReader` 两组共享原语统一处理“目录树”和“版本节点树”两种递归结构（含深度/环路/对齐/越界防护），各资源类型解析器只表达自身语义，最后由 presenter 与工厂把模型转成 UI 数据，做到了解析逻辑与展示逻辑彻底解耦。


## 4 ELF 解析核心


本节描述 PersonTools 中负责解析 ELF（Executable and Linkable Format）二进制文件的核心子系统。该子系统位于 `ELFAnalyzer/Core/` 与 `ELFAnalyzer/Models/`，采用「编排器 + 按职责拆分的解析类」的结构：单一编排器 `ELFParser` 持有所有共享状态与共享读取原语，各类静态解析器（`ELFHeaderInfo`、`ELFProgramHeaderInfo`、`ELFSectionHeaderReader`、`ELFSymbolTableReader`、`ELFDynamicReader`、`VersionSymbolParser`）只负责把对应区段从字节流解析进编排器的字段。解析结果再由 `ELFAnalyzer/UIHelper/` 下的辅助类转换为面向 UI 的视图模型。

---

### 1. 编排器 ELFParser（`ELFParser.cs`）

#### 1.1 职责

`ELFParser`（`internal sealed class`）是整个 ELF 解析的中枢，承担三类职责：

1. **持有解析全程的共享状态**：原始文件字节、位宽标志、各区段解析结果（头、程序头、节头、符号表、动态段、版本信息）。
2. **驱动解析步骤链**：在构造期一次性把整个文件解析完成（eager parsing）。
3. **提供共享读取原语**：节数据缓存读取（`GetSectionData`/`CopySectionData`），供后续符号名解析、重定位解析、版本信息解析复用，并集中做边界校验。

#### 1.2 关键字段与属性

```csharp
private bool _is64Bit;                                              // 32/64 位标志（由 EI_CLASS 决定）
private ELFHeader _header;                                          // ELF 头（值类型 struct）

public ELFHeader Header => _header;
public bool Is64Bit => _is64Bit;
public byte[] FileData { get; }                                    // 整个文件原始字节，常驻内存

public List<ELFProgramHeader>? ProgramHeaders { get; set; } = [];
public List<Models.ELFSectionHeader> SectionHeaders { get; set; } = [];
public Dictionary<SectionType, List<ELFSymbol>> Symbols { get; set; } = [];   // SHT_SYMTAB / SHT_DYNSYM → 符号列表
public List<ELFDynamic> DynamicEntries { get; set; } = [];

public Dictionary<SectionType, uint> LinkedStrTabIdx { get; } = [];          // 符号表 → 其 sh_link 指向的字符串表节索引
public ushort[] VersionSymbols { get; set; } = [];                            // .gnu.version 的逐符号版本索引数组
public Dictionary<ushort, string> VersionDefinitions { get; set; } = [];      // 版本索引 → 版本定义名（.gnu.version_d）
public Dictionary<ushort, string> VersionDependencies { get; set; } = [];     // 版本索引 → 版本依赖名（.gnu.version_r）

private readonly Dictionary<int, byte[]> _sectionDataCache = [];              // 节索引 → 已拷贝的节数据
```

构造函数提供两种入口，二者都在构造期立即调用 `ParseELFFile()`：

```csharp
public ELFParser(string filePath)   // 从磁盘 File.ReadAllBytes 读入全部字节
public ELFParser(byte[] fileData)   // 直接接收内存中的字节缓冲
```

#### 1.3 ParseELFFile 步骤链

`ParseELFFile()` 是整个解析的编排骨架。它在 `FileData` 之上建立一个 `MemoryStream`，再包一层 `BinaryReader`，然后**按固定顺序**依次调用各解析器，前一阶段的结果是后一阶段的前提：

```csharp
private void ParseELFFile()
{
    using MemoryStream ms = new(FileData);
    using BinaryReader reader = new(ms);
    bool isLittleEndian = true;

    _header = ELFHeaderInfo.ReadELFHeader(reader, ref _is64Bit, ref isLittleEndian);  // ① 头
    ELFProgramHeaderInfo.ReadProgramHeaders(this, reader, isLittleEndian);            // ② 程序头
    ELFSectionHeaderReader.ReadSectionHeaders(this, reader, isLittleEndian);          // ③ 节头
    ELFSymbolTableReader.ReadSymbolTables(this, reader, isLittleEndian);              // ④ 符号表
    ELFDynamicReader.ReadDynamicEntries(this, reader, isLittleEndian);                // ⑤ 动态段
    VersionSymbolParser.ReadVersionInformation(this);                                // ⑥ 版本信息
}
```

步骤链的**依赖关系**（解释为何顺序不可乱）：

- **① 头** 决定 `_is64Bit` 与 `isLittleEndian`（通过 `ref` 回写），这两个标志贯穿后续所有读取。`e_phoff`/`e_shoff`/`e_phnum`/`e_shnum`/`e_shstrndx` 也来自头，是后续定位的根据。
- **② 程序头**、**③ 节头** 各自 `Seek` 到 `e_phoff`/`e_shoff` 顺序读取。
- **④ 符号表** 遍历**已读取的节头**，只处理 `SHT_SYMTAB`/`SHT_DYNSYM` 类型节，因此必须在③之后。
- **⑤ 动态段** 在节头中查找 `SHT_DYNAMIC` 节，同样依赖③。
- **⑥ 版本信息** 依赖⑤（要从 `DynamicEntries` 找 `DT_VERSYM`/`DT_VERDEF`/`DT_VERNEED` 等标签的地址）以及③（按地址或类型定位 `.gnu.version*` 节）和④（版本符号与 `.dynsym` 一一对应）。

整个流程是**同步、一次性、全量**解析，没有惰性区段；唯一的惰性是节原始数据的按需拷贝与缓存（见 1.4）。

#### 1.4 节数据缓存与边界校验（GetSectionData / CopySectionData）

符号名解析、版本解析、重定位解析等会反复访问同一字符串表节，为避免反复整表拷贝，`ELFParser` 维护 `_sectionDataCache`（节索引 → 字节数组）。

```csharp
internal byte[] GetSectionData(int sectionIndex)               // 带缓存，按索引取节数据
internal byte[] CopySectionData(in Models.ELFSectionHeader section)  // 不缓存，按节结构取（只有节结构没有索引时）
```

- `GetSectionData` 先查缓存命中即返回；**索引越界**（如 `e_shstrndx` 超出节数）返回空数组 `[]` 而非抛 `IndexOutOfRange`；命中则调用 `CopySectionData` 拷贝并写回缓存。
- `CopySectionData` 是真正的边界校验点：当 `sh_size == 0`、`sh_size > int.MaxValue`、或 `sh_offset + sh_size > FileData.Length` 时，判定为畸形节，返回空数组，避免越界拷贝与 `int` 溢出；否则 `Array.Copy` 出该节字节。
- 注释明确约定：**调用方不得就地修改返回的数组**（缓存共享，修改会污染其他读取者）。重定位解析有意使用不缓存的 `CopySectionData`（拿到的是节结构而非索引）。

> 设计决策：把「越界/畸形返回空」收敛到这两个原语里，使上层解析代码无需到处写防御分支即可对恶意/损坏文件健壮。

### 2. 字节读取原语与端序处理（`ELFParserUtils.cs`）

#### 2.1 职责

`ELFParserUtils`（静态工具类）集中提供：32/64 位与大小端无关的整型读取、字符串提取、按地址/链接查找节、枚举名解析。它是「共享读取原语」的物理载体，被几乎所有解析器引用。

#### 2.2 两套 Read* 原语：流式 vs 缓冲

读取被刻意拆成两组，应对两种解析模式：

**流式（sequential parse）** —— 接收 `BinaryReader`，从 `BaseStream` 顺序读取，用于头/程序头/节头/符号表/动态段这类「按文件偏移 Seek 后顺序推进」的解析：

```csharp
long  ReadInt64 (BinaryReader reader, bool isLittleEndian)
int   ReadInt32 (BinaryReader reader, bool isLittleEndian)
ushort ReadUInt16(BinaryReader reader, bool isLittleEndian)
uint  ReadUInt32(BinaryReader reader, bool isLittleEndian)
ulong ReadUInt64(BinaryReader reader, bool isLittleEndian)
```

实现统一为：`stackalloc` 出对应宽度的 `Span<byte>` → `reader.BaseStream.ReadExactly(buffer)` → 按端序用 `BinaryPrimitives.Read*LittleEndian/BigEndian` 解码。`ReadExactly` 保证读满，端序由 `isLittleEndian` 选择两条分支之一。

**缓冲（already-loaded data）** —— 接收 `byte[] data, int offset`，从已加载缓冲随机位置读取，用于版本信息/重定位这类「先把节拷出来再按结构内偏移跳读」的解析：

```csharp
ushort ReadUInt16(byte[] data, int offset, bool isLittleEndian)
uint   ReadUInt32(byte[] data, int offset, bool isLittleEndian)
ulong  ReadUInt64(byte[] data, int offset, bool isLittleEndian)
int    ReadInt32 (byte[] data, int offset, bool isLittleEndian)
long   ReadInt64 (byte[] data, int offset, bool isLittleEndian)
```

实现为 `data.AsSpan(offset, N)` → 同样的 `BinaryPrimitives` 端序解码，不修改源缓冲（只读 span）。

> 设计决策：32/64 位差异不体现在这些原语里，而是**由调用方根据 `Is64Bit` 选择读 32 还是 64 位字段**（如 `is64Bit ? ReadUInt64(...) : ReadUInt32(...)`）；端序差异则**完全封装在原语内部**。这样位宽决策留在「懂结构布局」的解析器中，端序处理则一次写对、处处复用。

#### 2.3 字符串提取

```csharp
string ExtractStringFromBytes(byte[] data, int startOffset)              // 读到 NUL 结束
string ExtractStringFromBytes(byte[] data, int startOffset, int maxLength) // 限长读取
```

- 单参版本：从 `startOffset` 起扫描到 `\0` 或缓冲末尾，按 UTF-8 解码 `[startOffset, endOffset)`。用于字符串表（`.strtab`/`.dynstr`/`.shstrtab`）取名，offset 即 `sh_name`/`st_name`/`vn_file`/`vda_name` 等。
- 限长版本：把 `maxLength` 夹紧到剩余可读字节（防止 `namesz`/`descsz` 越界），再回退去掉尾部 NUL。

#### 2.4 共享查找原语

两个供「版本/动态解析」复用的查找函数：

```csharp
Models.ELFSectionHeader? FindSectionByAddress(ELFParser parser, ulong address)
```
按虚拟地址 `sh_addr` 线性匹配节。版本符号表/verdef/verneed 在动态段中以**虚拟地址**形式给出（`DT_VERSYM` 等），需用本函数把地址映射回节头。

```csharp
bool TryGetLinkedStringTable(ELFParser parser, Models.ELFSectionHeader section,
                             out byte[] strTabData, out bool isLittleEndian)
```
解析某节通过 `sh_link` 关联的字符串表。`verdef`/`verneed` 的解析与格式化两路都要拿到对应字符串表（`.dynstr`），本函数集中处理「`sh_link` 越界返回 false 且 `strTabData` 置空，但 `isLittleEndian` 始终给出」的语义。它内部调用 `GetSectionData`，因此也享受缓存。

#### 2.5 枚举名解析

```csharp
string GetTypeName(Type enumType, object type, string prefix)
```
把原始数值映射为枚举名，用于几乎所有「码 → 文本」转换（程序头类型、节类型、符号类型/绑定/可见性、动态标签、重定位类型……）。关键实现细节：先用 `Convert.ChangeType` 把数值转为枚举的**底层类型**，再 `Enum.IsDefined` 判定，避免类型不匹配导致 `Enum.IsDefined` 抛异常，也省去逐次异常开销；未命中或转换失败（捕获 `ArgumentException`/`InvalidCastException`/`OverflowException`）则返回 `"{prefix}UNKNOWN({type})"`。

> 设计决策：把「未知值不崩溃、统一回退格式」这一策略集中到一个函数，使所有码表转换行为一致。

### 3. ELF 头解析（`ELFHeaderInfo.cs` + `Models/ELFHeader.cs`）

#### 3.1 数据模型

`ELFHeader`（`[StructLayout(Sequential, Pack=1)]` 的 `struct`）以 64 位字段宽度（`ulong e_entry/e_phoff/e_shoff`）统一表示 32/64 位头，`e_ident` 拆成 `EI_MAG0..EI_PAD` 独立字段。

#### 3.2 ReadELFHeader 步骤

```csharp
static ELFHeader ReadELFHeader(BinaryReader reader, ref bool is64Bit, ref bool isLittleEndian)
```

1. 顺序读 16 字节 `e_ident`：4 字节魔数 + `EI_CLASS`/`EI_DATA`/`EI_VERSION`/`EI_OSABI`/`EI_ABIVERSION` + 7 字节 `EI_PAD`。
2. **魔数校验**：`0x7F 'E' 'L' 'F'`，不符抛 `InvalidDataException("File is not a valid ELF file")`。这是整条解析链唯一的硬失败点。
3. **从 `EI_DATA` 判定端序**：`isLittleEndian = EI_DATA == ELFData.LSB`（`ref` 回写到编排器）。
4. **从 `EI_CLASS` 判定位宽**：`is64Bit = EI_CLASS == ELFClass.BIT64`（`ref` 回写）。
5. 用 `ELFParserUtils.Read*` 按端序读其余字段；其中 `e_entry`/`e_phoff`/`e_shoff` 按 `is64Bit` 选择读 `ReadUInt64` 或 `ReadUInt32`，其余为定宽 16/32 位。

扩展方法 `IsLittleEndian(this ELFHeader)` 供后续阶段（不再持有 `ref isLittleEndian`）从头重新判定端序，多处使用。

#### 3.3 头信息派生（`ELFHeaderDescriptions.cs`）

`ELFHeaderDescriptions`（静态）把头字段转成人类可读文本，是「表驱动派发」的典型：

- **`e_machine`**：`GetMachineDescription` 用 `s_machineDescriptions`（`Dictionary<ushort,string>`，约 230 项，覆盖 EM_NONE..EM_CISCO_IOS64）取代原超长 switch；未命中回退 `GetArchitectureName`（再回退到 `GetTypeName(typeof(EMachine), ...)`）。
- **`EI_OSABI`**：`GetOSABIName` 用 `s_osabiNames`（`Dictionary<byte,string>`）映射，未命中 `"OS/ABI Unknown"`。
- **`e_type`**：`GetFileTypeDescription` 用 `switch` 给中文描述（可重定位/可执行/共享对象/核心转储），其余回退 `GetELFTypeName`。
- **`e_flags`（按架构解释）**：`GetFormattedELFFlags` 按 `e_machine` 分派到各架构专用格式化器，结果用 `ConvertUtils.EnumerableToString(", ", …)` 拼接：
  - **MIPS**：`s_mipsFlagBits` 平铺位标志（noreorder/pic/cpic/o32/o64/n32/nan2008/nan2001）经 `FormatFlagBits` 收集；再单独把高 4 位（`flags & 0xF0000000`）经 `GetMIPSArchitectureName` 映射出架构版本（mips1..mips64r2）追加到末尾。
  - **ARM**：`GetARMSFormattedELFFlags` 中 EABI 版本取自高 8 位（`EF_ARM_EABIMASK=0xFF000000`）的**精确值**（不能按位与），再按位测试浮点 ABI（hard/soft-float）、has entry point、BE8、interworking。
  - **PPC/PPC64**：`s_ppcFlagBits`（elfv1/elfv2 ABI）；**SPARC 系**：`s_sparcFlagBits`（sparc_ext/32bit/64bit）。
  - 其他架构：直接输出 `0x%08X`。
  - `FormatFlagBits(uint flags, (uint Mask,string Label)[] table)` 是纯位标志架构（PPC/SPARC/MIPS 平铺部分）共用的收集器：按掩码顺序把命中的标签收进列表。

> 设计决策：`e_machine` 这类巨型映射用静态 `Dictionary` 表驱动，既显著缩短代码、提升可维护性，又让查找 O(1)。架构相关的 `e_flags` 解释拆成各自函数，因为不同架构的位语义（位标志 vs 高位精确值）本质不同，无法统一成一张表。

### 4. 程序头解析（`ELFProgramHeaderInfo.cs` + `Models/ELFProgramHeader.cs`）

#### 4.1 数据模型

`ELFProgramHeader`（`struct`，属性形式）统一以 64 位宽度存 `p_offset/p_vaddr/p_paddr/p_filesz/p_memsz/p_align`。

#### 4.2 ReadProgramHeaders 步骤

```csharp
static void ReadProgramHeaders(ELFParser parser, BinaryReader reader, bool isLittleEndian)
```

1. `e_phnum == 0` 直接返回（无程序头）。
2. `Seek` 到 `e_phoff`。
3. 循环 `e_phnum` 次，每次读一个程序头。**注意 32/64 位字段顺序不同**：
   - 64 位顺序：`p_type, p_flags, p_offset, p_vaddr, p_paddr, p_filesz, p_memsz, p_align`（`p_flags` 紧跟 `p_type`）。
   - 32 位顺序：`p_type, p_offset, p_vaddr, p_paddr, p_filesz, p_flags, p_align`（`p_flags` 在 `p_filesz` 之后）。
   
   这一差异由 `parser.Is64Bit` 的 if/else 分支显式处理——正是 2.2 节所说「位宽决策留在解析器」的体现。

#### 4.3 显示派生

- `GetProgramHeaderType(uint)` → `GetTypeName(typeof(ProgramHeaderType), …)`。
- `GetProgramHeaderFlags(uint)` → 按 `PF_R/PF_W/PF_X` 位拼出 `"R"/"W"/"E"`（注意 X 显示为 `E`，与 readelf 习惯一致），`ConvertUtils.EnumerableToString("", …)` 无分隔拼接。
- 视图模型 `Models/ProgramHeaderInfo.cs`（`ProgramHeaderInfo`）：全字符串字段（Type/Offset/VirtAddr/PhysAddr/FileSize/MemSize/Flags/Align），供 UI 网格直接绑定。

### 5. 节头解析（`ELFSectionHeaderReader.cs` + `Models/ELFSectionHeader.cs`）

#### 5.1 数据模型

`ELFSectionHeader`（`struct`，字段形式）统一以 64 位宽度存 `sh_flags/sh_addr/sh_offset/sh_size/sh_addralign/sh_entsize`。

#### 5.2 ReadSectionHeaders 步骤

```csharp
static void ReadSectionHeaders(ELFParser parser, BinaryReader reader, bool isLittleEndian)
```

1. `e_shnum == 0` 直接返回。
2. `Seek` 到 `e_shoff`。
3. 循环 `e_shnum` 次读节头。节头字段**顺序在 32/64 位下相同**，只是若干字段宽度不同：`sh_flags/sh_addr/sh_offset/sh_size/sh_addralign/sh_entsize` 在 64 位读 `ReadUInt64`、32 位读 `ReadUInt32`；`sh_name/sh_type/sh_link/sh_info` 始终 32 位。

#### 5.3 显示派生

- `GetSectionType(uint)` → `GetTypeName(typeof(SectionType), …)`。
- `GetSectionFlags(ulong)`：用 `s_sectionFlagChars`（`(SectionAttributes, char)[]`，顺序与 readelf 一致：W A X M S I L O G T C）逐位测试拼成标志串。
- 节名解析见 6.3 的 `GetSectionName`。
- 视图模型 `Models/ELFSectionHeaderInfo.cs`（`ELFSectionHeaderInfo`）：`Index` + 全字符串字段。

### 6. 符号表解析（`ELFSymbolTableReader.cs`、`ELFSymbolNameResolver.cs`、`ELFSymbolInfo.cs`）

#### 6.1 ReadSymbolTables 步骤（`ELFSymbolTableReader`）

```csharp
static void ReadSymbolTables(ELFParser parser, BinaryReader reader, bool isLittleEndian)
```

遍历所有节头，对每个 `SHT_SYMTAB` 或 `SHT_DYNSYM` 节：

1. `sh_entsize == 0` 跳过（畸形文件，避免除零）。
2. `Seek` 到 `sh_offset`。
3. `symbolCount = sh_size / sh_entsize`，循环用 `ReadSymbol` 读每个符号。
4. 结果写入 `parser.Symbols[(SectionType)sh_type]`；同时把该符号表的 `sh_link`（指向其字符串表节的索引）记入 `parser.LinkedStrTabIdx[(SectionType)sh_type]`，供后续取名。

`ReadSymbol` 处理 **`Elf32_Sym` 与 `Elf64_Sym` 字段顺序不同**：

- 二者都先读 `StName`（4 字节）。
- 64 位顺序：`StInfo(1) StOther(1) StShndx(2) StValue(8) StSize(8)`。
- 32 位顺序：`StValue(4) StSize(4) StInfo(1) StOther(1) StShndx(2)`。

数据模型 `Models/ELFSymbol.cs`（`struct ELFSymbol`）字段：`StName/StValue/StSize/StInfo/StOther/StShndx`。

#### 6.2 符号字段拆分（`ELFSymbolInfo`）

`StInfo` 复用一字节存类型与绑定，`StOther` 低 2 位存可见性：

```csharp
GetSymbolType(byte stInfo)       => GetTypeName(typeof(SymbolType),       stInfo & 0x0F, "")  // 低 4 位
GetSymbolBinding(byte stInfo)    => GetTypeName(typeof(SymbolBinding),    stInfo >> 4,   "")  // 高 4 位
GetSymbolVisibility(byte stOther)=> GetTypeName(typeof(SymbolVisibility), stOther & 0x03,"")  // 低 2 位
```

#### 6.3 符号取名 + 版本后缀（`ELFSymbolNameResolver`）

```csharp
static string GetSymbolName(ELFParser parser, ELFSymbol symbol, SectionType sectionType, int symbolIndex)
```

取名步骤：

1. 从 `LinkedStrTabIdx[sectionType]` 取关联字符串表节索引，越界返回空串。
2. 校验该节确为 `SHT_STRTAB`，否则返回空串。
3. `GetSectionData` 取字符串表字节；以 `symbol.StName` 为偏移（越界返回空串）。
4. `ExtractStringFromBytes` 取基础名 `baseName`；为空则返回空串。
5. 调 `AppendVersionSuffix` 给动态符号追加版本后缀。

**版本后缀 `@@` / `@` 规则**（`AppendVersionSuffix`）：

- 取 `parser.VersionSymbols[symbolIndex] & 0x7fff`（屏蔽最高位「隐藏」标志）得 `versionIndex`。
- `versionIndex < 2`（0=本地、1=全局/基础）视为无外部版本，原样返回 `baseName`。
- 经 `GetVersionNameByVersionIndex` 取版本名：**优先 `VersionDefinitions`（verdef）→ 再 `VersionDependencies`（verneed）→ 回退 `VER_{index}`**。
- 经 `IsDefaultVersionSymbol` 判定是否默认版本：默认用 `@@`，否则用 `@`，拼成 `baseName + "@@"/"@" + versionName`。`IsDefaultVersionSymbol` 是简化实现——仅当符号绑定为 `STB_GLOBAL`/`STB_WEAK` 且 `versionIndex == 1` 时判为默认。

```csharp
static string GetSectionName(ELFParser parser, int index)
```
节名解析：从 `e_shstrndx` 指向的节字符串表（`GetSectionData(e_shstrndx)`）按 `section.sh_name` 偏移 `ExtractStringFromBytes`。

#### 6.4 符号表视图（`UIHelper/SymbolTableHelper`）

`GetSymbolTableInfoList(parser, sectionType)` 把 `parser.Symbols[sectionType]` 逐条转 `ELFSymbolTableInfo`：调 `GetSymbolName` 取名；若名为空且为 `STT_SECTION` 符号（`StShndx` 在 `(0, 0xFF00)` 内）则回退显示其所属节名；`Type/Bind/Vis` 经 `StripSymbolPrefix` 去掉 `STT_`/`STB_`/`STV_` 前缀（与 readelf 一致）；`Ndx` 把 `0→UND`、`0xFFF1→ABS`、`0xFFF2→COM`。

### 7. 符号版本信息（`VersionSymbolParser.cs`、`VersionSymbolParser.ParseSymbolTable.cs`、`VersionSymbolParser.ParseDependencies.cs`、`VersionSymbolParser.ParseDefinitions.cs`、`VersionSymbolFormatter.cs`）

符号版本由 `VersionSymbolParser`（`internal static partial class`，跨多个文件）填充编排器的三个字典/一个数组；`VersionSymbolFormatter` 负责渲染。

#### 7.1 编排入口（`VersionSymbolParser.cs`）

```csharp
static void ReadVersionInformation(ELFParser parser)
{
    parser.VersionDefinitions  = [];
    parser.VersionDependencies = [];
    ParseVersionSymbolTable(parser);   // .gnu.version  → VersionSymbols[]
    ParseVersionDefinitions(parser);   // .gnu.version_d → VersionDefinitions
    ParseVersionDependencies(parser);  // .gnu.version_r → VersionDependencies
}
```

#### 7.2 .gnu.version → VersionSymbols（`VersionSymbolParser.ParseSymbolTable.cs`）

```csharp
static void ParseVersionSymbolTable(ELFParser parser)
```

1. `FindDynamicValue(parser, DT_VERSYM)` 取版本符号表**虚拟地址**。
2. `FindSectionByAddress` 把地址映射到节，`CopySectionData` 拷出。
3. `count = sh_size / 2`（每项 2 字节），逐项 `ReadUInt16` 填 `parser.VersionSymbols[i]`。该数组与 `.dynsym` 索引一一对应，即 6.3 中以 `symbolIndex` 索引的来源。

```csharp
static long FindDynamicValue(ELFParser parser, DynamicTag tag, long defaultValue = 0)
```
**共享原语**：线性扫 `DynamicEntries` 取指定标签的 `d_val`；**同标签多次出现取最后一次**（逐项覆盖）；未找到返回默认值。verdef/verneed/versym 三处都用它定位。

#### 7.3 .gnu.version_d → VersionDefinitions（`VersionSymbolParser.ParseDefinitions.cs`）

```csharp
static void ParseVersionDefinitions(ELFParser parser)
```

1. 取 `DT_VERDEF`（地址）与 `DT_VERDEFNUM`（条数）。
2. 二者均 >0 时按地址定位节、`ParseVerDefEntries`；否则回退 `FindAndParseVersionDefinitionSection`（直接扫 `SHT_GNU_verdef` 类型节，条数用 `CalculateVerDefEntryCount = sh_size / sh_entsize`）。

`ParseVerDefEntries` 遍历 **Verdef 链表**（每项 ≥20 字节，循环条件 `offset + 20 <= FileData.Length`）：

- 读 `vd_ndx`(+4)、`vd_aux`(+12)、`vd_next`(+16)。**注意**：注释明确 Verdef 结构与位宽无关，`vd_aux`/`vd_next` 均为 4 字节。
- 紧随其后的 Verdaux 在 `offset + vd_aux` 处，其 `vda_name`(+0) 是版本名在字符串表（经 `TryGetLinkedStringTable` 取得）中的偏移，`ExtractStringFromBytes` 取名。
- 以 `vd_ndx & 0x7fff`（去隐藏标志）为键 `TryAdd` 进 `VersionDefinitions`。
- `vd_next == 0` 或 `offset + vd_next` 越界则停；否则 `offset += vd_next` 前进。

#### 7.4 .gnu.version_r → VersionDependencies（`VersionSymbolParser.ParseDependencies.cs`）

```csharp
static void ParseVersionDependencies(ELFParser parser)
```

1. 取 `DT_VERNEED`（地址）与 `DT_VERNEEDNUM`（条数），均 >0 时按地址定位节、`ParseVerNeedEntries`。
2. `ParseVerNeedEntries` 用 `TryGetLinkedStringTable` 取 `.dynstr`，再调**共享遍历 `WalkVerneed`**：`onVerneed` 空操作，`onVernaux` 读每个 vernaux 的 `flags`(+6)、`nameOffset`(+8) 取版本名，以 `flags & 0x7fff` 为版本索引键 `TryAdd`。

##### WalkVerneed —— verneed 链表遍历的单点实现

```csharp
internal static int WalkVerneed(ELFParser parser, long sectionStart, int maxCount, bool isLittleEndian,
                                Action<long,ushort> onVerneed, Action<long> onVernaux)
```

这是「verneed 解析填表」与「verneed 格式化输出」两路**共享**的链表遍历器，把链表步进与边界校验收敛到一处：

- 外层遍历 **verneed 项**（`Elf*_Verneed`），从 `sectionStart` 起，循环条件 `(maxCount<=0 || processed<maxCount) && offset+16<=FileData.Length`（`maxCount<=0` 表示不限项数，仅靠 `vn_next==0` 或越界停止）。
  - 读 `vn_cnt`(+2)、`vn_aux`(+8)、`vn_next`(+12)，回调 `onVerneed(offset, vn_cnt)`。
- 内层遍历该 verneed 下的 **vernaux 辅助项**（`Elf*_Vernaux`），从 `offset + vn_aux` 起，循环 `vn_cnt` 次且 `auxOffset+16<=FileData.Length`：
  - 读 `vna_next`(+12)，回调 `onVernaux(auxOffset)`，`vna_next==0` 则停，否则 `auxOffset += vna_next`。
- 外层 `vn_next==0` 停，否则 `offset += vn_next`。返回已处理 verneed 项数。

> 设计决策：verneed 是二层链表，步进偏移（+2/+8/+12 与辅助项 +12）和越界判断容易写错；用回调式 `WalkVerneed` 把「怎么走」固定下来，让「走到每一项做什么」由两个调用方各自决定（填字典 vs 拼输出），避免逻辑重复与不一致。

#### 7.5 版本信息渲染（`VersionSymbolFormatter.cs` → `VersionSymbolFormatter`）

把已填好的字典/数组渲染成类 readelf 文本：

- `GetFormattedVersionSymbolInfo`：定位 `.gnu.version` 节，逐项打印 `versionIndex` 与名称（`GetVersionInfoByIndex`：`0→*本地*`、`1→*全局*`、其余先 verdef 后 verneed 回退 `VER_n`），每 4 项换行（`(i & 0x3)==0x3`）。
- `GetFormattedVersionDependencyInfo`：定位 `.gnu.version_r`，调 `ParseAndAppendVersionNeeds`——内部以 `maxCount:-1` 调 `WalkVerneed`：`onVerneed` 打印 `版本/文件(库名)/计数`，`onVernaux` 打印 `名称/标志/版本`（`vna_other & 0x7fff` 为版本索引，`GetVerneedFlags` 解析 `VER_FLG_BASE=0x1`/`VER_FLG_WEAK=0x2`）。节内偏移按 `FormatVersionOffset`（0→`000000`，否则 `0x%04x`）格式化，与 readelf 对齐。
- `GetFormattedVersionDefinitionInfo`：定位 `SHT_GNU_verdef` 节，按 `VersionDefinitions` 升序打印每个版本定义（`Index==1` 标 `BASE`），并在条数超出 `CalculateVerDefEntryCount` 时追加 "Version definition past end of section"（仿 readelf）。

### 8. 动态段与重定位

#### 8.1 动态段读取（`ELFDynamicReader.cs` + `Models/ELFDynamic.cs`）

```csharp
static void ReadDynamicEntries(ELFParser parser, BinaryReader reader, bool isLittleEndian)
```

1. 线性查找首个 `SHT_DYNAMIC` 类型节。
2. `sh_entsize == 0` 返回（避免除零）。
3. `Seek` 到 `sh_offset`，`entryCount = sh_size / sh_entsize`。
4. 每项读 `d_tag`（64 位 `ReadInt64`/32 位 `ReadInt32`）与 `d_val`（64 位 `ReadUInt64`/32 位 `ReadUInt32`），存入 `parser.DynamicEntries`。

数据模型 `ELFDynamic`（`struct`）：`long d_tag` + `ulong d_val`。这正是 7.2 节 `FindDynamicValue` 扫描的数据源。

显示派生（`ELFDynamicInfo`）：`GetDynamicTagDescription(ulong)` → `GetTypeName(typeof(DynamicTag), …)`；`GetDynamicFlagDescription(uint)` 按 `DF_ORIGIN/DF_SYMBOLIC/DF_TEXTREL/DF_BIND_NOW/DF_STATIC_TLS` 位拼接（用于 `DT_FLAGS`）。视图模型 `Models/ELFDynamicSectionInfo.cs`：`Tag/Type/Value` 三字符串字段。

#### 8.2 重定位类型命名（`ELFRelocation.cs`）

`ELFRelocation.GetRelocationTypeName(uint type, ushort machine)` 是**按机器类型的表驱动派发**：用 `switch (machine)` 选择对应架构的重定位枚举与前缀，再交给 `GetTypeName`。覆盖约 25 个架构枚举，例如：

| machine | 枚举类型 | 前缀 |
|---|---|---|
| `EM_X86_64` | `X86_64RelocationType` | `R_X86_64_` |
| `EM_386` | `I386RelocationType` | `R_386_` |
| `EM_ARM` | `ARMRelocationType` | `R_ARM_` |
| `EM_AARCH64` | `AArch64RelocationType` | `R_AARCH64_` |
| `EM_MIPS`/`EM_MIPS_RS3_LE` | `MipsRelocationType` | `R_MIPS_` |
| `EM_LOONGARCH` | `LoongArchRelocationType` | `R_LARCH_` |
| `EM_PPC`/`EM_PPC64` | `PowerPCRelocationType`/`PowerPC64RelocationType` | `R_PPC_`/`R_PPC64_` |
| `EM_SPARC`/`EM_SPARCV9` | `SPARCRelocationType` | `R_SPARC_` |

（另含 68K/PARISC/ALPHA/IA64/S390/CRIS/MN10300/M32R/MicroBlaze/Nios2/TILEPro/TILEGx 等。）未支持的机器回退 `R_UNKNOWN({type})`。

> 设计决策：重定位类型号的语义**强依赖目标架构**（同一数字在不同架构含义完全不同），因此必须先按 `machine` 分派到对应枚举，再统一走 `GetTypeName`。本文件只做「类型号→名称」，条目的实际读取/拆分在 UIHelper 层（见 8.3）。

#### 8.3 重定位条目解析（`UIHelper/RelocationHelper`）

实际把重定位节解析为 `ELFRelocationInfo`（`Models/ELFRelocationInfo.cs`：Offset/Info/Type/SymbolValue/Symbol/Addend/SectionName 全字符串）的逻辑在此：

```csharp
static List<ELFRelocationInfo> GetRelocationInfoForSpecificSection(ELFParser parser, string sectionName)
```

1. `FindRelocationSection` 按名查到 `SHT_RELA`/`SHT_REL` 节。
2. `entryCount = sh_size / sh_entsize`，`CopySectionData` 拷出节数据；从 `sh_link` 取关联符号表，`ReadRelocationSymbols` 按 64 位 24 字节 / 32 位 16 字节解析符号。
3. 按节名是否含 `"rela"` 决定 `isRela`，逐条 `ReadRelocationEntry`：
   - **64 位**：项长 RELA=24 / REL=16；`offset`(+0,8B)、`info`(+8,8B)、`addend`(+16,8B，仅 RELA)。
   - **32 位**：项长 RELA=12 / REL=8；`offset`(+0,4B)、`info`(+4,4B)、`addend`(+8,4B，仅 RELA)。
4. `SplitRelocInfo` 拆 `r_info`：**64 位 `sym = info>>32, type = info&0xffffffff`；32 位 `sym = info>>8, type = info&0xff`**。
5. `ResolveRelocSymbol` 按 `sym` 取符号名/值（越界回退零值）；`type` 经 `ELFRelocation.GetRelocationTypeName(type, e_machine)` 取名。SECTION 类符号名为空时回退其所属节名。

### 9. 数据流总览（input → parse → display）

```
文件路径/字节
   │  new ELFParser(...)            （构造期 eager 解析）
   ▼
ParseELFFile()  ──► ELFHeaderInfo.ReadELFHeader   ──► _header, _is64Bit, isLittleEndian
   │                ELFProgramHeaderInfo           ──► ProgramHeaders
   │                ELFSectionHeaderReader         ──► SectionHeaders
   │                ELFSymbolTableReader.ReadSymbolTables    ──► Symbols + LinkedStrTabIdx
   │                ELFDynamicReader.ReadDynamicEntries      ──► DynamicEntries
   │                VersionSymbolParser             ──► VersionSymbols / VersionDefinitions / VersionDependencies
   ▼
ELFParser（持有全部状态 + GetSectionData/CopySectionData 缓存原语）
   ▼
UIHelper（SymbolTableHelper / RelocationHelper / …）+ ELFHeaderDescriptions/ELFSectionHeaderReader/ELFProgramHeaderInfo 等显示派生
   │  按需 GetSectionData / GetSymbolName / GetRelocationTypeName / Get*Description
   ▼
视图模型（ELFSymbolTableInfo / ELFRelocationInfo / ELFSectionHeaderInfo / ProgramHeaderInfo / ELFDynamicSectionInfo / ELFGotInfo）
   ▼
WPF 网格/文本绑定显示
```

- **input**：磁盘文件或内存字节 → `FileData`。
- **parse**：构造期一次性解析六大区段进编排器字段；其中节原始数据按需经缓存原语拷出。
- **display**：UIHelper 层惰性地从编排器读字段、调共享原语解析名称/数据、转成全字符串视图模型，交 WPF 绑定。

### 10. 关键设计决策小结

1. **编排器 + 按职责拆分的解析类**：`ELFParser` 只编排与持状态，每个区段一个静态解析器（`ELFHeaderInfo`/`ELFProgramHeaderInfo`/`ELFSectionHeaderReader`/`ELFSymbolTableReader`/`ELFDynamicReader`/`VersionSymbolParser`），通过 `partial class` 把版本解析进一步按子职责（versym/verdef/verneed）拆到独立文件。职责单一、可独立演进。
2. **共享读取原语集中化**：端序无关读取、字符串提取、按地址/链接查找节、枚举名解析全在 `ELFParserUtils`；位宽差异留给解析器、端序差异封装进原语。`FindDynamicValue`/`FindSectionByAddress`/`TryGetLinkedStringTable`/`WalkVerneed` 是被多路复用的高阶原语，把易错的链表步进与边界判断收敛到单点。
3. **节数据缓存 + 统一边界校验**：`GetSectionData`/`CopySectionData` 既避免重复拷贝，又把「越界/畸形返回空、不抛异常」的健壮性策略集中实现，使上层免于到处防御。
4. **表驱动派发**：`e_machine` 描述、`EI_OSABI`、节标志字符、各架构 `e_flags`、重定位类型按架构分派——用静态 `Dictionary`/数组/`switch` 取代巨型 switch，提升可读性、可维护性与查找效率。
5. **解析与渲染分离**：Core 只产出结构化状态，渲染/格式化（含类 readelf 文本对齐、`@@`/`@` 后缀、`VER_n` 回退）放在 `VersionSymbolFormatter` 与 UIHelper 层，保证 Core 纯净、可测试。


## 5 ELF 高级解析（Notes / ARM 属性 / Exidx）


本节描述 PersonTools 中 ELF 分析器的三块「高级」解析能力——`SHT_NOTE` 备注、ARM/GNU/RISC-V 构建属性（build attributes）、ARM EHABI 异常展开索引（`.ARM.exidx` / `.ARM.extab`），以及把这些解析结果送到 WPF 界面的 UIHelper 薄封装层与 `ELFAnalyzer` 门面。

涉及源文件：

- 解析核心：`ELFAnalyzer/Core/ELFNoteInfo.cs`、`ELFAttributeInfo.cs`、`ELFExidxInfo.cs`
- 共享原语：`ELFAnalyzer/Core/ELFParser.cs`（`ELFParser`、`CopySectionData`/`GetSectionData`、节数据缓存）、`ELFParserUtils.cs`（`ELFParserUtils` 读取原语与字符串抽取）
- UI 薄封装：`ELFAnalyzer/UIHelper/ExidxInfoHelper.cs`、`AttributesHelper.cs`
- 门面：`ELFAnalyzer/ELFAnalyzer.cs`
- 宿主控件（数据流终点）：`UserControls/ELFAnalyzerControl.xaml.cs`

---

### 1. 总体架构与设计决策

#### 1.1 编排器 + 按职责拆分的解析类

`ELFParser`（`ELFParser.cs`）是唯一的编排器：构造时 `ParseELFFile()` 依次读取 ELF 头、程序头、节头、符号表、动态段、版本信息，并把结果暴露为公有属性（`Header`、`SectionHeaders`、`Symbols`、`DynamicEntries`、`FileData`、`Is64Bit` 等）。它本身**不**承担 Note/属性/Exidx 这类「高级」解析；这三块各自被拆到独立的静态类中：

- `ELFNoteInfo`（`ELFNoteInfo.cs`）
- `ELFAttributeInfo`（`ELFAttributeInfo.cs`）
- `ELFExidxInfo`（`ELFExidxInfo.cs`）

每个类对外只暴露一个入口方法，签名统一为「接收 `ELFParser`、返回格式化字符串」：

```csharp
internal static string ELFNoteInfo.GetFormattedNotesInfo(ELFParser parser)
public  static string ELFAttributeInfo.GetFormattedAttributeInfo(ELFParser parser)
internal static string ELFExidxInfo.GetFormattedExidxInfo(ELFParser parser)
```

这样设计的原因：这三块解析互相独立、各自体量大、且都是「遍历相关类型的节 → 解析 → 拼成 readelf 风格文本」的同构流程。把它们从编排器剥离，使 `ELFParser` 保持瘦身（只负责把文件结构读进内存），而每个高级解析器可以独立演进、独立按需调用（只有打开对应面板时才触发）。

#### 1.2 共享读取原语

三块解析全部复用 `ELFParserUtils`（`ELFParserUtils.cs`）中的缓冲区读取原语，而非各自手写字节拼接：

- `ReadUInt32(byte[] data, int offset, bool isLittleEndian)`、`ReadInt32(...)`、`ReadUInt16/64`：基于 `System.Buffers.Binary.BinaryPrimitives`，按字节序读取定长整数。
- `ExtractStringFromBytes(data, startOffset)`：读到 NUL 终止的 UTF-8 字符串。
- `ExtractStringFromBytes(data, startOffset, maxLength)`：带长度上界版本，内部把 `maxLength` 夹紧到剩余可读字节并去掉尾部 NUL，专门用于 Note 的 `namesz` 可能越界的场景。

字节序在每个入口方法里统一从 `parser.Header.IsLittleEndian()` 取得一次（`ELFHeader.EI_DATA`，见 `Models/ELFHeader.cs`），再透传给各读取调用。

#### 1.3 节数据获取：`CopySectionData` vs `GetSectionData`

属性与 Exidx 都通过 `parser.CopySectionData(in section)`（`ELFParser.cs`）拿到「单节的原始字节副本」。该方法会校验 `sh_offset + sh_size` 落在文件内、`sh_size` 适配 `int`，畸形节返回空数组，从源头杜绝越界拷贝。注意它**不缓存**（适用于「只有节结构、没有节索引」的场景）；带缓存的版本是 `GetSectionData(int sectionIndex)`，主要供符号名/重定位反复查表使用。

Note 解析则是个例外：`ELFNoteInfo` 不拷贝节，而是**直接在 `parser.FileData` 上按 `sh_offset`/`sh_size` 游走**（见下），并把结束偏移 `endOffset` 夹紧到 `FileData.Length`。

#### 1.4 表驱动派发与 UI 解耦

UI 侧 `ELFAnalyzerControl`（`UserControls/ELFAnalyzerControl.xaml.cs`）用一张 `Action<ELFAnalyzer>[] DisplaySteps` 表把所有展示步骤串起来，`AnalyzeELFFile` 只是 `foreach` 遍历执行每一步。Note/Attribute/Exidx 是表尾三项：`SetNoteInfo`、`SetAttributeInfo`、`SetExidxInfo`。增删一个面板只需改这张表，不动调用逻辑。

### 2. Note 解析（`ELFNoteInfo.cs`）

#### 2.1 职责

遍历所有 `SHT_NOTE` 节，把其中的 note 条目解析为 readelf `--notes` 风格的文本（Owner / Data size / Description 三列），支持 GNU 与 Android 两个命名空间下的常见 note 类型，重点是 NT_GNU_BUILD_ID 与 NT_GNU_ABI_TAG。

#### 2.2 关键类型与方法

```csharp
internal static string GetFormattedNotesInfo(ELFParser parser)            // 入口
private  static string ParseNoteSection(ELFParser, ELFSectionHeader)       // 单节 → FormatNoteSection
private  static string FormatNoteSection(ELFParser, ulong offset, ulong size)  // 核心遍历
private  static ulong  AlignNoteOffset(ulong value, bool is64Bit)          // 对齐
private  static string ProcessNoteEntry(ELFParser, uint type, string owner, byte[] data, int descOffset, int descSize)
private  static string GetNoteDescription(uint type, string owner)         // type → NT_* 名称
private  static string GetBuildID(byte[] data, int descOffset, int descSize)
private  static string GetABIVersion(ELFParser, byte[], int, int)
private  static string FormatNoteDescriptionData(byte[], int, int)
```

#### 2.3 核心算法：SHT_NOTE 遍历、namesz/descsz 与对齐

`GetFormattedNotesInfo` 扫描 `parser.SectionHeaders`，对每个 `sh_type == SHT_NOTE` 的节调 `ParseNoteSection`，后者把 `sh_offset` / `sh_size` 传给 `FormatNoteSection`。若一个 note 都没找到，输出 `"No note segments or sections found."`。

`FormatNoteSection` 在 `parser.FileData` 上按 ELF note 三字段头逐条游走：

1. `endOffset = min(offset + size, FileData.Length)`——夹紧到文件实际长度，防止节大小撒谎。
2. 循环条件 `offset + 12 <= endOffset`，保证至少能读完 `namesz`(4) / `descsz`(4) / `type`(4) 三个 32 位字段。
3. 读三字段后，name 紧跟在头之后：`nameOffset = offset + 12`，用带长度版 `ExtractStringFromBytes` 读取 `namesz` 字节得到 `owner`（如 `"GNU"`、`"Android"`）。
4. **对齐**：描述区紧跟在 name 之后但要对齐——`descOffset = AlignNoteOffset(nameOffset + namesz, is64Bit)`。`AlignNoteOffset` 对 64 位向上对齐到 8 字节边界 `(value + 7) & ~7`，对 32 位对齐到 4 字节 `(value + 3) & ~3`。
5. `ProcessNoteEntry` 用 `type` + `owner` + 描述区字节生成这条 note 的文本；非空则按 readelf 列宽格式输出：`  {owner,-18}0x{descsz:x8}           {noteInfo}`。
6. **推进到下一条**：`next = AlignNoteOffset(descOffset + descsz, is64Bit)`（描述区结束后同样对齐）。若 `next <= offset`（不前进或回绕）立即 `break`，防死循环。

#### 2.4 GNU / Android 类型分派与 build-id / abi-tag

`GetNoteDescription` 把 `(owner, type)` 映射到 NT_* 名称；`ProcessNoteEntry` 在此基础上对若干类型附加描述数据，按 `owner` 嵌套 `switch`：

- **GNU**：
  - `type==1` NT_GNU_ABI_TAG：附加 `GetABIVersion`。该函数要求 `descSize >= 16`，读 4 个 32 位字 `v0.v1.v2.v3`，格式化为 `(ABI version: v0.v1.v2.v3)`（按 OS、major、minor、subminor）。
  - `type==2` NT_GNU_HWCAP：仅名称。
  - `type==3` NT_GNU_BUILD_ID：调 `GetBuildID`。要求 `descSize >= 20`，把整个描述区用 `ConvertUtils.ToHexString(data, descOffset, descSize)` 转成大写十六进制串，输出 `(NT_GNU_BUILD_ID (unique build ID bitstring)\n    Build ID: ...`。
  - `type==4` NT_GNU_GOLD_VERSION：附加从描述区读到的版本字符串 `gold <ver>`。
  - `type==5` NT_GNU_PROPERTY_TYPE_0：仅名称。
- **Android**：
  - `type==1` NT_VERSION：附加 `FormatNoteDescriptionData`——把描述区每字节以小写 `x2` 输出、空格分隔（与 readelf 的 `description data:` 一致）。
  - 其他：`Unknown Android note type N`。
- 其他 owner：`{description} (type: {type})`，未知 type 走 `GetNoteDescription` 的回退分支（`Unknown GNU note type N` / `type 0x...`）。

#### 2.5 数据流

`FileData(SHT_NOTE 区段) → FormatNoteSection 逐条解码 → ProcessNoteEntry 拼描述 → 单一字符串`，由门面 `ELFAnalyzer.GetFormattedNotesInfo()` 返回，最终 `SetNoteInfo` 写入 `ELFNoteInfoControl`（见 §6）。

### 3. ARM/GNU/RISC-V 构建属性解析（`ELFAttributeInfo.cs`）

#### 3.1 职责

遍历 `SHT_GNU_ATTRIBUTES` 与 `SHT_ARM_ATTRIBUTES` 节，按「供应商方言」解析 build attributes 子节，输出 readelf `-A` 风格文本：`Attribute Section: <vendor>` / `File Attributes` / 缩进的 `Tag_xxx: value` 列表。支持的方言：`aeabi`（ARM EABI）、`riscv`，以及通用 GNU（按 `e_machine` 细分 MIPS / PowerPC，其余走通用规则）。

#### 3.2 关键类型与方法

```csharp
public  static string GetFormattedAttributeInfo(ELFParser parser)                 // 入口
private static string ParseAttributeSection(ELFParser, ELFSectionHeader)          // 'A' 版本头 + 多个供应商子节
private static bool   ParseVendorSubsection(ELFParser, byte[], ref int offset, bool le, StringBuilder) // 单子节
private static bool   TryEnterFileScope(byte[], ref int offset, int limit, StringBuilder)  // 进入 Tag_File 作用域
private static void   ParseAEABIAttributes(...) / ParseAEABIFileAttributes(...)   // aeabi 方言
private static void   ParseRiscvAttributes(...)                                   // riscv 方言
private static void   ParseGnuAttributes(ELFParser, ...)                          // gnu 通用 + MIPS/PowerPC
private static int    ReadULEB128(byte[] data, int offset, int endOffset, out int value)  // ULEB128 解码
```

外加大量 `s_aeabi*` 取值表数组、`GetAEABITagName` / `GetAEABIValueTable` / `GetRiscvTagName` 名称映射，以及各类 `AppendAEABI*` 专项格式化函数。

#### 3.3 核心算法：段格式、供应商子节、ULEB128

**段整体格式**（`ParseAttributeSection`）：属性段首字节必须是 `'A'`（0x41，格式版本），否则放弃。其后是连续的供应商子节，循环 `while (offset < data.Length && ParseVendorSubsection(...))`，直到数据耗尽或子节读取返回 `false`。

**供应商子节**（`ParseVendorSubsection`，ARM/GNU 规范布局：`[uint32 length][vendor-name\0][data...]`）：

1. 记 `subSectionStart = offset`，读 4 字节 `subSectionLength`（**该长度包含其自身 4 字节与 vendor 名**），`offset += 4`。
2. 从当前位置读 NUL 终止的 `vendorName`（UTF-8），跳过终止符。输出 `Attribute Section: <vendorName>`。
3. 子节结束位置 `subSectionEnd = subSectionStart + subSectionLength`；若越界或 `length==0` 则夹紧到 `data.Length`。
4. 按 vendor 分派：`"aeabi"` → `ParseAEABIAttributes`；`"riscv"` → `ParseRiscvAttributes`；其余（通常 `"gnu"`）→ `ParseGnuAttributes`。
5. 解析完后若 `offset < subSectionEnd`，把 `offset` 推到 `subSectionEnd`，保证下一个子节定位正确、且 `offset` 不倒退。

**ULEB128 解码**（`ReadULEB128`）：标准小端可变长编码。从 `offset` 起逐字节累加 `value |= (b & 0x7f) << shift`，每字节贡献低 7 位，`shift += 7`；当某字节最高位（续位 0x80）为 0 时停止。`max = min(endOffset, data.Length)` 限定不越界。返回**消费的字节数**（`cur - offset`），由调用方据此推进 offset。`ReadAEABIUleb128` 是 AEABI 路径的便捷包装：内部调 `ReadULEB128` 并就地推进 `offset`、直接返回 value。

#### 3.4 Tag_File 作用域

build attributes 在 vendor data 内是「子-子节」结构，每个子-子节有 1 字节作用域标签 + 4 字节长度。`TryEnterFileScope` 处理这层：

1. 需 `offset + 5 <= limit`，否则返回 `false`。
2. 读作用域标签 `scopeTag = data[offset]`，`offset += 5`（跳过标签 + 4 字节长度）。
3. **只展开 `Tag_File`（scopeTag == 1）**：是则输出 `File Attributes` 并返回 `true`；`Tag_Section`/`Tag_Symbol` 等其它作用域返回 `false`，本解析器不展开它们。

三个方言入口（`ParseAEABIAttributes`/`ParseRiscvAttributes`/`ParseGnuAttributes`）都先调 `TryEnterFileScope`，进入成功后才解析 `(tag, value)` 序列。

#### 3.5 AEABI 方言出表

`ParseAEABIFileAttributes` 循环读 tag（ULEB128），按 tag 分派到专项处理：

- 字符串型 tag `4/5/67`（CPU_raw_name / CPU_name / conformance）→ `AppendAEABIStringAttr`：读 NUL 终止串，`offset += value.Length + 1`，输出 `Tag_xxx: "value"`。
- `7` Tag_CPU_arch_profile → `AppendAEABIProfileAttr`：ULEB128 值映射 `'A'→Application`、`'R'→Realtime`、`'M'→Microcontroller`、`'S'→Application or Realtime`、`0→None`。
- `24/25` 对齐 tag → `AppendAEABIAlignAttr`：值 0→None，1→8-byte，`<=12` 时按 `1<<val` 给出「扩展对齐」；`preserved` 参数区分 needed/preserved 文案（如 `8-byte, except leaf SP`）。
- `32` Tag_compatibility → `AppendAEABICompatibilityAttr`：先 ULEB128 flag，再 NUL 终止 vendor；flag==0 输出 `No`，否则 `flag = N, vendor = ...`。
- `64` Tag_nodefaults → 读取但忽略其值，固定输出 `Tag_nodefaults: True`。
- `65` Tag_also_compatible_with → `AppendAEABIAlsoCompatibleAttr`：先读内层 tag，若内层是 6（CPU_arch）再读一个 ULEB128 走 `s_aeabiCpuArch` 表；内层 0 输出 `None`（吞掉一个空串终止符）；否则当字符串读。
- 其余 tag → `AppendAEABIIntAttr`（默认通道）。

`AppendAEABIIntAttr` 是核心的「表驱动出值」逻辑：

1. `GetAEABIValueTable(tag)` 取该 tag 的取值名称数组（如 `s_aeabiCpuArch`、`s_aeabiFpArch`…）。表存在则读 ULEB128，`val` 落在表范围内取 `table[val]`，否则 `??? (val)`，输出 `Tag_name: text`。
2. 表不存在且 **tag 为奇数** → 当字符串读（readelf 约定：奇数 tag 取字符串）。
3. 表不存在且 **tag 为偶数** → 当 ULEB128 整数读。

`GetAEABITagName` 把 tag 数字映射到名称（4..70）；`GetAEABIValueTable` 把有枚举取值的 tag 映射到对应字符串数组。所有取值表（`s_aeabiCpuArch` 列举到 `v9`、`s_aeabiFpArch`、`s_aeabiVfpArgs` 等）都与 binutils 的 `arm_attr_tag_*` 保持一致。

##### Tag_MPextension_use 的 42/70 双编码

历史上 `Tag_MPextension_use` 存在两个编码：标准的 **42** 和早期工具链误用的 **70**。本解析器在两处都登记了这一别名：

- `GetAEABITagName`：`42 => "Tag_MPextension_use"` 且 `70 => "Tag_MPextension_use"`（两者同名）。
- `GetAEABIValueTable`：`42 => s_aeabiMpExtensionUse` 且 `70 => s_aeabiMpExtensionUse`（共用同一张 `["Not Allowed","Allowed"]` 取值表）。

因此无论目标文件用 42 还是 70 编码该属性，都会被识别为同一语义并以相同文案输出，避免把 70 当成未知 tag。

#### 3.6 RISC-V 方言（`ParseRiscvAttributes`）

进入 File 作用域后循环读 tag，遇 `tag==0` 结束（结束/填充）：

- `5` Tag_RISCV_arch → 字符串。
- `4` Tag_RISCV_stack_align → ULEB128，输出 `N-bytes`。
- `6` Tag_RISCV_unaligned_access → ULEB128，0→`No unaligned access`，1→`Unaligned access`。
- `8/10/12/14/16`（priv_spec / minor / revision / atomic_abi / x3_reg_usage）→ ULEB128，名称由 `GetRiscvTagName` 给出。
- 其它 → `FormatGenericTagValue`（奇字符串、偶 ULEB128），名称 `Tag_unknown_N`。

#### 3.7 通用 GNU 方言与按架构出表（`ParseGnuAttributes`）

进入 File 作用域后循环读 tag（ULEB128），`tag==0` 结束：

- `tag==32` Tag_compatibility → `AppendGnuCompatibility`：ULEB128 flag + NUL vendor，输出 `flag = N, vendor = ...`，返回消费字节数。
- 否则按 `parser.Header.e_machine` 选架构专属表：
  - `EM_MIPS` → `TryFormatMipsAttr`：tag 4 = Tag_GNU_MIPS_ABI_FP（走 `s_mipsFpAbi`），tag 8 = Tag_GNU_MIPS_ABI_MSA（走 `s_mipsMsa`）。
  - `EM_PPC` / `EM_PPC64` → `TryFormatPowerAttr`：tag 4 Power_ABI_FP（拆 `val & 3` 浮点位 + `val & 0xC` long double 位）、tag 8 Power_ABI_Vector（`val & 3`）、tag 12 Power_ABI_Struct_Return。
  - 这些函数返回消费字节数，**返回 -1 表示该 tag 非本架构专属**。
- 当架构表返回 -1（未知/无专属）→ 回退到 `FormatGenericTagValue`（奇字符串、偶 ULEB128，名称 `Tag_unknown_N`）。
- 用返回的消费字节数推进 `offset`。

`FormatGenericTagValue` 实现 readelf 的 `display_tag_value` 通用规则：奇数 tag 读 NUL 字符串、输出 `"value"`，返回 `value.Length + 1`；偶数 tag 读 ULEB128、输出 `val (0xval)`，返回消费字节数。

#### 3.8 数据流

`CopySectionData(section) → 校验 'A' → 逐供应商子节 → TryEnterFileScope → 逐 tag 解码出值 → 单一字符串`。UI 经 `AttributesHelper.GetAttributeInfo(parser)` 取得（见 §5）。

### 4. ARM EHABI 异常展开（`ELFExidxInfo.cs`）

#### 4.1 职责

遍历 `SHT_ARM_EXIDX` 节，解析 ARM Exception Handling ABI 的异常索引表（`.ARM.exidx`），对每个函数条目给出其地址/符号名，并解码三类展开信息：cantunwind、内联 Compact 模型、指向 `.ARM.extab` 的条目（通用 personality routine 模型或 extab 内 Compact 模型），再把展开操作码逐条反汇编成可读文本（`vsp = ...`、`pop {...}`、`finish` 等）。

#### 4.2 关键类型与方法

```csharp
internal static string GetFormattedExidxInfo(ELFParser parser)                   // 入口
private static string ParseExidxSection(ELFParser, ELFSectionHeader)             // 逐条目
private static void   AppendInlineCompact(int unwindInfo, string symbolDesc, StringBuilder)
private static void   AppendExtabEntry(ELFParser, ELFSectionHeader, int offset, int unwindInfo, string symbolDesc, bool le, StringBuilder)
private static byte[] BuildCompactInstructions(ELFParser, int unwindInfo, int compactIndex, long extabFileOff)
private static int    SignExtendPrel31(int value)                                // prel31 符号扩展
private static long   VaddrToFileOffset(ELFParser, ulong vaddr)                  // vaddr → 文件偏移
private static string DescribeAddress(ELFParser, ulong address)                  // 地址 → "0x.. <name>"
private static string FindContainingSymbolName(ELFParser, ulong address)
private static bool   IsAddressInSymbol(ELFSymbol, ulong address)
private static void   ParseUnwindInstructions(byte[] instruction, int len, StringBuilder)
private static void   ProcessSingleByteInstruction(byte cmd, StringBuilder)
private static void   ProcessDoubleByteInstruction(ushort cmd, StringBuilder)
```

#### 4.3 核心算法：prel31 符号扩展

`.ARM.exidx` 与 extab 中的所有相对偏移都是 **prel31**：31 位有符号、相对「该偏移字所在的虚拟地址」。`SignExtendPrel31`：

```csharp
value &= 0x7FFFFFFF;                 // 取低 31 位
if ((value & 0x40000000) != 0)      // bit30 是符号位
    value |= unchecked((int)0x80000000);  // 向上符号扩展到 32 位
return value;
```

即把 bit30 作为符号位扩展到完整 `int`，从而可表示前后两个方向的相对偏移。

#### 4.4 核心算法：索引表条目遍历

`ParseExidxSection`：`data = CopySectionData(section)`，条目数 `entryCount = sh_size / 8`（每条 8 字节 = 2 个字）。对第 `idx` 条（`offset = idx * 8`）：

1. 第一个字：`addrOffset = SignExtendPrel31(ReadInt32(data, offset))`——函数地址的 prel31 偏移。
2. **函数绝对虚拟地址**：`absAddr = sh_addr + offset + addrOffset`（prel31 相对该字的虚拟地址，即 `sh_addr + offset`）。
3. `symbolDesc = DescribeAddress(parser, absAddr)`——解析为 `0x.. <name>`（见 §4.7）。
4. 第二个字 `unwindInfo` 决定分支：
   - `unwindInfo == 1`：特殊值 cantunwind，输出 `... : @0x1 (cantunwind)`。
   - `(unwindInfo & 0x80000000) != 0`（bit31 置位）：**内联 Compact 模型**，指令就在条目内 → `AppendInlineCompact`。
   - 否则：`unwindInfo` 是指向 `.ARM.extab` 的 prel31 偏移 → `AppendExtabEntry`。
5. 每条目后追加一个空行分隔。

#### 4.5 内联 Compact 模型（`AppendInlineCompact`）

条目第二字 bit31==1 时，该字自带展开信息：高 7 位（bits 30-24）是 Compact 模型索引 `compactIndex = (unwindInfo >> 24) & 0x7F`，低 24 位 `instruction = unwindInfo & 0xFFFFFF` 是展开操作码。把低 24 位拆成 3 个字节（高位在前）组成 `byte[]`，交给 `ParseUnwindInstructions` 解码。输出 `... : @0x<unwindInfo>` + `Compact model index: N` + 解码出的指令行。

#### 4.6 extab 条目：personality routine vs extab Compact（`AppendExtabEntry` / `BuildCompactInstructions`）

当条目指向 `.ARM.extab` 时：

1. `extabRel = SignExtendPrel31(unwindInfo)`；`extabVaddr = sh_addr + offset + 4 + extabRel`（注意是相对**第二字**的地址，故 `+4`）。输出 `... : @0x<extabVaddr>`。
2. **vaddr → 文件偏移**：`extabFileOff = VaddrToFileOffset(parser, extabVaddr)`，因为虚拟地址通常≠文件偏移，必须换算后才能在 `FileData` 上读。越界则返回。
3. 读 extab 首字 `unwindInfo = ReadInt32(FileData, extabFileOff)`：
   - **bit31 == 0（通用模型）**：首字低 31 位是指向 personality routine 的 prel31 偏移。`personalityAddr = extabVaddr + SignExtendPrel31(unwindInfo)`，再 `DescribeAddress`，输出 `Personality routine: <desc>`。
   - **bit31 == 1（extab 内 Compact 模型）**：`compactIndex = (unwindInfo >> 24) & 0x7F`，输出 `Compact model index: N`，再由 `BuildCompactInstructions` 装配指令字节数组并解码。

`BuildCompactInstructions` 区分 compactIndex：

- `compactIndex != 1`：指令仅在首字低 24 位，拆 3 字节返回。
- `compactIndex == 1`：首字 bits 16-23 是「后续额外 dword 数」`remainDWords`。指令字节数组 = 首字低 16 位的 2 字节 + `remainDWords * 4` 字节；后续 dword 从 `extabFileOff + 4` 起逐个读取，小端时 `Array.Reverse` 每 4 字节翻转使字节顺序符合「高位在前」的解码约定。越界即停止。

#### 4.7 地址→符号名（`DescribeAddress` / `FindContainingSymbolName` / `IsAddressInSymbol`）

`DescribeAddress` 把地址格式化为 `0x..`（无符号）或 `0x.. <name>`/`0x.. <name+0xoff>`（有包含符号）。

`FindContainingSymbolName` 遍历 `parser.Symbols`（按 `SectionType` 分组的符号列表），找「包含该地址且起始地址最大」的命名符号（贪心取最贴近的）：

- `IsAddressInSymbol` 判定：符号已定义（`StShndx != 0`）；类型为 `STT_FUNC` / `STT_OBJECT` / `STT_NOTYPE`；`StValue <= address`；若 `StSize != 0` 则 `address < StValue + StSize`。
- 跳过名字以 `$` 开头的 **ARM 映射符号**（`$a`/`$t`/`$d`/`$x`），与 readelf 行为一致。
- 命中后 `delta = address - bestStart`，delta==0 输出纯名字，否则 `name+0xdelta`。

`VaddrToFileOffset` 遍历节头，跳过 `sh_addr==0` / `sh_size==0` / `SHT_NOBITS` 的节，找到 `[sh_addr, sh_addr+sh_size)` 覆盖 vaddr 的节，返回 `sh_offset + (vaddr - sh_addr)`；找不到返回 -1。

#### 4.8 展开操作码解码（`ParseUnwindInstructions`）

把上面装配好的指令字节序列按 ARM EHABI 操作码逐条解码。主循环：取当前字节 `cmd`，若 `IsValidSingleByteInstruction(cmd)` 为真或剩余不足 2 字节，则按单字节指令处理（`ProcessSingleByteInstruction`，前缀打印 `0xXX`，offset+1）；否则取两字节 `doubleCmd = (b0<<8)|b1` 按双字节指令处理（`ProcessDoubleByteInstruction`，前缀打印 `0xXX 0xXX`，offset+2）。

`IsValidSingleByteInstruction` 用区间判定哪些首字节是单字节指令（`<=0x7F`、`0x90-0xAF`、`0xB0`、`0xB4-0xBF`、`0xC0-0xC5`、`0xCA-0xFF`），其余首字节按双字节解码。

**单字节指令**（`ProcessSingleByteInstruction`，部分代表）：

- `0xB0` → `finish`（BX r14）。
- `0x00-0x3F` → `vsp = vsp + CalcVsp(cmd)`，`CalcVsp = ((cmd & 0x3F) + 1) << 2`（即 `(imm6<<2)+4`）。
- `0x40-0x7F` → `vsp = vsp - CalcVsp(cmd)`。
- `0x90-0x9F`（且非 0x9C/0x9F）→ `vsp = r<imm4>`；0x9C/0x9F → `reserved`。
- `0xA0-0xAF` → `pop {r4-rN[, r14]}`，由 `AppendPopList(prefix="r", base=4, includeLr=true)` 生成：低 3 位 `regMask` 决定 r4..r(4+mask)，bit3 置位追加 r14(LR)。
- `0xB4` → `pop`；`0xB5` → `pop vsp`。
- `0xB8-0xBF` / `0xD0-0xD7` → `pop {D8-DN}`（`AppendPopList(prefix="D", base=8)`）。
- `0xC0-0xC5` → `pop {wR10-wRN}`。
- `0xB6/0xB7`、`0xCA-0xCF`、`0xD8-0xFF` → `Spare`；其它 → `unknown command`。

**双字节指令**（`ProcessDoubleByteInstruction`，部分代表）：

- `0x8000` → `Refuse to unwind`。
- `0x8001-0x8FFF` → `pop {r4-r15}` 位掩码：低 12 位作掩码，`AppendMaskedRegs(mask, 12, "r", 4)`，bit i 置位则列出 r(4+i)。
- `0xB101-0xB10F` → `pop {r0-r3}` 位掩码（4 位，base 0）。
- `0xB200-0xB2FF` → `vsp = vsp + (0x204 + (uleb128 << 2))`，`uleb128 = cmd & 0xFF`。
- `0xB300-0xB3FF` / `0xC800-0xC8FF` / `0xC900-0xC9FF` → `pop` VFP D 寄存器范围：`ssss=(cmd>>4)&0xF`、`cccc=cmd&0xF`，`0xC8xx` 段 base 偏移 +16，`AppendRegRange("D", ssss+idx, cccc)`。
- `0xC600-0xC6FF` → wR 范围；`0xC701-0xC70F` → wCGR 位掩码。
- 若干区间 → `Spare`；其它 → `unknown double-byte command`。

寄存器列表构造复用三个 helper：`AppendPopList`（连续 + 可选 LR）、`AppendMaskedRegs`（按位掩码）、`AppendRegRange`（连续范围），最终都经 `AppendPopRegs` 在列表非空时输出 `pop {<逗号分隔>}`（用 `ConvertUtils.EnumerableToString`）。

#### 4.9 数据流

`CopySectionData(.ARM.exidx) → 逐条目解码 (prel31→vaddr→符号名 + unwindInfo 分支) →（需要时）vaddr→fileoff 读 .ARM.extab → 装配指令字节 → ParseUnwindInstructions 反汇编 → 单一字符串`。无条目时输出 `"There are no exception index entries in this file."`，UI 据此隐藏面板。

### 6. `ELFAnalyzer` 门面与端到端数据流

`ELFAnalyzer`（`ELFAnalyzer/ELFAnalyzer.cs`）是对外门面：构造时按文件路径或字节数组创建 `ELFParser`（即触发整套结构解析），并暴露公有 `Parser` 属性。它直接转发若干格式化方法，其中本节相关的是：

```csharp
public string GetFormattedNotesInfo() => ELFNoteInfo.GetFormattedNotesInfo(Parser);
```

属性与 Exidx 不在门面上开方法，而是由 UI 直接经 helper + `analyzer.Parser` 取得。

**端到端数据流**（以宿主 `ELFAnalyzerControl` 为终点，见 `UserControls/ELFAnalyzerControl.xaml.cs`）：

1. `AnalyzeELFFile(filePath)` → `new ELFAnalyzer(filePath)`（构造即完成 ELF 结构解析）。
2. `foreach` 遍历 `DisplaySteps` 表驱动数组，依次执行每个展示步骤；表尾三项是本节关注的：
   - `SetNoteInfo`：`analyzer.GetFormattedNotesInfo()` → `ELFNoteInfoControl.SetNoteInfo(text)`；`text.Length > 0` 决定 Tab 可见性。
   - `SetAttributeInfo`：`AttributesHelper.GetAttributeInfo(analyzer.Parser)` → `ELFAttributeInfoControl.SetAttributeInfo(text)`；非空决定可见性。
   - `SetExidxInfo`：`ExidxInfoHelper.GetExidxInfo(analyzer.Parser)` → `ELFExidxInfoControl.SetExidxInfo(text)`；通过判断文本是否含 `"There are no exception index entries"` 决定面板是否隐藏。
3. 解析过程中可能抛出的一组 IO/格式异常被 `AnalyzeELFFile` 统一捕获并弹 `MessageBox`，不致使整个分析流程崩溃。

整体上：**文件字节 → `ELFParser` 解析为结构（节头/符号/动态段）→ 三个高级解析器各自遍历相关节并产出 readelf 风格文本 → helper/门面转发 → 控件显示**。每个高级解析器只读不写 `parser` 状态，输入只有 `ELFParser`、输出只有字符串，保证了可独立测试与按需调用。


## 6 加解密与哈希工具


本章描述 PersonalTools 工具箱中"加解密与哈希"功能域的设计。该功能域包含五个对用户可见的子工具：AES 对称加解密、RSA 非对称加解密/签名、Base64 编解码、CRC 校验、哈希（MD5/SHA 系列/SHA3）。

### 总体设计

#### 分层结构（核心服务 + 控件）

整个功能域遵循同一条设计主线：**把"加密/算法核心"与"WPF 控件交互逻辑"分离**。

- 算法核心放在项目根目录的纯静态服务类中，不引用任何 WPF 类型：`AesCryptoService`、`RsaCryptoService`、`CrcCalculator`、`Sha224`、`Sha3`。这些类只接受字节/字符串与算法参数，返回结果，便于复用与单元测试。
- 交互逻辑放在 `UserControls/` 下对应的 `*.xaml.cs` 代码隐藏类：`AesEncryptionControl`、`RsaEncryptionControl`、`Base64EncoderDecoderControl`、`CrcComputeControl`、`HashComputeControl`。它们负责读取 UI 输入、校验、调用核心服务、把结果写回 UI、弹窗报错。

核心服务类的 XML 文档注释中明确写了这一动机："与 UI 无关。把 Aes/CryptoStream 等加密类型从控件中剥离，降低控件类耦合"（见 `AesCryptoService.cs`、`CrcCalculator.cs` 注释）。

#### 共享读取原语（`ConvertUtils` / `FileDropHelper`）

所有子工具共享一组底层原语，避免在每个控件里重复样板代码：

- `ConvertUtils.HexStringToByteArray(string hex)`（`ConvertUtils.cs`）：把十六进制字符串转字节数组。先 `Replace` 去掉空格和 `0x` 前缀，再校验长度必须为偶数（否则抛 `ArgumentException`），最后两两调用 `Convert.ToByte(..., 16)` 解析。
- `ConvertUtils.ToHexString(byte[])` / `ToHexString(byte[], start, length)`：基于 `Convert.ToHexString` 的大写无分隔十六进制输出。
- `ConvertUtils.InputBytes(string text, bool isHex)`：按输入模式统一取字节——`isHex=true` 走 `HexStringToByteArray`，否则走 `Encoding.UTF8.GetBytes`。这是"明文/输入既可按字符串又可按 hex"的统一入口。
- `ConvertUtils.ReverseBits(uint value, int width)`：逐位反转，供 CRC 的 refin/refout 使用。
- `FileDropHelper.GetFirstDroppedFile(DragEventArgs)`：从拖放事件取第一个文件路径（无则返回 null）。
- `FileDropHelper.ReadAllBytes(string)`：用 `FileStream` + `ReadExactly` 读全部字节。

#### 统一的输入模式约定

每个子工具的输入框旁都有一对互斥单选按钮（String / Hex）。约定是：

- "字符串模式" → 输入按 UTF-8 编码取字节。
- "Hex 模式" → 输入按十六进制解析取字节。
- 密文/编码结果一律以十六进制（AES/RSA/CRC/Hash）或 Base64（Base64 工具）字符串呈现。
- 文件拖放统一行为：读文件字节 → 把内容以 hex 显示在输入框 → 自动把单选切到 Hex 模式（保证二进制数据不丢失）。

#### 错误处理风格

控件层统一采用"分类型 catch + `MessageBox.Show` 弹窗"的风格。典型捕获类型为 `CryptographicException`、`ArgumentException`、`FormatException`（hex 解析失败）、`ArgumentNullException`、`ArgumentOutOfRangeException`、`IOException`、`UnauthorizedAccessException`。服务层则倾向于把底层异常包装成带中文消息的 `CryptographicException` 再抛出（RSA），或直接让校验失败抛标准异常（AES/CRC）。

### AES 对称加解密

#### 职责

提供 AES 的加密与解密，支持多种工作模式与填充方式，支持密钥/IV 以字符串或 hex 两种形式输入，支持把明文一侧按文本或 hex 处理。核心在 `AesCryptoService`（`AesCryptoService.cs`），UI 在 `AesEncryptionControl`（`UserControls/AesEncryptionControl.xaml.cs`）。

#### 关键类型与方法

`AesCryptoService`（internal static）：

- `string Encrypt(string input, byte[] key, byte[]? iv, CipherMode mode, PaddingMode padding, bool hexMode)`
- `string Decrypt(string input, byte[] key, byte[]? iv, CipherMode mode, PaddingMode padding, bool hexMode)`
- `private static Aes CreateAes(byte[] key, byte[]? iv, CipherMode mode, PaddingMode padding)`

`hexMode` 控制明文侧表示：加密时为输入侧、解密时为输出侧。密文一律是十六进制字符串。

控件侧类型：
- `AesModeOption { string Name; CipherMode Mode }`、`AesPaddingOption { string Name; PaddingMode Padding }`：下拉框数据项。
- `AesEncryptionControl.TryGetAesParams(out byte[] key, out byte[]? iv, out CipherMode mode)`：参数校验与提取。

#### 核心算法与逻辑

**`CreateAes` 按密钥长度设 KeySize**：先 `Aes.Create()`，然后 `aesAlg.KeySize = key.Length * 8`——根据传入密钥的实际字节长度推导出 128/192/256 位的 KeySize，再赋 `Key`。若 `iv != null` 则赋 `IV`，最后设置 `Mode` 与 `Padding`。这样调用方不必显式指定 KeySize，长度本身即决定 AES-128/192/256。

**加密的数据流与 FlushFinalBlock**（`Encrypt`）：

1. 按 `hexMode` 取明文字节：`hexMode ? ConvertUtils.HexStringToByteArray(input) : Encoding.UTF8.GetBytes(input)`。
2. `CreateEncryptor(aesAlg.Key, aesAlg.IV)` 得到 `ICryptoTransform`（此处 `#pragma warning disable CA5401` 抑制"不要用非随机 IV"的分析告警，因为 IV 由用户指定）。
3. 建立 `MemoryStream msEncrypt`，把 `CryptoStream`（Write 模式）包在外面，`Write(inputBytes)` 写入。
4. **关键点**：`CryptoStream` 放在 `using (...) { ... }` 块里，块结束时 `Dispose` 会自动触发 `FlushFinalBlock`，把最后一个分组（含 PKCS7 等填充）写出。必须等 `CryptoStream` 释放之后再 `msEncrypt.ToArray()` 取字节，否则会丢掉尾块（代码注释明确说明了这一点）。
5. `ConvertUtils.ToHexString(encryptedBytes)` 输出十六进制密文。

**解密的数据流**（`Decrypt`）：

1. 密文一律 `ConvertUtils.HexStringToByteArray(input)` 解析为字节。
2. `CreateDecryptor` 得到解密变换。
3. 把密文字节包进 `MemoryStream`，外层 `CryptoStream`（Read 模式），`CopyTo(resultStream)` 流式读出全部明文字节。
4. 按明文表示输出：`hexMode ? ConvertUtils.ToHexString(...) : Encoding.UTF8.GetString(...)`。hex 模式用于避免二进制明文经 UTF-8 解码丢失。

**`TryGetAesParams` 校验逻辑**：

1. 密钥非空校验；
2. 密钥长度校验：`IsValidKeyLength` 调 `GetKeyBytes` 取字节后判断 `Length is 16 or 24 or 32`（按实际字节数校验，兼容多字节字符）；
3. 从下拉框取 `mode`；
4. 仅当模式为 CBC/CFB/OFB 时要求 IV：IV 非空 + `IsValidIVLength`（字节长度须等于 16）。`GetIVBytes` 注释强调"不做截断/补齐"，避免静默改变用户 IV；
5. 任一校验失败弹提示并返回 `false`，加解密入口据此提前 return。

控件下拉框初始化（`InitializeAesComboBoxes`）：模式有 CBC（默认）/ECB/CFB/OFB（ECB/CFB/OFB 用 `#pragma warning disable CA5358` 抑制"不安全模式"告警）；填充有 PKCS7（默认）/Zeros/PKCS5（实际映射到 PKCS7）/ISO7816（映射到 ISO10126）/ANSI X923。

#### 数据流（input → parse → display）

加密：`AesInput.Text` →（`TryGetAesParams` 校验 key/iv/mode；按 `AesInputStringRadio` 取 hexMode）→ `AesEncryptString` → `AesCryptoService.Encrypt` → 十六进制密文写入 `AesResult.Text`。解密方向相反：`AesResult.Text` → `AesCryptoService.Decrypt` → 明文写入 `AesInput.Text`。文件拖放 `AesTab_Drop` → `ProcessFileForAesEncryption`：读字节 → `AesInput` 显示 hex → 勾选 `AesInputHexRadio`。

#### 事件流与错误处理

- `AesEncrypt_Click` / `AesDecrypt_Click`：先非空校验，再 `TryGetAesParams`，再调服务。捕获 `CryptographicException`（如填充/密钥错）、`ArgumentException`、`FormatException`（hex 非法）分别弹窗。
- `AesClear_Click`：清空输入/结果/密钥/IV 四个框。
- `Grid_PreviewDragOver`：`e.Handled = true`，保证拖放事件不被子控件拦截。

### RSA 非对称加解密 / 签名

#### 职责

提供 RSA 密钥对生成、PEM 导入与校验、加密/解密、签名/验签。核心在 `RsaCryptoService`（`RsaCryptoService.cs`），UI 在 `RsaEncryptionControl`（`UserControls/RsaEncryptionControl.xaml.cs`）。

#### 关键类型与方法

`RsaCryptoService`（internal static）：

- `string Encrypt(string input, string publicKey, bool isString)`
- `string Decrypt(string input, string privateKey)`
- `string Sign(string input, string privateKey, bool isString)`
- `bool Verify(string input, string signature, string publicKey, bool isString)`
- `(string PublicKey, string PrivateKey) GenerateKeyPair(int keySize)`
- `bool IsValidPem(string pem)`

`isString=true` 时明文按 UTF-8，否则按十六进制（`ConvertUtils.HexStringToByteArray`）。

控件侧：`RsaEncryptionControl.RsaKeySizeOption { string Name; int KeySize }`（私有嵌套类，下拉框数据项）。

#### 核心算法与逻辑

**填充/算法选择**：加密/解密统一使用 **OAEP-SHA256**（`RSAEncryptionPadding.OaepSHA256`）。代码注释解释了选择动机："避免 PKCS#1 v1.5 的填充预言攻击（会减小可加密明文上限，512 位密钥过小无法使用）"。签名/验签使用 **PKCS#1 + SHA256**（`HashAlgorithmName.SHA256`、`RSASignaturePadding.Pkcs1`）。

**加密**（`Encrypt`）：`RSA.Create()` → 按 `isString` 取明文字节 → `rsa.ImportFromPem(publicKey)` 导入公钥 → `rsa.Encrypt(inputBytes, OaepSHA256)` → `ConvertUtils.ToHexString` 输出十六进制密文。导入或加密任何异常（`ArgumentException` / `CryptographicException` / 其它 `Exception`）都被统一包装为带中文消息 `"导入公钥或加密失败: {ex.Message}"` 的 `CryptographicException` 重新抛出。

**解密**（`Decrypt`）：密文 hex 解析 → `ImportFromPem(privateKey)` → `rsa.Decrypt(..., OaepSHA256)` → `Encoding.UTF8.GetString` 输出文本（解密侧固定按 UTF-8 文本呈现，无 hex 选项）。同样把异常包装为 `"导入私钥或解密失败: ..."`。

**签名**（`Sign`）：按 `isString` 取数据字节 → 导入私钥 → `rsa.SignData(inputBytes, SHA256, Pkcs1)` → 十六进制签名值。

**验签**（`Verify`）：取数据字节 + `signature` hex 解析 → 导入公钥 → `rsa.VerifyData(...)`，直接返回 `bool`。这里与其它方法不同：捕获 `CryptographicException` / `ArgumentException` 时**返回 false 而非抛异常**（验签失败本身是正常业务结果）。

**生成密钥对**（`GenerateKeyPair`）：`RSA.Create(keySize)` → `(ExportRSAPublicKeyPem(), ExportRSAPrivateKeyPem())` 返回 PKCS#1 格式的 PEM 元组。

**PEM 校验**（`IsValidPem`）：尝试 `RSA.Create()` + `ImportFromPem(pem)`，成功返回 true，捕获 `CryptographicException` / `ArgumentException` 返回 false。用于导入时判断文件是否为合法密钥。

#### 数据流（input → parse → display）

- 加密：`RsaInput.Text` + `RsaPublicKey.Text` → `Encrypt` → 十六进制密文写 `RsaResult.Text`。
- 解密：`RsaResult.Text` + `RsaPrivateKey.Text` → `Decrypt` → 明文写 `RsaInput.Text`。
- 签名：`RsaInput.Text`（数据）+ `RsaPrivateKey.Text` → `Sign` → 签名写 `RsaResult.Text`。
- 验签：`RsaInput.Text`（原始数据）+ `RsaResult.Text`（签名）+ `RsaPublicKey.Text` → `Verify` → 弹"验签成功/失败"。
- 生成密钥对：下拉框选 KeySize → `GenerateKeyPair` → 公私钥分别写入两个文本框。

#### 事件流与错误处理

- `RsaEncrypt_Click` / `RsaDecrypt_Click` / `RsaSign_Click` / `RsaVerify_Click`：先逐项非空校验（输入、对应密钥/签名），再用**内层 try-catch**（捕获 `CryptographicException` / `ArgumentException` 弹专门消息如"RSA加密失败"）包住服务调用，外层 try-catch 兜底。`RsaInputStringRadio.IsChecked == true` 决定 `isString`。
- `RsaGenerateKeyPair_Click`：取下拉框 `RsaKeySizeOption`，调 `GenerateKeyPair`，成功弹"密钥对生成成功"。
- `RsaImportPublicKey_Click` / `RsaImportPrivateKey_Click` → 共用 `ImportKey(title, isPublic)`：`OpenFileDialog`（过滤 *.pem/*.txt/*.*）→ `File.ReadAllText` → `IsValidPem` 校验失败弹"不是有效的公钥/私钥文件"→ 写入对应文本框。捕获 `IOException` / `UnauthorizedAccessException`。
- `RsaClear_Click`：清空输入/结果/公钥/私钥。
- `RsaTab_Drop` → `ProcessFileForRsaEncryption`：读首个文件字节 → `RsaInput` 显示 hex → 勾 `RsaInputHexRadio`。
- 下拉框（`InitializeRsaComboBoxes`）：512（不推荐）/1024（不推荐）/2048（推荐，默认 SelectedIndex=2）/3072/4096。

### Base64 编解码

#### 职责

提供 Base64 编码与解码。编码支持输入按字符串或 hex；解码时自动检测结果是否含不可见字符，决定以文本还是 hex 显示。UI 在 `Base64EncoderDecoderControl`（`UserControls/Base64EncoderDecoderControl.xaml.cs`）。本工具直接使用 .NET 内置的 `Convert.ToBase64String` / `Convert.FromBase64String`，没有单独的服务类。

#### 关键类型与方法

- `Base64Encode_Click` / `Base64Decode_Click`：编解码入口。
- `private static bool ContainsInvisibleCharacters(byte[] bytes)`：不可见字符检测。

#### 核心算法与逻辑

**编码**（`Base64Encode_Click`）：按单选 `Base64HexInputRadio` 取字节——Hex 模式走 `ConvertUtils.HexStringToByteArray`，否则 `Encoding.UTF8.GetBytes`——再 `Convert.ToBase64String(bytes)` 写入 `Base64Result.Text`。

**解码**（`Base64Decode_Click`）：`Convert.FromBase64String(input)` 得字节；调用 `ContainsInvisibleCharacters` 判断：

- 含不可见字符 → 用 `ConvertUtils.ToHexString(bytes)` 以 hex 显示（避免二进制数据在文本框里被破坏）。
- 否则 → `Encoding.UTF8.GetString(bytes)` 以文本显示。

**不可见字符检测**（`ContainsInvisibleCharacters`）：遍历每个字节，若 `b < 32 且 b != 9 && b != 10 && b != 13`（即排除 Tab/换行/回车的控制字符）则判定为不可见，返回 true。这是"自动选 hex / 文本呈现"的判据。

#### 数据流（input → parse → display）

编码：`Base64Input.Text` →（按模式取字节）→ Base64 字符串写 `Base64Result.Text`。解码：`Base64Result.Text` → 字节 →（不可见字符检测）→ hex 或文本写回 `Base64Input.Text`。文件拖放 `Base64Tab_Drop` → `ProcessFileForBase64Encoding`：`Base64Input` 显示 hex、`Base64Result` 显示 Base64，并切到 Hex 模式。

#### 事件流与错误处理

- 编/解码先非空校验。捕获 `ArgumentNullException` / `FormatException`（hex 或 Base64 非法）/ `ObjectDisposedException` 弹窗。
- `Base64Clear_Click`：清空两个框。`Grid_PreviewDragOver`：吞掉拖放预览事件。

### CRC 校验

#### 职责

按一组内置的常用 CRC 算法参数对输入字节做位串行 CRC 计算。核心在 `CrcCalculator`（`CrcCalculator.cs`），UI 在 `CrcComputeControl`（`UserControls/CrcComputeControl.xaml.cs`）。

#### 关键类型与方法

`CrcAlgorithm`（internal sealed，参数模型）：

- `string Name`、`string PolynomialFormula`（如 `x16 + x12 + x5 + 1`，仅供显示）
- `int Width`（位宽：8/16/32）
- `uint Polynomial`（多项式）
- `uint InitialValue`（init 初值）
- `uint FinalXor`（xorout 输出异或值）
- `bool ReverseInput`（refin，输入字节是否反转）
- `bool ReverseOutput`（refout，结果是否反转）

`CrcCalculator`（internal static）：

- `static IReadOnlyList<CrcAlgorithm> Algorithms`：内置算法表（CRC-8 系列、CRC-16 各变体 IBM/MAXIM/USB/MODBUS/CCITT/CCITT-FALSE/X25/XMODEM/DNP、CRC-32、CRC-32/MPEG-2）。
- `static uint Compute(byte[] data, CrcAlgorithm algo)`：核心计算。

#### 核心算法与逻辑

`Compute` 实现的是按位（bit-serial）的 CRC：

1. `crc = algo.InitialValue`。
2. 预算掩码与最高位：`mask = (1UL << Width) - 1`（用于把 crc 限制在 Width 位内）、`topBit = 1UL << (Width - 1)`（当前宽度的最高位）。
3. 对每个输入字节 `value`：
   - 若 `ReverseInput` 为真，先 `ConvertUtils.ReverseBits(data[i], 8)` 反转该字节的 8 位（refin）。
   - `crc ^= (uint)(value << (Width - 8))`：把字节对齐到 crc 的高 8 位再异或进去。
   - 内层循环 8 次（逐位处理）：若 `crc & topBit != 0` 则 `crc = (crc << 1) ^ Polynomial`，否则 `crc <<= 1`；每步后 `crc &= mask` 保持在 Width 位内。
4. 全部字节处理完后：若 `ReverseOutput` 为真，`crc = ConvertUtils.ReverseBits(crc, Width)`（refout，按整个 Width 反转）。
5. `crc ^= FinalXor`（xorout），再 `crc &= mask`。
6. 返回 `crc`。

`ConvertUtils.ReverseBits(value, width)` 的逻辑是：循环 `width` 次，每次把 `result` 左移一位、把 `value` 的最低位移入 `result`、`value` 右移一位——即把低 `width` 位按位反序。

**按宽度格式化输出**（控件 `CalculateCRC_Click`）：根据 `selectedAlgorithm.Width` 选格式串——`<=8 → "X2"`、`<=16 → "X4"`、否则 `"X8"`——用 `crcResult.ToString(formatString, CultureInfo.InvariantCulture)` 输出对应位宽的大写十六进制（不足补零）。

#### 数据流（input → parse → display）

`CRCInputTextBox.Text` →（`ConvertUtils.InputBytes(input, CRCHexInputRadio.IsChecked == true)` 取字节）→ `CrcCalculator.Compute(bytes, selectedAlgorithm)` → 按宽度格式化 → 写入 `CRCResultLabel.Content`。文件拖放 `CrcTab_Drop` → `ProcessFileForCrcCalculation`：输入框显示 hex，切到 Hex 模式。

#### 事件流与错误处理

- 控件构造时 `InitializeCRCAlgorithmComboBox`：`CRCAlgorithmComboBox.ItemsSource = CrcCalculator.Algorithms`、`DisplayMemberPath = "Name"`、默认选第 0 项。这是数据驱动的下拉绑定，新增算法只需往 `Algorithms` 表追加一项。
- `CalculateCRC_Click`：先校验是否选了算法、输入是否为空，再计算。捕获 `FormatException` / `ArgumentNullException` / `ArgumentOutOfRangeException` 弹窗。
- `ClearCRC_Click`：清空输入并把结果标签复位为"等待计算..."。`Grid_PreviewDragOver` 吞掉预览拖放。

### 哈希（MD5 / SHA 系列 / SHA3）

#### 职责

提供两类哈希计算：(1) 6 种定长哈希（MD5、SHA1、SHA224、SHA256、SHA384、SHA512），用数据驱动方式渲染成多行；(2) SHA3（256/384/512），单独一块带算法下拉框。UI 在 `HashComputeControl`（`UserControls/HashComputeControl.xaml.cs`），行视图模型在 `HashAlgorithmRow`（`UserControls/HashAlgorithmRow.cs`），自实现的 SHA-224 在 `Sha224`（`Sha224.cs`），SHA3 包装在 `Sha3`（`Sha3.cs`）。

#### 关键类型与方法

`HashAlgorithmRow`（internal sealed，实现 `INotifyPropertyChanged`）：

- `string Name`、`Func<byte[], byte[]> HashFunc`（该行的哈希函数）。
- 派生显示属性：`Header`（"{Name}计算"）、`InputPrompt`、`CalcButtonText`、`ResultPrompt`。
- `string InputText`（带变更通知）、`bool IsHexMode`（是否按 hex 解析输入）、`string Result`（默认"等待计算..."，带变更通知）。
- 构造：`HashAlgorithmRow(string name, Func<byte[], byte[]> hashFunc)`。

`HashComputeControl`：
- 私有字段 `List<HashAlgorithmRow> hashRows`，依次绑定 `MD5.HashData`、`SHA1.HashData`、`Sha224.HashData`、`SHA256.HashData`、`SHA384.HashData`、`SHA512.HashData`。除 SHA224 外均直接用 .NET 内置静态 `HashData`。
- `static void ComputeRow(HashAlgorithmRow row)`、`void CalculateAndDisplayHashValues(byte[] data)`。

`Sha3AlgorithmOption(string name, int value)`：SHA3 下拉框数据项。

`Sha224`（internal static）：`static byte[] HashData(byte[] data)`、`private static uint RotR(uint x, int n)`。

`Sha3`（internal sealed，`IDisposable`，主构造 `Sha3(int variant)`）：
- 实例 `byte[] ComputeHash(byte[] input)`（用构造传入的位宽）。
- 静态 `string ComputeHash(string input, int outputLengthBits = 256)`、`byte[] ComputeHash(byte[] input, int outputLengthBits = 256)`。
- 静态 `bool IsSupported`。

#### 核心算法与逻辑

**数据驱动渲染（表驱动派发）**：6 行哈希不再为每个算法各写一套命名控件，而是把 `(Name, HashFunc)` 装进 `hashRows` 列表，`HashItemsControl.ItemsSource = hashRows`，由 XAML 的 `ItemsControl` 数据模板统一渲染（`HashAlgorithmRow` 注释说明这是为了"压低 XAML 生成代码的复杂度"）。计算/清空按钮通过 `Button.DataContext` 拿到对应行：`HashCompute_Click` 用模式匹配 `sender is Button { DataContext: HashAlgorithmRow row }` 取出该行再 `ComputeRow(row)`。

`ComputeRow`：输入非空校验后，按 `row.IsHexMode` 取字节（hex → `ConvertUtils.HexStringToByteArray`，否则 UTF-8），调用 `row.HashFunc(inputBytes)`，`ConvertUtils.ToHexString` 后写回 `row.Result`（经 `INotifyPropertyChanged` 自动刷新 UI）。`CalculateAndDisplayHashValues` 用于文件拖放：对所有行直接用同一份 `data` 调 `HashFunc` 并写结果。

**自实现 SHA-224**（`Sha224.HashData`，遵循 FIPS 180-4）：.NET 的 `System.Security.Cryptography` 未内置 SHA-224，这里**不是对 SHA-256 结果的简单截断**，而是用 SHA-256 的压缩函数配 SHA-224 专属初值：

1. 初值用 SHA-224 专属的 8 个 `h0..h7`（`0xc1059ed8` 等）；轮常量 `K[64]` 与 SHA-256 相同。
2. 预处理（消息填充）：`bitLength = data.Length * 8`；追加一个 `0x80` 字节；补 0 直到总长度 `≡ 56 (mod 64)`；再追加 8 字节的 64 位大端比特长度（`BinaryPrimitives.WriteInt64BigEndian`）。
3. 按 64 字节分块：前 16 个字 `w[0..15]` 用大端读入（`ReadUInt32BigEndian`）；`w[16..63]` 按消息扩展公式生成：`s0 = RotR(w[i-15],7)^RotR(w[i-15],18)^(w[i-15]>>3)`、`s1 = RotR(w[i-2],17)^RotR(w[i-2],19)^(w[i-2]>>10)`、`w[i] = w[i-16]+s0+w[i-7]+s1`。
4. 主压缩循环 64 轮，工作变量 `a..h`：`bigS1 = RotR(e,6)^RotR(e,11)^RotR(e,25)`、`ch = (e&f)^(~e&g)`、`temp1 = h+bigS1+ch+K[i]+w[i]`、`bigS0 = RotR(a,2)^RotR(a,13)^RotR(a,22)`、`maj = (a&b)^(a&c)^(b&c)`、`temp2 = bigS0+maj`，然后整体右移并 `e = d+temp1`、`a = temp1+temp2`。
5. 块尾把 `a..h` 累加回 `h0..h7`。
6. 输出：取**前 7 个字**（即 224 位 = 28 字节，丢弃 `h7`），每个字大端写出，返回 28 字节摘要。`RotR(x, n)` 为 32 位循环右移 `(x >> n) | (x << (32 - n))`。

**SHA3 包装**（`Sha3`）：本质是对 .NET 内置 `SHA3_256` / `SHA3_384` / `SHA3_512` 的薄封装。

- `ComputeHash(byte[], outputLengthBits)`：先检查 `IsSupported`（三种内置 SHA3 是否都受平台支持），不支持抛 `PlatformNotSupportedException("当前平台不支持SHA3算法")`；然后用 `switch` 按 `outputLengthBits` 派发到 `SHA3_256/384/512.HashData`，其它值抛 `ArgumentException`。代码里以注释保留了"224 需用 256 截断"的未启用分支。
- `IsSupported` = 三种内置 SHA3 的 `IsSupported` 全为真。
- 实例方法用构造时记录的 `_outputLengthBits` 调静态实现；`Dispose` 仅 `GC.SuppressFinalize`（无非托管资源，为满足 `using` 语义而实现 `IDisposable`）。

#### 数据流（input → parse → display）

- 定长哈希：每行 `InputText` →（按 `IsHexMode` 取字节）→ `HashFunc` → `ToHexString` → `Result`（数据绑定刷新）。
- SHA3：`SHA3InputTextBox.Text` →（按 `SHA3StringInputRadio` / `SHA3HexInputRadio` 取字节，默认 UTF-8）→ 取下拉框 `Sha3AlgorithmOption` → `new Sha3(value).ComputeHash(bytes)` → `ToHexString` → `SHA3ResultLabel.Content`。
- 文件拖放 `HashTab_Drop` → `ProcessFileForHashCalculation`：读字节 → `CalculateAndDisplayHashValues` 一次性算出 6 种定长哈希 → SHA3 输入框显示 hex 并切到 Hex 模式 → `FileDropHint` 提示已加载文件名、请选 SHA3 类型再算。

#### 事件流与错误处理

- `HashCompute_Click` / `HashClear_Click`：按 `Button.DataContext` 定位行；清空时把该行 `InputText` 置空、`Result` 复位"等待计算..."。
- `ComputeRow`：捕获 `FormatException` / `ArgumentException` 弹窗（消息含算法名）。
- `CalculateSHA3_Click`：非空 + 算法选择校验，捕获 `FormatException` / `ArgumentNullException` / `ArgumentOutOfRangeException`。
- `ClearSHA3_Click`：清 SHA3 输入与结果。`Grid_PreviewDragOver` 吞预览拖放。

---

### 设计决策小结

- **编排器/控件 + 按职责拆分的核心服务**：控件只做"UI 输入校验 + 弹窗 + 写回"，把加密/算法逻辑下沉到不依赖 WPF 的静态服务（`AesCryptoService` / `RsaCryptoService` / `CrcCalculator` / `Sha224` / `Sha3`），降低控件耦合、便于复用与测试。
- **共享读取原语**：hex/UTF-8 取字节（`ConvertUtils.InputBytes` / `HexStringToByteArray`）、hex 输出（`ToHexString`）、位反转（`ReverseBits`）、文件拖放读字节（`FileDropHelper`）在五个工具间统一复用，消除样板。
- **表驱动派发**：CRC 用 `Algorithms` 参数表 + 单一 `Compute` 覆盖十余种 CRC 变体；哈希用 `hashRows` 列表 + `ItemsControl` 数据模板覆盖 6 种定长算法，并借 `Button.DataContext` 模式匹配做事件分发——新增算法只需加一行数据。
- **统一的输入模式与文件拖放约定**：所有工具都以 String/Hex 单选切换输入解释，拖入文件统一转 hex 并切到 Hex 模式，保证二进制安全。
- **MVVM 式视图模型**：`HashAlgorithmRow` 实现 `INotifyPropertyChanged`，结果写回即自动刷新，避免为每个算法手写命名控件。
- **安全相关取舍**：RSA 默认 OAEP-SHA256（抗填充预言）/2048 位默认密钥；AES 保留 ECB/CFB/OFB 但以 `#pragma` 抑制分析告警；SHA-224 严格按标准实现而非截断。

### 1. 通用字节与编码原语（`ConvertUtils.cs`）

#### 1.1 职责

`ConvertUtils` 是一个 `internal static` 工具类，集中提供与具体二进制格式无关的字节操作：十六进制与字节数组互转、按位反转、字符串拼接，以及"按输入模式取字节"的统一入口。它主要服务于加解密/哈希工具域（AES/RSA/CRC/SHA 等控件的输入处理），但其十六进制格式化能力也被解析显示层间接复用。

#### 1.2 关键类型与方法

| 方法签名 | 作用 |
| --- | --- |
| `static byte[] HexStringToByteArray(string hex)` | 十六进制字符串 → 字节数组 |
| `static string ToHexString(byte[] bytes)` / `ToHexString(byte[] bytes, int startIndex, int length)` | 字节数组 → 大写十六进制字符串（委托 `Convert.ToHexString`） |
| `static uint ReverseBits(uint value, int width)` | 在指定位宽内按位镜像 |
| `static string EnumerableToString(string? separator, IEnumerable<string> values)` | 非空集合 join，空集合返回 `string.Empty` |
| `static byte[] InputBytes(string text, bool isHex)` | 按 UI 输入模式（Hex / UTF-8 文本）统一取字节 |

#### 1.3 核心算法与逻辑

**`HexStringToByteArray`（十六进制 → 字节）**

1. `ArgumentNullException.ThrowIfNull` 防御 null 入参。
2. 先用 `string.Replace` 去除空格与 `0x` 前缀（容忍用户粘贴带分隔的十六进制串），且显式传入 `StringComparison.CurrentCulture`。
3. 长度奇偶校验：长度非偶数抛 `ArgumentException("Hex字符串长度必须是偶数")`。
4. 每两个字符为一组，用 `Convert.ToByte(hex.Substring(i, 2), 16)` 解析为一个字节，写入 `bytes[i/2]`。

**`ReverseBits`（位反转）**——CRC 计算的核心子步骤。逐位将 `value` 的最低位"挤入"`result` 的最低位，同时左移 `result`、右移 `value`，循环 `width` 次，从而在给定位宽内完成比特镜像：

```csharp
for (int i = 0; i < width; i++)
{
    result <<= 1;
    result |= value & 1;
    value >>= 1;
}
```

它在 `CrcCalculator.cs` 中被两处调用，体现 CRC 的"反射"参数（RefIn/RefOut）：
- 输入反射：`(byte)(algo.ReverseInput ? ConvertUtils.ReverseBits(data[i], 8) : data[i])` —— 对每个输入字节按 8 位宽镜像；
- 输出反射：`crc = ConvertUtils.ReverseBits(crc, algo.Width)` —— 对最终 CRC 值按算法位宽镜像。

**`InputBytes`（表驱动的输入取字节）**——加解密/哈希控件统一通过它把"UI 文本框 + Hex/文本单选钮"映射成字节流：`isHex ? HexStringToByteArray(text) : Encoding.UTF8.GetBytes(text)`。例如 `CrcComputeControl.xaml.cs` 中 `ConvertUtils.InputBytes(input, CRCHexInputRadio.IsChecked == true)`，使各工具控件无需各自重写 Hex/文本分支。

#### 1.4 设计决策

把字节/十六进制/位操作收敛到一个无状态静态类，是为了让加解密工具的多个控件共享同一份"输入解析约定"（Hex 容错规则、奇偶校验、UTF-8 编码）。`ToHexString` 直接委托 BCL 的 `Convert.ToHexString` 而非手写循环，既保证正确性又避免分配额外缓冲。

---

### 2. 文件拖放与整文件读取（`FileDropHelper.cs`）

#### 2.1 职责

`FileDropHelper` 是 `internal static` 辅助类，消除各 WPF 控件在 `Drop` 事件处理里重复出现的"判断拖放数据格式 → 取第一个文件 → 读全部字节"样板。

#### 2.2 关键方法

- `static string? GetFirstDroppedFile(DragEventArgs e)` —— 用 `e.Data.GetDataPresent(DataFormats.FileDrop)` 判定存在文件拖放，并用列表模式 `is string[] { Length: > 0 } files` 同时完成类型与非空校验，返回 `files[0]`；无文件返回 `null`。
- `static byte[] ReadAllBytes(string filePath)` —— 以 `FileMode.Open, FileAccess.Read` 打开 `FileStream`，按 `fileStream.Length` 分配缓冲并用 `ReadExactly` 一次性读满全部字节。

#### 2.3 设计决策

拖放是多个文件类工具（PE/ELF 分析、哈希校验等）共有的输入方式；把"取首文件 + 整文件读取"提炼为两个纯函数，让控件的事件处理只关心业务流转。`ReadExactly` 取代旧式 `Read` 循环，避免部分读取的隐患。需要注意此处与解析器内部的读取策略不同：`FileDropHelper.ReadAllBytes` 与 `ELFParser`（`File.ReadAllBytes`）是"一次性全量缓冲"，而 `PEParser` 走的是流式 `FileStream`（见第 6 节端序与读取策略）。

---

### 3. 端序无关的读取原语

应用同时面对小端/大端 ELF 与始终小端的 PE，因此读取原语必须既能处理端序，又能在"流式顺序解析"与"已加载缓冲随机访问"两种场景下复用。两套原语分别落在 ELF 与 PE 侧。

#### 3.1 ELF 侧：`ELFParserUtils`（`ELFAnalyzer/Core/ELFParserUtils.cs`）

这是 ELF 解析的共享读取核心，全部基于 `System.Buffers.Binary.BinaryPrimitives`，每个读取方法都带一个 `bool isLittleEndian` 参数，依据它在 `ReadXxxLittleEndian` / `ReadXxxBigEndian` 之间分派。提供两族重载：

- **流式读取（顺序解析）**：参数为 `BinaryReader`，内部 `Span<byte> buffer = stackalloc byte[N]` + `reader.BaseStream.ReadExactly(buffer)`，再交给 `BinaryPrimitives`。涵盖 `ReadInt64 / ReadInt32 / ReadUInt16 / ReadUInt32 / ReadUInt64`。用于解析头部、程序头、节头时按文件顺序推进流位置。
- **缓冲读取（已加载数据，随机访问）**：参数为 `(byte[] data, int offset, bool isLittleEndian)`，用 `data.AsSpan(offset, N)` 取只读片段后解码，不移动也不修改源数组。涵盖 `ReadUInt16 / ReadUInt32 / ReadUInt64 / ReadInt32 / ReadInt64`。用于符号名、重定位、版本信息等"已把整节拷进字节数组、随机定位字段"的场景（如 `RelocationHelper.cs` 中按条目偏移反复 `ReadUInt64(data, b, little)`）。

`stackalloc` 固定大小栈缓冲避免每次读取的堆分配；缓冲族用 `ReadOnlySpan` 保证调用方拿到的节数据不被就地篡改（与第 7 节缓存只读约定一致）。

端序判定的单一真相来源是 `ELFHeaderInfo.cs` 中的扩展方法：

```csharp
public static bool IsLittleEndian(this ELFHeader header)
    => header.EI_DATA == (byte)ELFData.LSB;
```

各上层（重定位、Exidx、版本解析）统一调用 `parser.Header.IsLittleEndian()` 取得端序后再驱动上述读取原语，避免端序判断逻辑散落。注意 Exidx 解析里还有依端序对指令字节做 `Array.Reverse` 的特殊处理，同样以 `IsLittleEndian()` 为开关。

#### 3.2 PE 侧：`PEParserUtils`（`PEAnalyzer/Parsers/PEParserUtils.cs`）

PE 始终小端，故 PE 侧原语不带端序参数，重点在"定位/对齐/RVA 换算/有界字符串读取"：

- `static T ReadAtOffset<T>(FileStream fs, long offset, T fallback, Func<T> read)` —— "临时定位执行后恢复"原语：`offset` 无效（`<0` 或 `>= fs.Length`）直接返回 `fallback` 且不移动流；否则保存 `fs.Position`，定位到 `offset` 执行 `read()`，通过 `finally` 在正常/异常路径上都恢复原位置。异常本身不在此吞掉，向上层传播，与各导入/导出读取方法原有 try/finally 语义保持一致。
- `static long AlignTo4(long position)` —— `(position + 3) & ~3`，向上 4 字节对齐，供 VS_VERSIONINFO 等版本资源结构使用。
- `static bool Is64Bit / Is32Bit(IMAGE_OPTIONAL_HEADER)` —— 以 `optionalHeader.Magic` 对比 `PEConstants.Pe32PlusMagic` / `Pe32Magic` 判定位数。

#### 3.3 设计决策

ELF 与 PE 的读取原语没有强行合并，因为两者关注点不同：ELF 必须显式处理端序与两种访问模式，PE 则恒小端但需要丰富的"定位/对齐/RVA"语义。各自独立既保持职责清晰，又便于各解析模块共享同一组原语而不重复实现。

---

### 4. 字符串提取（有界、边界夹紧）

二进制文件中的字符串可能缺失终止符或长度字段被损坏，因此所有字符串读取都做了硬上限与边界夹紧。

#### 4.1 ELF 侧（`ELFParserUtils`）

- `static string ExtractStringFromBytes(byte[] data, int startOffset)` —— 从字符串表数据中按 NUL 截止读取：先校验 `data` 非空且 `startOffset` 在 `[0, data.Length)` 内，再从 `startOffset` 向后扫描直到遇到 `0` 或到达数组末尾，最后用 `Encoding.UTF8.GetString` 解码 `[startOffset, endOffset)`；空区间返回 `string.Empty`。
- `static string ExtractStringFromBytes(byte[] data, int startOffset, int maxLength)` —— 带最大长度版本，关键在**边界夹紧**：先 `maxLength = Math.Min(maxLength, data.Length - startOffset)` 把长度夹到剩余可读字节内，从而即使 note 段的 `namesz`/`descsz` 等长度字段被构造为越界值也不会触发 `IndexOutOfRangeException`；随后从尾部回退跳过尾随 NUL（`while (maxLength > 0 && data[startOffset + maxLength - 1] == 0) maxLength--`），最后 UTF-8 解码。

辅助方法 `static string FormatAddress(ulong address)` 统一以 `0x{address:X}` 形式格式化地址，供各显示层复用。

#### 4.2 PE 侧（`PEParserUtils`）

- `static string ReadNullTerminatedString(BinaryReader reader)` —— 读 ASCII C 字符串，带三重保护：`MaxLength = 1024` 上限防畸形文件无终止符导致超长读取；每次循环检查 `Position < Length` 防越界；并对字节做可打印性过滤（`>=32 && <=126` 直通、`9/10/13` 制表/换行/回车直通、其余替换为 `'?'`）。捕获 `EndOfStreamException` 后返回已读部分。
- `static string ReadUnicodeStringWithMaxLength(BinaryReader reader, int maxLength)` —— 读 UTF-16 字符串：`maxReadAttempts = Math.Min(maxLength, 1000)` 双重上限防死循环；每步检查 `Position + 2 > Length` 防越界；遇 `0` 终止。捕获 `IOException`/`UnauthorizedAccessException`/`ArgumentOutOfRangeException` 返回空串。
- `static string ReadUnicodeStringWithMaxBytes(BinaryReader reader, int maxBytes)` —— `maxBytes <= 0` 直接返回空，否则换算为字符上限（`maxBytes / 2`）委托上者。

#### 4.3 设计决策

ELF 字符串提取面向"已加载字符串表字节数组 + 偏移"，故为缓冲式且强调长度夹紧；PE 字符串读取面向"在流上顺序前进"，故为流式且强调硬上限与可打印过滤。两者共同体现"绝不信任二进制文件中的长度/偏移字段"的防御原则。

### 5. 枚举名解析与枚举/常量表

#### 5.1 通用枚举名解析（`ELFParserUtils.GetTypeName`）

```csharp
static string GetTypeName(Type enumType, object type, string prefix)
```

这是把任意原始数值映射为枚举名的统一入口，逻辑：

1. 用 `Convert.ChangeType(type, Enum.GetUnderlyingType(enumType), CultureInfo.InvariantCulture)` 先把传入数值转换成目标枚举的**底层类型**。这一步是关键——若直接对类型不匹配的值调用 `Enum.IsDefined` 会抛异常，先归一化底层类型既规避异常又省去每次调用的异常开销。
2. `Enum.IsDefined(enumType, value)` 判定是否为已定义成员，是则 `Enum.GetName` 返回名称。
3. 未定义或转换失败时返回 `"{prefix}UNKNOWN({type})"`，并分别捕获 `ArgumentException`/`InvalidCastException`/`OverflowException` 三类转换异常，统一按"未知"处理。

调用方以"表驱动"的薄封装复用它，例如：
- `ELFHeaderDescriptions.cs`：`GetTypeName(typeof(EMachine), header.e_machine, "")`、`typeof(ELFClass)`、`typeof(ELFData)`；
- `ELFDynamicInfo.cs`：`GetTypeName(typeof(DynamicTag), dTag, "")`。

每个枚举对应一行薄封装，新增机器类型/动态标签等只需扩充枚举表，无需改动解析逻辑。

#### 5.2 ELF 枚举表（`Enums/*.cs`）

`Enums` 目录共 50+ 个枚举，按底层类型精确匹配 ELF 字段宽度，构成解析与显示共享的"语义字典"：

- **标识与分类**：`ELFType : ushort`（ET_NONE/REL/EXEC/DYN/CORE）、`ELFClass : byte`（NONE/BIT32/BIT64）、`ELFData : byte`（NONE/LSB/MSB）、`EMachine : ushort`（数百种机器类型，EM_386/EM_X86_64/EM_ARM/EM_AARCH64 等）。
- **节相关**：`SectionType : uint`（SHT_*，含 GNU/ARM/Sun 扩展如 SHT_GNU_verdef/verneed/versym、SHT_ARM_EXIDX）、`[Flags] SectionAttributes : ulong`（SHF_WRITE/ALLOC/EXECINSTR/MERGE/STRINGS… 及 MASKOS/MASKPROC 掩码）。
- **段相关**：`ProgramHeaderType : uint`（PT_LOAD/DYNAMIC/INTERP/NOTE/TLS、PT_GNU_EH_FRAME/STACK/RELRO、PT_EXIDX）、`[Flags] ProgramHeaderFlags : uint`（PF_X/W/R + 掩码）。
- **符号相关**：`SymbolBinding : byte`（STB_*）、`SymbolType : byte`（STT_*）、`SymbolVisibility : byte`（STV_*）。
- **动态链接**：`DynamicTag : ulong`（DT_*，含 GNU 版本/MIPS 专有大量条目）、`[Flags] DynamicFlags : uint`（DF_*）。
- **重定位类型**：按架构各成一表（`X86_64RelocationType`、`ARMRelocationType`、`AArch64RelocationType`、`MipsRelocationType`、`PowerPC64RelocationType` 等十数种），供按 `e_machine` 选表解释重定位。
- **ARM ABI 属性**：`ABIFP*`、`ABIPCS*`、`ARMCPUArch*`、`ARMISAUse`、`THUMBISAUse`、`AdvancedSIMDArch`、`WMMXArch`、`VirtualizationUse` 等，服务 `.ARM.attributes` 解析。

枚举中刻意保留了 ELF 标准里值相同的别名（如 `SHT_HIOS = SHT_GNU_versym`、`SHF_LINK_ORDER = SHF_INFO_LINK`、`SHT_SUNW_move = SHT_LOSUNW`），并以注释标注，保证与官方定义一致。

#### 5.3 PE 常量表（`PEAnalyzer/Models/PEConstants.cs`）

PE 侧不用枚举而用 `internal static class PEConstants` 集中常量，避免解析器中散落魔法数字：

- **签名/魔数**：`DosSignature = 0x5A4D`（"MZ"）、`NtSignature = 0x00004550`（"PE\0\0"）、`Pe32Magic = 0x10b`、`Pe32PlusMagic = 0x20b`。
- **结构尺寸**：`Pe32OptionalHeaderBaseSize = 96` / `Pe32PlusOptionalHeaderBaseSize = 112`、`SectionHeaderSize = 40`、`ImportDescriptorSize = 20`、`DelayLoadDescriptorSize = 32`、`ExportDirectorySize = 40`、`Cor20HeaderMinSize = 72`、`DataDirectoryEntrySize = 8`、`MaxDataDirectories = 16`。
- **导入序号标志**：`OrdinalFlag32 = 0x80000000U`、`OrdinalFlag64 = 0x8000000000000000UL`。
- **数据目录索引**：`DirectoryExport=0`、`DirectoryImport=1`、`DirectoryResource=2`、`DirectorySecurity=4`、`DirectoryDelayImport=13`、`DirectoryClrHeader=14`。

#### 5.4 语言与全局状态（`Enums/LanguageType.cs`、`GlobalState.cs`）

`LanguageType`（English/SimplifiedChinese/TraditionalChinese）与 `GlobalState` 配套：`GlobalState.CurrentLanguageType` 是静态只读属性，**程序启动时求值一次并缓存**，内部 `GetCurrentLanguageType` 用 `CultureInfo.CurrentCulture.Name` 做 `switch`（zh-CN→简中，zh-TW/zh-HK/zh-MO→繁中，其余→英文）。它驱动 `ConstString/*` 下按语言分文件的常量字符串表（如 MySQL/Windows 系统错误的本地化文案）。一次性缓存避免每次取文案都重复读取区域信息。

### 6. RVA→文件偏移换算（PE 共享原语）

`PEParserUtils.RvaToOffset` 是导入/导出/CLR/资源解析共用的地址换算核心。该原语**与位数无关**（只用 32 位节表字段），是 PE 解析能从"虚拟地址视角"读取磁盘数据的桥梁：

```csharp
static long RvaToOffset(uint rva, List<IMAGE_SECTION_HEADER> sections, uint requiredLength = 1)
```

步骤：

1. **基础校验**：`rva == 0` 或节列表为空直接返回 `-1`。
2. **遍历节表**找包含该 RVA 的节：跳过 `SizeOfRawData == 0 || PointerToRawData == 0`（无磁盘数据）的节。
3. **VirtualSize 兼容**：`sectionVirtualSize = section.VirtualSize != 0 ? section.VirtualSize : section.SizeOfRawData`。部分旧链接器把 `VirtualSize` 置 0，此时退化用 `SizeOfRawData` 当作内存大小，否则落在该节的 RVA 会被误判为不可解析、静默丢失导入/导出/CLR 数据。
4. **区间判定**：`rva >= VirtualAddress && rva < VirtualAddress + sectionVirtualSize`。
5. **跨节/越界守卫**：`relativeOffset = rva - VirtualAddress`，要求 `relativeOffset + requiredLength <= SizeOfRawData`，否则返回 `-1`。这同时排除了 `.bss` 等未初始化数据节（RVA 落在 VirtualSize 内但磁盘上没有那么多字节）和"跨节读到下一节"的情况。`requiredLength` 让调用方声明"从该 RVA 起需要连续读取多少字节"，把整段都纳入校验。
6. 用 `long` 运算 `PointerToRawData + relativeOffset` 得到文件偏移，避免溢出；无法解析返回 `-1`。

设计上 `RvaToOffset` 与 `ReadAtOffset` 常配合使用：先 RVA→偏移，再 `ReadAtOffset` 临时定位读取并恢复流位置，构成 PE 内容目录解析的标准读取范式。

### 7. 性能：节数据缓存与零拷贝读取

#### 7.1 ELF 节数据缓存（`ELFParser.cs`）

`ELFParser` 持有 `private readonly Dictionary<int, byte[]> _sectionDataCache`，避免符号名/重定位/版本解析过程中反复整表拷贝同一节：

- `internal byte[] GetSectionData(int sectionIndex)` —— 先查缓存命中即返回；索引越界（如 `e_shstrndx` 超出节数）返回空数组避免 `IndexOutOfRange`；否则调 `CopySectionData` 拷贝并写入缓存。返回的数组**调用方不得就地修改**（缓冲式读取原语用 `ReadOnlySpan` 配合此约定）。
- `internal byte[] CopySectionData(in ELFSectionHeader section)` —— 实际拷贝并做完整边界校验：`sh_size == 0`、`sh_size > int.MaxValue`、`sh_offset + sh_size > FileData.Length` 任一成立即返回空数组，再 `Array.Copy` 截取。`in` 传引用避免结构体拷贝。

`TryGetLinkedStringTable` 进一步在缓存之上工作：根据节的 `sh_link` 取关联字符串表数据（verdef/verneed 解析与格式化共用），`sh_link` 越界返回 `false` 且置空数组，端序始终通过 `out isLittleEndian` 给出。`FindSectionByAddress` 则按 `sh_addr` 线性查找节，供版本/动态解析定位目标节。

#### 7.2 读取策略对比

- **PE**：`PEParser.ParsePEFile` 用流式 `FileStream(FileShare.Read)` + `BinaryReader`，按需定位读取（配合 `ReadAtOffset`/`RvaToOffset`），不一次性把整文件载入内存。
- **ELF**：`ELFParser` 构造时即 `File.ReadAllBytes` 全量缓冲（或直接接收 `byte[]`），后续解析在内存数组与按需 `MemoryStream` 上进行，并以节数据缓存复用片段。

两种策略反映各自访问模式：PE 内容目录通过 RVA 离散定位，适合流式随机读；ELF 大量按偏移随机访问且节会被重复读取，适合全量缓冲 + 缓存。

### 8. 横切关注点

#### 8.1 错误处理与健壮性策略

整个系统采用**分层防御 + 顶层兜底**的错误策略：

1. **防御性解析（不信任输入）**：所有读取前校验偏移/长度落在文件内（`RvaToOffset` 的 requiredLength 守卫、`CopySectionData` 的三重边界校验、`ReadAtOffset` 的 offset 有效性、字符串提取的 `Math.Min` 夹紧）。畸形结构一律返回哨兵值（`-1`、空数组、`string.Empty`）而非崩溃。
2. **边界/除零守卫**：循环读取均设硬上限（C 字符串 1024、Unicode 1000）防死循环；尺寸换算用 `long` 防溢出、用 `> int.MaxValue` 防 int 截断；上层在用条目大小做除法前先校验非零（ELF 重定位 `symTabData.Length / symEntrySize` 等场景配合非零项尺寸）。
3. **窄类型过滤的 `catch ... when`**：解析模块统一用异常过滤器只捕获预期的解析异常，绝不裸 `catch (Exception)`。典型集合见 `PEImportParser.cs`：
   ```csharp
   catch (Exception ex) when (ex is IOException or UnauthorizedAccessException or ArgumentOutOfRangeException)
   ```
   表现层 `ELFAnalyzerControl.xaml.cs` 的过滤集合更宽，覆盖一次解析全流程可能出现的全部受控异常：
   ```csharp
   catch (Exception ex) when (ex is UnauthorizedAccessException or ArgumentException
       or IndexOutOfRangeException or EndOfStreamException or IOException
       or OverflowException or DivideByZeroException or InvalidDataException or FormatException)
   ```
   过滤式捕获的好处是：非预期异常（如 `OutOfMemoryException`、空引用编程错误）不会被误吞，仍会上浮，从而既保证健壮又不掩盖真实 bug。各内容目录解析在 `PEParser.ParseContentDirectories` 中逐个独立 `catch`，**一个目录解析失败不影响其它目录**。
4. **格式合法性硬校验**：`PEParser.ParseHeaders` 在 DOS 签名、`e_lfanew` 越界、NT 头不完整、NT 签名、可选头大小、节头完整性等关键点抛 `InvalidDataException`；`ELFHeaderInfo.ReadELFHeader` 校验 `0x7F 'E' 'L' 'F'` 魔数。这些是"无法继续解析"的致命错误，主动抛出由上层捕获并提示用户。
5. **全局兜底（`App.xaml.cs`）**：`App.OnStartup` 挂接两个全局处理器作为最后防线——
   - `DispatcherUnhandledException`：UI 线程未处理异常 → `MessageBox` 提示"发生未处理的错误"，并 `e.Handled = true` 阻止应用崩溃；
   - `AppDomain.CurrentDomain.UnhandledException`：非 UI 线程的严重错误 → `MessageBox` 提示"发生严重错误"。

   这保证即便某条解析/控件路径漏掉了局部捕获，应用也不会静默崩溃。

#### 8.2 端序与 32/64 位统一

- **端序**：ELF 唯一真相源是 `ELFHeader.IsLittleEndian()`（基于 `EI_DATA`），所有 `ELFParserUtils.ReadXxx` 以 `bool isLittleEndian` 显式驱动 `BinaryPrimitives` 的大/小端解码；解析入口 `ParseELFFile` 在读头时用 `ref isLittleEndian` 求得并逐级下传。PE 恒小端，原语不带端序参数。
- **32/64 位**：ELF 在 `ReadELFHeader` 中由 `EI_CLASS` 求 `is64Bit`，对 `e_entry/e_phoff/e_shoff` 等指针宽度字段用 `is64Bit ? ReadUInt64 : ReadUInt32` 分派；`ELFParser.Is64Bit` 暴露给上层决定符号/重定位条目尺寸（如重定位中 `Is64Bit ? 24 : 16`）。PE 通过 `PEParserUtils.Is64Bit/Is32Bit`（比对 `Magic`）区分，并以 `PEConstants.Pe32*OptionalHeaderBaseSize`、`OrdinalFlag32/64` 适配两种布局。

#### 8.3 文本编码

- **UTF-8**：ELF 字符串表与 `ConvertUtils.InputBytes` 的文本模式均用 `Encoding.UTF8`。
- **UTF-16/Unicode**：PE 资源（版本信息等）的宽字符串用 `ReadUnicodeStringWithMaxLength` 逐 `ushort` 读取。
- **ASCII 可打印过滤**：PE 的 C 字符串读取对非可打印字节替换为 `'?'`，保证 UI 显示安全。

#### 8.4 关注点分离的整体架构基调

共享层的形态直接服务于上层的"编排器 + 按职责拆分的解析类 + 表驱动派发 + 共享读取原语"模式：

- **编排器**：`PEParser`（头部 → 内容目录两阶段）、`ELFParser.ParseELFFile`（头/程序头/节头/符号/动态/版本顺序编排）。
- **表驱动派发**：`PEParser.ContentParsers` 是 `Action<FileStream, BinaryReader, PEInfo>[]` 数组，把对各内容目录解析模块（导入/导出/版本/证书/CLR/图标）的依赖以**数据**形式集中表达，新增/删除一种目录只改数组，核心方法零改动；`GetTypeName` + 各枚举薄封装同样是表驱动思想在"数值→名称"上的体现。
- **共享读取原语**：本节的 `ELFParserUtils` / `PEParserUtils` / `ConvertUtils` / `FileDropHelper` 让所有解析与工具控件站在同一组经过加固的底层函数之上，把防御性逻辑写一次、用多处，是整个系统健壮性与可维护性的根基。
